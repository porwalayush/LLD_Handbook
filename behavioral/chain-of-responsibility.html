Behavioral Pattern

Chain of Responsibility

Pass a request along a chain of handlers until one of them handles it.

🎧 Real-World Analogy

Think of customer-support escalation. You call in and reach an L1 agent. If they can solve it, great — it stops there. If not, they escalate to L2. Still stuck? It goes to a manager. You, the caller, don't need to know who will ultimately fix your issue; you just send the request into the front of the chain and it travels until someone takes responsibility.

An ATM dispensing cash works the same way: a request for $80 flows through a $50 handler (gives one note, passes $30 on), then a $20 handler, then a $10 handler — each taking what it can and forwarding the rest.

The Problem It Solves

You have a request that could be handled by one of several objects, but you don't want the sender to hardcode which one. Hardwiring a giant if/else if/else ladder couples the sender to every possible handler and makes the logic brittle.

Chain of Responsibility lets you decouple the sender from the receiver. You build a chain of handler objects; the request flows down it until a handler decides to process it (and optionally stop) or passes it on. You can reorder, add, or remove handlers without touching the sender.

The Roles

  • Handler — an interface/base class declaring handle(request) and a way to link the next handler (setNext).
  • ConcreteHandler — decides whether it can handle the request; if not (or if it should continue), it forwards to the next handler.
  • The chain wiring — the client links handlers together (a → b → c) and sends the request to the head.

Implementation

Here is an expense-approval chain. A purchase request flows through approvers with rising spending limits: a Team Lead approves small amounts, a Manager more, and a Director the largest. Each handler either approves the request or passes it up the chain.

// --- Handler: base class with handle() + setNext() ---
abstract class Approver {
    protected Approver next;          // the next link in the chain

    public Approver setNext(Approver next) {
        this.next = next;
        return next;                    // return next to allow fluent chaining
    }

    public abstract void handle(double amount);

    // shared helper: forward to the next handler if there is one
    protected void passOn(double amount) {
        if (next != null) next.handle(amount);
        else System.out.println("  No one could approve $" + amount);
    }
}

// --- ConcreteHandlers: each handles up to its limit, else passes on ---
class TeamLead extends Approver {
    public void handle(double amount) {
        if (amount <= 1_000) System.out.println("TeamLead approved $" + amount);
        else passOn(amount);
    }
}
class Manager extends Approver {
    public void handle(double amount) {
        if (amount <= 10_000) System.out.println("Manager approved $" + amount);
        else passOn(amount);
    }
}
class Director extends Approver {
    public void handle(double amount) {
        if (amount <= 100_000) System.out.println("Director approved $" + amount);
        else passOn(amount);
    }
}

// --- Client: build the chain, then send requests to the head ---
Approver lead = new TeamLead();
lead.setNext(new Manager()).setNext(new Director());

lead.handle(800);      // TeamLead approved $800
lead.handle(7_500);    // Manager approved $7500
lead.handle(90_000);   // Director approved $90000
lead.handle(500_000);  // No one could approve $500000

from abc import ABC, abstractmethod

# --- Handler: base class with handle() + set_next() ---
class Approver(ABC):
    def __init__(self):
        self._next = None      # the next link in the chain

    def set_next(self, nxt):
        self._next = nxt
        return nxt          # return next to allow fluent chaining

    @abstractmethod
    def handle(self, amount): ...

    # shared helper: forward to the next handler if there is one
    def pass_on(self, amount):
        if self._next:
            self._next.handle(amount)
        else:
            print(f"  No one could approve ${amount}")


# --- ConcreteHandlers: each handles up to its limit, else passes on ---
class TeamLead(Approver):
    def handle(self, amount):
        if amount <= 1_000: print(f"TeamLead approved ${amount}")
        else: self.pass_on(amount)

class Manager(Approver):
    def handle(self, amount):
        if amount <= 10_000: print(f"Manager approved ${amount}")
        else: self.pass_on(amount)

class Director(Approver):
    def handle(self, amount):
        if amount <= 100_000: print(f"Director approved ${amount}")
        else: self.pass_on(amount)


# --- Client: build the chain, then send requests to the head ---
lead = TeamLead()
lead.set_next(Manager()).set_next(Director())

lead.handle(800)       # TeamLead approved $800
lead.handle(7_500)     # Manager approved $7500
lead.handle(90_000)    # Director approved $90000
lead.handle(500_000)   # No one could approve $500000
You use this every day. Web middleware (Express, ASP.NET) and Java servlet filters are textbook Chain of Responsibility: each middleware inspects the request, optionally handles or rejects it, and otherwise calls next() to pass control along — authentication, logging, compression, and routing all stacked as links in a chain.

When to Use It

  • More than one object may handle a request and the handler isn't known until runtime.
  • You want to issue a request without specifying the receiver explicitly.
  • The set of handlers — and their order — should be configurable on the fly.
  • You're building a pipeline: logging levels, middleware, filters, validators, event escalation.

Trade-offs

✓ Pros

  • Decouples sender from receiver.
  • Add, remove, or reorder handlers without changing the sender (Open/Closed).
  • Each handler has one clear responsibility (Single Responsibility).
  • Flexible: build the chain dynamically at runtime.

✗ Cons

  • A request may fall off the end unhandled — handle that case explicitly.
  • No guarantee any handler will process the request.
  • Debugging chains is harder — flow is spread across many objects.
  • A long chain can hurt performance if every link is visited.
Watch the end of the chain. Always decide what happens when no handler accepts the request — log it, throw, or use a default "catch-all" handler. Silently dropping requests is a classic Chain of Responsibility bug.

Related Patterns

Command turns a request into an object; it pairs well with Chain of Responsibility when the command is what travels down the chain. Decorator looks structurally similar (a linked sequence wrapping a call) but every decorator always contributes, whereas a chain handler may pass the request on untouched. Mediator centralizes communication instead of letting a request hop linearly between objects.

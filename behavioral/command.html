Behavioral Pattern

Command

Encapsulate a request as an object, letting you parameterize, queue, log, and undo operations.

🍽️ Real-World Analogy

Think of a restaurant order ticket. When you order, the waiter doesn't run into the kitchen and start cooking. They write your request on a ticket — an object that fully describes what to make — and hand it to the kitchen. The waiter (who takes the request) is completely decoupled from the chef (who fulfills it). The ticket can be queued on a rail, logged, re-fired, or even cancelled.

A TV remote is the same idea: each button is bound to a command. The "volume up" button doesn't know how the TV raises volume — it just holds a command object that, when pressed, tells the TV (the receiver) to act.

The Problem It Solves

Sometimes you need to issue a request without knowing — or caring — exactly which object will handle it or how. You want to decouple the object that invokes an operation (the invoker) from the object that knows how to perform it (the receiver).

Doing this opens up powerful capabilities that are awkward with plain method calls:

  • Undo / redo — if a request is an object, it can remember how to reverse itself.
  • Queuing & scheduling — store requests in a list and run them later, in another thread, or after a delay.
  • Logging — persist commands to replay after a crash.
  • Macros — bundle several commands into one composite command.

The Roles

  • Command — an interface declaring execute() (and usually undo()).
  • ConcreteCommand — binds a receiver to an action; its execute() calls the receiver.
  • Receiver — the object that knows how to do the real work (the TV, the document, the light).
  • Invoker — holds a command and triggers it (the remote button, the menu item). It never talks to the receiver directly.
  • Client — creates concrete commands and wires them to receivers.

Implementation

Below, a text editor exposes commands like "type text" and "delete". Each command knows how to execute() and undo() itself. The editor (invoker) keeps an undo stack of executed commands so it can roll them back one by one.

import java.util.*;

// --- Command interface ---
interface Command {
    void execute();
    void undo();
}

// --- Receiver: knows how to do the real work ---
class Document {
    private final StringBuilder text = new StringBuilder();
    public void append(String s) { text.append(s); }
    public void deleteLast(int n) { text.delete(text.length() - n, text.length()); }
    public String getText() { return text.toString(); }
}

// --- ConcreteCommand: binds receiver + action + undo data ---
class TypeCommand implements Command {
    private final Document doc;
    private final String textToAdd;

    TypeCommand(Document doc, String textToAdd) {
        this.doc = doc;
        this.textToAdd = textToAdd;
    }
    public void execute() { doc.append(textToAdd); }
    public void undo()    { doc.deleteLast(textToAdd.length()); }
}

// --- Invoker: triggers commands and keeps an undo stack ---
class Editor {
    private final Deque<Command> history = new ArrayDeque<>();

    public void run(Command cmd) {
        cmd.execute();
        history.push(cmd);          // remember it so we can undo
    }
    public void undo() {
        if (!history.isEmpty()) history.pop().undo();
    }
}

// --- Client: wires it all together ---
Document doc = new Document();
Editor editor = new Editor();

editor.run(new TypeCommand(doc, "Hello"));
editor.run(new TypeCommand(doc, ", World"));
// doc -> "Hello, World"
editor.undo();                  // undoes last command
// doc -> "Hello"

from abc import ABC, abstractmethod

# --- Command interface ---
class Command(ABC):
    @abstractmethod
    def execute(self): ...
    @abstractmethod
    def undo(self): ...


# --- Receiver: knows how to do the real work ---
class Document:
    def __init__(self):
        self.text = ""

    def append(self, s):
        self.text += s

    def delete_last(self, n):
        self.text = self.text[:-n]


# --- ConcreteCommand: binds receiver + action + undo data ---
class TypeCommand(Command):
    def __init__(self, doc, text_to_add):
        self.doc = doc
        self.text_to_add = text_to_add

    def execute(self):
        self.doc.append(self.text_to_add)

    def undo(self):
        self.doc.delete_last(len(self.text_to_add))


# --- Invoker: triggers commands and keeps an undo stack ---
class Editor:
    def __init__(self):
        self.history = []          # used as a stack

    def run(self, cmd):
        cmd.execute()
        self.history.append(cmd)   # remember it so we can undo

    def undo(self):
        if self.history:
            self.history.pop().undo()


# --- Client: wires it all together ---
doc = Document()
editor = Editor()

editor.run(TypeCommand(doc, "Hello"))
editor.run(TypeCommand(doc, ", World"))
print(doc.text)        # "Hello, World"
editor.undo()            # undoes last command
print(doc.text)        # "Hello"
The magic is the undo stack. Because each request is a self-contained object that remembers both how to do and undo its work, the invoker can keep a history and reverse operations in order. Add a second "redo" stack and you get full redo support — exactly how editors, IDEs, and design tools implement Ctrl+Z / Ctrl+Y.

When to Use It

  • You need undo / redo, or want operations you can replay or roll back.
  • You want to queue, schedule, or run requests later (task queues, job runners, thread pools).
  • You want to log requests so they can be re-executed after a failure (transactional behavior).
  • You want to parameterize objects (buttons, menu items, shortcuts) with the action they perform.
  • You need macros — compose several commands into one.

Trade-offs

✓ Pros

  • Decouples the invoker from the receiver (Single Responsibility).
  • Makes undo / redo, logging, and queuing straightforward.
  • Easy to add new commands without touching existing code (Open/Closed).
  • Commands can be composed into macros.

✗ Cons

  • Adds a lot of small classes — one per action.
  • Can feel like over-engineering for trivial, one-off calls.
  • Storing undo state for complex operations can be tricky and memory-heavy.
Interview tip: Command shines whenever you hear "undo/redo", "task queue", or "transactional / replayable operations". Mention that the key insight is turning a method call into a first-class object — once a request is data, you can store it, log it, schedule it, and reverse it.

Related Patterns

Command often stores state to support undo, where Memento can hold the snapshot to restore. Observer also decouples senders from receivers but broadcasts events rather than encapsulating a single request. A command can hold a Strategy object to choose how its action is performed.

Behavioral Pattern

Interpreter

Given a language, define a representation for its grammar along with an interpreter that uses the representation to interpret sentences in that language.

🎼 Real-World Analogy

Think of a musician reading sheet music. The sheet is a "sentence" written in the language of music; each symbol — a note, a rest, a time signature — is a small rule with its own meaning. The musician reads the symbols one by one and interprets them into sound, combining little pieces into a whole performance.

Or picture a simple calculator reading 3 + 5 * 2. It must understand that numbers are values, that * binds tighter than +, and how to combine them. In the Interpreter pattern, each grammar rule becomes a class, and evaluating the sentence is just asking the top of that structure to interpret itself.

The Problem It Solves

Sometimes you need to evaluate sentences in a small, well-defined language that recurs often enough to deserve its own representation. Examples:

  • Arithmetic or boolean expressions (3 + 5 * 2, price > 100 AND inStock).
  • A rule engine evaluating business conditions against some data.
  • Simple query filters, search criteria, or regex-like matchers.

The Interpreter pattern says: model the grammar as a set of classes, build the sentence into a tree of those classes (an expression tree, also called an AST), then evaluate the tree recursively. Because the grammar maps one-to-one onto classes, each rule's behavior lives in exactly one place and the structure mirrors how the language is defined.

The Roles

  • AbstractExpression — declares the single interpret(context) operation shared by every node.
  • TerminalExpression — the leaves: literals and variables (e.g. a Number or a variable lookup). They interpret to a value directly.
  • NonTerminalExpression — the operators that combine sub-expressions (e.g. Add, Multiply). They interpret their children, then combine the results.
  • Context — any global information the interpreter needs, such as variable bindings (x = 10).

How It Works

  1. Define one class per grammar rule, all sharing an interpret() method.
  2. Build an expression tree for the sentence — 3 + 5 * 2 becomes Add(3, Multiply(5, 2)).
  3. Call interpret() on the root. Each non-terminal recursively interprets its children and combines them; each terminal returns its own value. The recursion bottoms out at the leaves.

Implementation

Below is a small arithmetic evaluator. Number is the terminal expression; Add, Subtract, and Multiply are non-terminals that each hold two sub-expressions. Evaluating (3 + 5) * 2 is simply building the tree and calling interpret() on its root.

// --- AbstractExpression: every node knows how to interpret itself ---
interface Expression {
    int interpret();
}

// --- TerminalExpression: a literal value (a leaf of the tree) ---
class Number implements Expression {
    private final int value;
    Number(int value) { this.value = value; }
    public int interpret() { return value; }
}

// --- NonTerminalExpressions: operators combining sub-expressions ---
class Add implements Expression {
    private final Expression left, right;
    Add(Expression l, Expression r) { left = l; right = r; }
    public int interpret() {
        return left.interpret() + right.interpret();   // recurse, then combine
    }
}

class Subtract implements Expression {
    private final Expression left, right;
    Subtract(Expression l, Expression r) { left = l; right = r; }
    public int interpret() {
        return left.interpret() - right.interpret();
    }
}

class Multiply implements Expression {
    private final Expression left, right;
    Multiply(Expression l, Expression r) { left = l; right = r; }
    public int interpret() {
        return left.interpret() * right.interpret();
    }
}

// --- Usage: build the tree for (3 + 5) * 2, then interpret it ---
Expression expr =
    new Multiply(
        new Add(new Number(3), new Number(5)),
        new Number(2));

System.out.println(expr.interpret());   // 16

from abc import ABC, abstractmethod

# --- AbstractExpression: every node knows how to interpret itself ---
class Expression(ABC):
    @abstractmethod
    def interpret(self):
        ...

# --- TerminalExpression: a literal value (a leaf of the tree) ---
class Number(Expression):
    def __init__(self, value):
        self.value = value
    def interpret(self):
        return self.value

# --- NonTerminalExpressions: operators combining sub-expressions ---
class Add(Expression):
    def __init__(self, left, right):
        self.left, self.right = left, right
    def interpret(self):
        return self.left.interpret() + self.right.interpret()

class Subtract(Expression):
    def __init__(self, left, right):
        self.left, self.right = left, right
    def interpret(self):
        return self.left.interpret() - self.right.interpret()

class Multiply(Expression):
    def __init__(self, left, right):
        self.left, self.right = left, right
    def interpret(self):
        return self.left.interpret() * self.right.interpret()

# --- Usage: build the tree for (3 + 5) * 2, then interpret it ---
expr = Multiply(
    Add(Number(3), Number(5)),
    Number(2),
)

print(expr.interpret())   # 16
Tip: Notice that this pattern only covers the evaluation step — it assumes you already have the expression tree. Turning raw text like "(3 + 5) * 2" into that tree is a separate job called parsing, which the Interpreter pattern deliberately does not address. In interviews, build the tree by hand (as above) and say you would use a real parser for the text-to-tree step.
Be honest about scope. Interpreter is one of the rarest patterns in practice and is almost never the right answer for a real language. The moment your grammar grows operators, precedence, functions, and error handling, the number of classes explodes and the code becomes hard to maintain. For anything beyond a tiny DSL, reach for a real parser generator such as ANTLR, or a hand-written recursive-descent parser, instead of hand-rolling Interpreter classes.

When to Use It

  • You have a simple, stable grammar — a small DSL, a rule engine, or a search/filter mini-language.
  • Efficiency is not critical; building and walking a tree of objects is acceptable.
  • The grammar is small enough that one-class-per-rule stays manageable.

Trade-offs

✓ Pros

  • The grammar maps cleanly onto classes — easy to read and reason about.
  • Adding a new grammar rule is just adding a new expression class.
  • Recursive evaluation falls out naturally from the tree structure.

✗ Cons

  • Class explosion — complex grammars need a class per rule, quickly unmanageable.
  • Hard to maintain and extend once the language grows.
  • Does not include parsing; you still need a separate tokenizer/parser.
  • Rarely the right tool — a real parser library usually wins.

Related Patterns

The expression tree built by Interpreter is essentially a Composite — terminals are leaves and non-terminals are branches. A Visitor is often used to add new operations over that tree (pretty-printing, type-checking, optimizing) without bloating each expression class. And an Iterator can traverse the nodes of the tree when you need to walk it rather than evaluate it.

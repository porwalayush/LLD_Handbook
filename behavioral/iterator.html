Behavioral Pattern

Iterator

Provide sequential access to the elements of a collection without exposing its underlying representation.

📺 Real-World Analogy

Think of the channel up / channel down buttons on a TV remote. You step through channels one at a time without ever knowing how the TV stores them — an array, a sorted list, a tuner frequency table, who cares? The remote gives you a uniform "next" and "previous" no matter what's behind the scenes.

A music playlist's next / previous works the same way. You traverse songs in order without touching the data structure holding them. That clean "give me the next item" handle is the Iterator.

The Problem It Solves

Collections come in many shapes — arrays, linked lists, trees, hash tables. Each stores elements differently. If client code had to know those internals to walk the elements, it would be tightly coupled and break whenever the structure changed.

The Iterator pattern pulls traversal logic out into a separate object. This lets you:

  • Traverse different collections uniformly with the same hasNext() / next() interface.
  • Hide the internal representation — clients don't see arrays or nodes.
  • Support multiple simultaneous traversals — each iterator keeps its own position.

The Roles

  • Iterator — interface declaring hasNext() and next().
  • ConcreteIterator — tracks the current position while walking a specific collection.
  • Aggregate / Collection — interface declaring createIterator().
  • ConcreteCollection — returns a concrete iterator over its own data.

Implementation

Below we hand-roll a custom Playlist collection with its own iterator — to see the moving parts — and then show the language-native way, which is what you'd actually use in real code.

import java.util.*;

// === The pattern, hand-rolled ===
// In Java you implement the built-in Iterable / Iterator interfaces,
// which unlocks the enhanced for-each loop for free.
class Playlist implements Iterable<String> {
    private final String[] songs;
    Playlist(String... songs) { this.songs = songs; }

    // createIterator(): hand back a fresh iterator with its own cursor
    public Iterator<String> iterator() {
        return new Iterator<>() {
            private int pos = 0;                // each iterator tracks its own position
            public boolean hasNext() { return pos < songs.length; }
            public String next()     { return songs[pos++]; }
        };
    }
}

// --- Usage: the iterator powers the for-each loop ---
Playlist playlist = new Playlist("Song A", "Song B", "Song C");
for (String song : playlist) {       // calls iterator() under the hood
    System.out.println(song);
}

// === The language-native way (what you actually write) ===
// ArrayList already IS an Iterable -- you almost never hand-roll this.
List<String> channels = List.of("News", "Sports", "Movies");
for (String c : channels) {           // built-in iterator
    System.out.println(c);
}
// Or get the iterator explicitly:
Iterator<String> it = channels.iterator();
while (it.hasNext()) System.out.println(it.next());

# === The pattern, hand-rolled ===
# In Python you implement __iter__ and __next__ (the iterator protocol).
class Playlist:
    def __init__(self, *songs):
        self.songs = songs

    # __iter__ == createIterator(): return an object with its own cursor
    def __iter__(self):
        return PlaylistIterator(self.songs)


class PlaylistIterator:
    def __init__(self, songs):
        self._songs = songs
        self._pos = 0                  # each iterator tracks its own position

    def __iter__(self):
        return self

    def __next__(self):
        if self._pos >= len(self._songs):
            raise StopIteration      # signals "no more items"
        item = self._songs[self._pos]
        self._pos += 1
        return item


# --- Usage: the protocol powers the for loop ---
playlist = Playlist("Song A", "Song B", "Song C")
for song in playlist:               # calls __iter__ then __next__
    print(song)

# === The language-native way (what you actually write) ===
# A generator with yield IS an iterator -- the simplest way by far.
def channels():
    yield "News"
    yield "Sports"
    yield "Movies"

for c in channels():               # generators are lazy iterators
    print(c)

# And every list/tuple/dict is already iterable:
for c in ["News", "Sports", "Movies"]:
    print(c)
You rarely hand-roll this — the pattern is baked into the language. Java has Iterable / Iterator driving the enhanced for-each loop, and every ArrayList, HashSet, etc. already implements it. Python has the __iter__ / __next__ protocol driving the for loop, plus yield generators as the dead-simple shortcut. The pattern is the concept; the languages give you the plumbing.

When to Use It

  • You want to traverse a collection without exposing how it stores elements.
  • You need multiple independent traversals over the same collection at once.
  • You want one uniform traversal interface across different collection types.
  • You're building a custom data structure (a tree, a graph, a paged result set) and want it to play nicely with the language's loops.

Trade-offs

✓ Pros

  • Decouples traversal from the collection's internals (Single Responsibility).
  • Same interface works across many collection types.
  • Supports several simultaneous, independent traversals.
  • Enables lazy iteration — fetch elements only as needed (generators).

✗ Cons

  • Overkill for a simple array you can index directly.
  • A hand-rolled iterator adds an extra class for little gain.
  • Modifying a collection mid-iteration can break the iterator (concurrent-modification bugs).
Interview tip: Know the built-in iterator protocols cold — Java's Iterable/Iterator + for-each, and Python's __iter__/__next__ + yield generators. Interviewers love to see you implement a custom collection that hooks into the native loop, and to hear you say "I'd reach for a generator rather than write an iterator class by hand."

Related Patterns

Composite structures (trees) are commonly walked with an Iterator that hides the recursive traversal. Strategy is conceptually close — an iterator is, in a sense, a strategy for traversal. Visitor pairs with Iterator when you want to apply an operation to every element you visit.

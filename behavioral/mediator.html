Behavioral Pattern

Mediator

Define an object that centralizes communication between a set of objects, so they don't refer to each other directly — turning a tangled many-to-many web into a clean many-to-one hub.

🛫 Real-World Analogy

Picture an air-traffic control tower. Dozens of planes are landing and taking off at once. If every pilot had to negotiate directly with every other pilot — "you go first, I'll circle, watch my wing" — it would be chaos and a guaranteed collision.

Instead, no plane talks to another plane. They all talk only to the tower, and the tower coordinates everyone. The planes don't even need to know each other exist. That tower is the Mediator: a single hub that all participants route their messages through.

The Problem It Solves

When a set of objects all need to interact, the easy thing is to let each one hold references to the others and call them directly. With n objects that's up to n×(n-1) connections — a dense, tangled web:

  • Every object must know about every other object → tight coupling everywhere.
  • You can't reuse one object without dragging its neighbours along.
  • Changing how two objects interact means editing both — and often a third that depended on the interaction.
  • The interaction logic is smeared across all the objects with no single place to understand or change it.

Mediator collapses that mesh: each object talks only to one central mediator (many-to-one), and the mediator decides who to forward to. The objects become decoupled and reusable; the coordination logic lives in one well-defined place.

How It Works

  1. Mediator — an interface declaring how components notify the hub (e.g. notify(sender, event)).
  2. ConcreteMediator — implements the coordination logic; knows all the components and routes messages between them.
  3. Colleagues (Components) — the participating objects. Each holds a reference to the mediator only, never to other colleagues. To communicate, a colleague tells the mediator, and the mediator forwards.

Implementation

Here a ChatRoom acts as the mediator routing messages between User colleagues. Users never reference each other — they only send() through the room, and the room broadcasts to everyone else.

import java.util.*;

// --- Mediator interface ---
interface ChatMediator {
    void register(User user);
    void send(String msg, User sender);
}

// --- ConcreteMediator: holds the coordination logic ---
class ChatRoom implements ChatMediator {
    private final List<User> users = new ArrayList<>();

    public void register(User user) { users.add(user); }

    public void send(String msg, User sender) {
        for (User u : users) {          // route to everyone but the sender
            if (u != sender) u.receive(msg, sender.name);
        }
    }
}

// --- Colleague: talks ONLY to the mediator ---
class User {
    final String name;
    private final ChatMediator room;   // reference to the hub only

    User(String name, ChatMediator room) {
        this.name = name;
        this.room = room;
        room.register(this);
    }

    public void send(String msg) { room.send(msg, this); }
    public void receive(String msg, String from) {
        System.out.println(name + " got from " + from + ": " + msg);
    }
}

// Usage
ChatMediator room = new ChatRoom();
User alice = new User("Alice", room);
User bob   = new User("Bob", room);
alice.send("Hi everyone!");   // Bob receives; Alice does not

from abc import ABC, abstractmethod

# --- Mediator interface ---
class ChatMediator(ABC):
    @abstractmethod
    def register(self, user): ...
    @abstractmethod
    def send(self, msg, sender): ...


# --- ConcreteMediator: holds the coordination logic ---
class ChatRoom(ChatMediator):
    def __init__(self):
        self.users = []

    def register(self, user):
        self.users.append(user)

    def send(self, msg, sender):
        for u in self.users:        # route to everyone but the sender
            if u is not sender:
                u.receive(msg, sender.name)


# --- Colleague: talks ONLY to the mediator ---
class User:
    def __init__(self, name, room):
        self.name = name
        self.room = room          # reference to the hub only
        room.register(self)

    def send(self, msg):
        self.room.send(msg, self)

    def receive(self, msg, frm):
        print(f"{self.name} got from {frm}: {msg}")


# Usage
room = ChatRoom()
alice = User("Alice", room)
bob   = User("Bob", room)
alice.send("Hi everyone!")   # Bob receives; Alice does not
Mediator vs. Observer: Both decouple objects, but the direction differs. Observer is a one-to-many broadcast — one subject notifies many listeners that don't talk back. Mediator is two-way coordination — many peers exchange messages through a central hub that contains the routing logic. Remember the air-traffic tower: planes both send to and receive from the tower, and the tower actively decides who hears what.

When to Use It

  • A set of objects communicate in complex, many-to-many ways that are hard to follow.
  • Reusing a component is hard because it's tightly coupled to too many others.
  • Behaviour spread across several classes should be customizable in one place without subclassing all of them.
  • Classic fit: UI dialogs where one widget's change enables/disables/updates others.

Trade-offs

✓ Pros

  • Removes the tangled web — components stay decoupled.
  • Centralizes interaction logic in one understandable place.
  • Components become reusable and individually testable.
  • Easy to change how peers interact — edit only the mediator.

✗ Cons

  • The mediator can become a god object — a bloated hub that knows too much.
  • Complexity migrates into the mediator rather than disappearing.
  • A single point of failure for the whole interaction.
  • Can become a bottleneck for high-frequency messaging.
Watch the god object. The biggest risk is letting the mediator absorb every rule until it's an unmaintainable monolith. If your ConcreteMediator keeps growing, split it by concern (multiple smaller mediators) or push logic that truly belongs to a component back into that component.
Interview tip: Reach for Mediator when you describe "every object knows about every other object" pain. Say the magic phrase — it converts many-to-many coupling into many-to-one — and proactively call out the god-object risk so the interviewer sees you know the cost, not just the benefit.

Related Patterns

Observer is the close cousin — Mediator often uses Observer internally to notify colleagues. Facade also provides a single entry point to a subsystem, but it's a one-way simplifier — clients call in, the subsystem doesn't call back, and there's no peer-to-peer coordination. Command pairs well with Mediator: colleagues can send command objects through the hub instead of raw method calls.

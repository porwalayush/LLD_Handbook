Behavioral Pattern

Memento

Capture and externalize an object's internal state so it can be restored later — without violating the object's encapsulation.

🎮 Real-World Analogy

Think of a save point in a video game. Before a tough boss fight you hit "save". The game writes a sealed snapshot of your progress — health, inventory, position — to a slot. If you die, you "load" and the world is restored to exactly that moment.

Crucially, you hold the save slot, but you can't open it up and hand-edit your health to 9999 — its contents are opaque. Only the game knows how to read it back. That's Memento: a sealed snapshot (the save file), the game that creates/restores it, and you, the keeper, who just holds the slots. It's the same idea behind Ctrl+Z undo and document version history.

The Problem It Solves

You need to implement undo, rollback, or checkpoints — restoring an object to a previous state. The obvious approach is to read out all the object's fields, store them somewhere, and write them back later. But that breaks encapsulation: the object would have to expose its private internals to whoever does the saving.

  • A text editor needs unlimited undo of every edit.
  • A drawing app must roll back the canvas after a mistaken stroke.
  • A transaction must restore state if a step fails partway.
  • A game needs save/load without exposing the player object's guts.

Memento's insight: let the object itself produce a snapshot of its own state (so private fields stay private), wrap that snapshot in an opaque object, and hand it to a keeper who stores it but can't peek inside. To undo, hand the snapshot back to the object and let it restore itself.

How It Works

  1. Originator — the object whose state we save. It creates a memento (save()) capturing its current state, and restores itself from one (restore(memento)).
  2. Memento — an immutable snapshot of the originator's state. It's opaque to everyone except the originator: the caretaker can hold it but cannot read or modify its contents.
  3. Caretaker — keeps the mementos (e.g. an undo-history stack). It asks the originator to save, stores the result, and later hands one back to restore — but never inspects it.

Implementation

A text editor is the Originator. EditorMemento is the opaque snapshot. History is the Caretaker holding a stack of mementos for undo. Notice the caretaker only ever passes mementos around — it never reads their fields.

import java.util.*;

// --- Memento: immutable, opaque snapshot ---
// final + private state + no setters = caretaker can't tamper with it
final class EditorMemento {
    private final String content;
    EditorMemento(String content) { this.content = content; }
    // package-private getter: only the Originator (same package) reads it
    String getContent() { return content; }
}

// --- Originator: creates and restores from mementos ---
class Editor {
    private String content = "";

    public void type(String words) { content += words; }
    public String getContent() { return content; }

    public EditorMemento save() {          // snapshot current state
        return new EditorMemento(content);
    }
    public void restore(EditorMemento m) { // roll back to a snapshot
        this.content = m.getContent();
    }
}

// --- Caretaker: keeps mementos, never inspects them ---
class History {
    private final Deque<EditorMemento> stack = new ArrayDeque<>();
    public void push(EditorMemento m) { stack.push(m); }
    public EditorMemento pop() { return stack.pop(); }
}

// Usage
Editor editor = new Editor();
History history = new History();

editor.type("Hello");
history.push(editor.save());   // checkpoint: "Hello"
editor.type(", world");     // now "Hello, world"
editor.restore(history.pop());  // undo -> back to "Hello"

from dataclasses import dataclass

# --- Memento: immutable, opaque snapshot ---
# frozen=True -> the caretaker can hold it but cannot mutate it
@dataclass(frozen=True)
class EditorMemento:
    _content: str        # conventionally "private" to the Originator


# --- Originator: creates and restores from mementos ---
class Editor:
    def __init__(self):
        self._content = ""

    def type(self, words):
        self._content += words

    @property
    def content(self):
        return self._content

    def save(self):                  # snapshot current state
        return EditorMemento(self._content)

    def restore(self, memento):       # roll back to a snapshot
        self._content = memento._content


# --- Caretaker: keeps mementos, never inspects them ---
class History:
    def __init__(self):
        self._stack = []
    def push(self, memento):
        self._stack.append(memento)
    def pop(self):
        return self._stack.pop()


# Usage
editor = Editor()
history = History()

editor.type("Hello")
history.push(editor.save())     # checkpoint: "Hello"
editor.type(", world")        # now "Hello, world"
editor.restore(history.pop())   # undo -> back to "Hello"
The encapsulation boundary is the whole point. The caretaker must be able to store a memento but never read or alter its contents — otherwise the originator's private state leaks. In Java this is enforced with a final class, private fields, and package-private (or nested-class) access for the originator. In Python it rests on convention (the leading-underscore "private" field) plus immutability (frozen=True).

When to Use It

  • You need undo / redo, rollback, or checkpoints / save points.
  • A snapshot of an object's state must be saved, but exposing its internals directly would break encapsulation.
  • You want a clean separation between the object that owns state and the object that stores its history.

Trade-offs

✓ Pros

  • Saves and restores state without breaking encapsulation.
  • Simplifies the originator — it doesn't manage its own history.
  • Clean foundation for undo/redo stacks and snapshots.

✗ Cons

  • Memory cost — each snapshot copies state; deep histories add up.
  • Caretakers must manage memento lifecycle (when to discard old ones).
  • Frequent saves of large objects can hurt performance.
  • In dynamic languages, the encapsulation boundary is convention, not enforced.
Pairs with Command for undo/redo. Memento captures what the state was; Command captures what action changed it. A common robust undo system stores a command together with a memento of the state before it ran — execute pushes onto the stack, undo pops and restores the memento. For huge states, consider storing an incremental diff in each memento instead of a full copy to cut memory cost.

Related Patterns

Command is the natural partner for building undo/redo systems. Prototype offers a simpler alternative when an object's whole state can be captured by cloning it — though a clone isn't opaque the way a memento is. State models the object's current behaviour as objects, whereas Memento preserves a past state for later restoration.

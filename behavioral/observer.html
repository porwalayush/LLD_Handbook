Behavioral Pattern

Observer

Define a one-to-many dependency between objects so that when one object changes state, all of its dependents are notified and updated automatically.

📺 Real-World Analogy

Think of a YouTube channel. When you subscribe, you've registered interest. Every time the creator uploads a new video, YouTube pushes a notification to every subscriber at once — the creator doesn't call each of you individually, and doesn't even know who you are. Unsubscribe and the notifications simply stop.

Newspaper subscriptions work the same way, and so does a spreadsheet: change one cell and every formula that depends on it recalculates instantly. The "subject" publishes; the "observers" react. That is exactly the Observer (a.k.a. publish/subscribe) pattern.

The Problem It Solves

Often, many objects need to stay in sync with the state of one other object. The naive approach hard-wires them together:

  • The subject keeps direct references to every dependent and calls each one by name — tight coupling.
  • Adding or removing a dependent means editing the subject's code every time.
  • The alternative — having dependents constantly poll "has it changed yet?" — wastes work and is always either too slow or too busy.

Observer decouples the two sides: the subject only knows it has a list of "observers" implementing a common interface. It can notify them all without knowing their concrete types, and observers can come and go freely.

How It Works — The Roles

  1. Subject (Publisher) — maintains a list of observers and exposes attach(), detach(), and notify(). When its state changes, it notifies everyone on the list.
  2. Observer — an interface with an update() method the subject calls.
  3. Concrete Observers — react to updates however they like (display, log, send an email…), each unaware of the others.
Push vs Pull. In the push model the subject sends the changed data straight into update(data) — simple, but the subject must guess what observers want. In the pull model it just signals "something changed" and each observer calls back to read only the fields it cares about — more flexible, slightly more coupling to the subject's interface. The example below uses push.

Implementation

A WeatherStation (the Subject) publishes temperature updates to any number of displays (the Observers). Notice how subscribing, unsubscribing, and notifying all happen through the interface — the station never names a concrete display.

import java.util.*;

// --- Observer: what every subscriber must implement ---
public interface Observer {
    void update(float temperature);   // push model: data is handed in
}

// --- Subject: manages subscribers and broadcasts changes ---
public class WeatherStation {
    private final List<Observer> observers = new ArrayList<>();
    private float temperature;

    public void subscribe(Observer o)   { observers.add(o); }
    public void unsubscribe(Observer o) { observers.remove(o); }

    private void notifyObservers() {
        // iterate a copy so an observer may safely unsubscribe mid-notify
        for (Observer o : new ArrayList<>(observers)) {
            o.update(temperature);
        }
    }

    public void setTemperature(float t) {   // state change triggers notify
        this.temperature = t;
        notifyObservers();
    }
}

// --- Concrete Observers ---
public class PhoneDisplay implements Observer {
    public void update(float temperature) {
        System.out.println("Phone shows " + temperature + "°C");
    }
}
public class LogDisplay implements Observer {
    public void update(float temperature) {
        System.out.println("Logged temperature: " + temperature);
    }
}

// Usage
WeatherStation station = new WeatherStation();
Observer phone = new PhoneDisplay();
station.subscribe(phone);
station.subscribe(new LogDisplay());
station.setTemperature(21.5f);     // both observers notified
station.unsubscribe(phone);
station.setTemperature(19.0f);     // only LogDisplay notified

from abc import ABC, abstractmethod

# --- Observer: what every subscriber must implement ---
class Observer(ABC):
    @abstractmethod
    def update(self, temperature: float) -> None:   # push model
        ...

# --- Subject: manages subscribers and broadcasts changes ---
class WeatherStation:
    def __init__(self):
        self._observers: list[Observer] = []
        self._temperature = 0.0

    def subscribe(self, observer: Observer) -> None:
        self._observers.append(observer)
    def unsubscribe(self, observer: Observer) -> None:
        self._observers.remove(observer)

    def _notify(self) -> None:
        # iterate a copy so an observer may unsubscribe mid-notify
        for observer in list(self._observers):
            observer.update(self._temperature)

    def set_temperature(self, t: float) -> None:   # state change -> notify
        self._temperature = t
        self._notify()

# --- Concrete Observers ---
class PhoneDisplay(Observer):
    def update(self, temperature: float) -> None:
        print(f"Phone shows {temperature}°C")

class LogDisplay(Observer):
    def update(self, temperature: float) -> None:
        print(f"Logged temperature: {temperature}")

# Usage
station = WeatherStation()
phone = PhoneDisplay()
station.subscribe(phone)
station.subscribe(LogDisplay())
station.set_temperature(21.5)        # both observers notified
station.unsubscribe(phone)
station.set_temperature(19.0)        # only LogDisplay notified

Where You've Already Seen It

Observer is everywhere in real frameworks. Java shipped java.util.Observable/Observer (now deprecated — it had design flaws, so prefer your own interface, PropertyChangeListener, or a library). GUI event listeners (addActionListener), event buses, message brokers, and the entire reactive world (RxJava, Reactor, RxJS) are large-scale Observer implementations.

When to Use It

  • A change to one object requires changing an unknown number of others, and you don't want them tightly coupled.
  • You're building a publish/subscribe or event-driven flow (UI events, domain events, live dashboards).
  • Observers should be able to join and leave at runtime without the subject knowing their concrete types.

Trade-offs

✓ Pros

  • Loose coupling: subject and observers vary independently.
  • Open/Closed: add new observer types without touching the subject.
  • Dynamic subscribe/unsubscribe at runtime.
  • Supports broadcast communication to many listeners.

✗ Cons

  • Notification order is not guaranteed — don't rely on it.
  • Memory leaks if observers forget to unsubscribe (the subject keeps them alive).
  • Cascade updates: one notify can trigger another, causing storms or loops.
  • Hard to debug: control flow jumps to many places implicitly.
Watch out for leaks. The classic Observer bug: a short-lived observer subscribes but never unsubscribes, so the long-lived subject holds a reference to it forever and it can never be garbage-collected. Always pair subscribe with unsubscribe (use weak references or a try/finally / lifecycle hook), and copy the observer list before iterating so an observer can detach itself mid-notification.
Interview tip: Name the players precisely — Subject, Observer interface, Concrete Observers — and call out subscribe/unsubscribe/notify. Bonus points: mention push vs pull, the deprecated java.util.Observable, and the two big gotchas (memory leaks and undefined notification order).

Related Patterns

Mediator also reduces coupling between many objects, but it centralizes communication in one hub rather than broadcasting one-to-many. Command pairs naturally with Observer — an event notification often carries a command to execute. A Facade simplifies a subsystem's interface; combine it with Observer when the facade needs to broadcast subsystem events to interested clients.

Behavioral Pattern

State

Let an object alter its behavior when its internal state changes. The object will appear to change its class — the same method does something different depending on which state it's in.

🚦 Real-World Analogy

Watch a traffic light. It cycles Red → Green → Yellow → Red, and "what happens next" depends entirely on the current light. The light fixture is one object, but it behaves like a different machine in each state — and each state knows which one comes after it.

A vending machine is the same: press "dispense" while it's idle and nothing happens, but press it after you've paid and a soda drops. Or a media player, where the single Play/Pause button starts the music when Stopped, pauses it when Playing, and resumes when Paused. One button, behavior driven by state.

The Problem It Solves

Objects with a lifecycle usually end up riddled with conditionals that check a status flag inside every method:

  • if (state == IDLE) {...} else if (state == HAS_MONEY) {...} else if (state == DISPENSING) {...}
  • The same sprawling switch is copy-pasted into every action, and each must remember every valid transition.
  • Adding one new state means hunting down and editing every method — easy to miss a case and create an illegal transition.

State replaces that tangle with one class per state. Each state class implements the actions for that state only and decides which state to transition into. The conditionals vanish, transitions become explicit, and adding a state means adding a class.

How It Works — The Roles

  1. Context — the object whose behavior changes (the vending machine). It holds a reference to the current State and forwards requests to it.
  2. State — an interface declaring the actions that vary by state (e.g. insertCoin(), dispense()).
  3. Concrete States — one per state. Each implements the actions for its state and, when appropriate, tells the context to switch to the next state.

Implementation

A VendingMachine (the Context) delegates insertCoin() and dispense() to its current state. The same two actions behave differently per state, and each state object decides the transition — no central conditional anywhere.

// --- State: the actions that vary by state ---
public interface State {
    void insertCoin(VendingMachine machine);
    void dispense(VendingMachine machine);
}

// --- Context: holds the current state, delegates to it ---
public class VendingMachine {
    private State state = new IdleState();   // start idle

    public void setState(State state) { this.state = state; }

    // same calls -> different behavior depending on current state
    public void insertCoin() { state.insertCoin(this); }
    public void dispense()   { state.dispense(this); }
}

// --- Concrete States: each implements + sets the next state ---
public class IdleState implements State {
    public void insertCoin(VendingMachine m) {
        System.out.println("Coin accepted.");
        m.setState(new HasCoinState());   // transition!
    }
    public void dispense(VendingMachine m) {
        System.out.println("Insert a coin first.");
    }
}

public class HasCoinState implements State {
    public void insertCoin(VendingMachine m) {
        System.out.println("Coin already inserted.");
    }
    public void dispense(VendingMachine m) {
        System.out.println("Dispensing your drink. Enjoy!");
        m.setState(new IdleState());      // back to idle
    }
}

// Usage
VendingMachine machine = new VendingMachine();
machine.dispense();      // "Insert a coin first."  (Idle)
machine.insertCoin();    // "Coin accepted."        -> HasCoin
machine.dispense();      // "Dispensing your drink." -> Idle

from __future__ import annotations
from abc import ABC, abstractmethod

# --- State: the actions that vary by state ---
class State(ABC):
    @abstractmethod
    def insert_coin(self, machine: VendingMachine) -> None: ...
    @abstractmethod
    def dispense(self, machine: VendingMachine) -> None: ...

# --- Context: holds the current state, delegates to it ---
class VendingMachine:
    def __init__(self):
        self._state: State = IdleState()    # start idle

    def set_state(self, state: State) -> None:
        self._state = state

    # same calls -> different behavior depending on current state
    def insert_coin(self) -> None: self._state.insert_coin(self)
    def dispense(self) -> None:    self._state.dispense(self)

# --- Concrete States: each implements + sets the next state ---
class IdleState(State):
    def insert_coin(self, machine: VendingMachine) -> None:
        print("Coin accepted.")
        machine.set_state(HasCoinState())     # transition!
    def dispense(self, machine: VendingMachine) -> None:
        print("Insert a coin first.")

class HasCoinState(State):
    def insert_coin(self, machine: VendingMachine) -> None:
        print("Coin already inserted.")
    def dispense(self, machine: VendingMachine) -> None:
        print("Dispensing your drink. Enjoy!")
        machine.set_state(IdleState())        # back to idle

# Usage
machine = VendingMachine()
machine.dispense()       # "Insert a coin first."  (Idle)
machine.insert_coin()    # "Coin accepted."        -> HasCoin
machine.dispense()       # "Dispensing your drink." -> Idle
Document workflows are a perfect fit. A doc that moves Draft → Moderation → Published is a state machine: in Draft, "publish" first sends it to Moderation; in Moderation, only a reviewer can push it to Published or back to Draft; once Published, editing it drops it back to Draft. Each rule lives in its own state class instead of one giant if.

When to Use It

  • An object's behavior depends on its state, and it must change behavior at runtime as that state changes.
  • Methods are full of large conditionals or switches on a state/status field, repeated across the class.
  • You have a genuine state machine with well-defined states and transitions (workflows, parsers, connections, game entities).

Trade-offs

✓ Pros

  • Eliminates big state-based conditionals.
  • Each state's logic is isolated in one class (Single Responsibility).
  • Transitions are explicit and easy to follow.
  • Open/Closed: add a new state without editing the others.

✗ Cons

  • More classes — overkill for two or three trivial states.
  • Transition logic is spread across state classes, so the full state diagram isn't visible in one place.
  • States may need references to the context and to each other.
State vs Strategy: They look identical on a class diagram — a context delegating to an interchangeable object — but the intent differs. In State, the objects are stages of a lifecycle and the transitions are driven internally: a state object decides which state comes next. In Strategy, the objects are independent algorithms and the client chooses which one to use; strategies don't swap themselves out. Rule of thumb: transitions from inside = State; chosen from outside = Strategy.

Related Patterns

Strategy shares State's structure but varies a client-chosen algorithm rather than modeling internal transitions. State objects are often stateless and shareable, so they're frequently kept as Singletons or flyweights to avoid re-creating them on every transition. Command pairs well with State when each transition should also be recorded or made undoable.

Behavioral Pattern

Strategy

Define a family of interchangeable algorithms, encapsulate each one behind a common interface, and let the client swap them at runtime without touching the code that uses them.

🗺️ Real-World Analogy

Open a navigation app and ask for directions. The same "find me a route" request can be answered by different strategies: a driving route that prefers highways, a walking route that cuts through parks, or a cycling route that avoids busy roads. The app's job — "get from A to B" — never changes; only the algorithm it plugs in does, and you pick it with a single tap.

Or think of checkout: the store knows it must "charge the customer", but how — credit card, PayPal, or UPI — is a strategy the shopper chooses at the last moment. Strategy is the code version of that swappable choice.

The Problem It Solves

Imagine a checkout class that does the payment itself with a growing chain of conditionals:

  • if (method == CREDIT_CARD) { ... } else if (method == PAYPAL) { ... } else if (method == UPI) { ... }
  • Every new payment method means editing and re-testing that one giant method.
  • The unrelated algorithms are tangled together in a single class, each able to break the others.

This violates the Open/Closed Principle: the class should be open to new behavior but closed to modification. Strategy fixes it by pulling each algorithm into its own class behind a shared interface. Adding a new method means writing a new class — the checkout code never changes. That is the OCP win.

How It Works — The Roles

  1. Strategy — an interface that declares the common operation (e.g. pay(amount)).
  2. Concrete Strategies — each class implements the operation differently (CreditCard, PayPal, UPI).
  3. Context — holds a reference to a Strategy and delegates the work to it. The client configures which strategy the context uses, and can replace it any time.

Implementation

Below, a ShoppingCart (the Context) delegates checkout to a PaymentStrategy. The cart neither knows nor cares how the payment happens — and the customer can swap strategies right up to the moment of checkout.

// --- Strategy: the common interface for every algorithm ---
public interface PaymentStrategy {
    void pay(int amount);
}

// --- Concrete Strategies: one class per algorithm ---
public class CreditCardPayment implements PaymentStrategy {
    private final String cardNumber;
    public CreditCardPayment(String cardNumber) { this.cardNumber = cardNumber; }
    public void pay(int amount) {
        System.out.println("Paid " + amount + " via credit card " + cardNumber);
    }
}

public class PayPalPayment implements PaymentStrategy {
    private final String email;
    public PayPalPayment(String email) { this.email = email; }
    public void pay(int amount) {
        System.out.println("Paid " + amount + " via PayPal " + email);
    }
}

public class UpiPayment implements PaymentStrategy {
    private final String vpa;        // e.g. user@bank
    public UpiPayment(String vpa) { this.vpa = vpa; }
    public void pay(int amount) {
        System.out.println("Paid " + amount + " via UPI " + vpa);
    }
}

// --- Context: delegates to whatever strategy it holds ---
public class ShoppingCart {
    private PaymentStrategy strategy;          // set or swapped at runtime

    public void setPaymentStrategy(PaymentStrategy strategy) {
        this.strategy = strategy;
    }
    public void checkout(int amount) {
        strategy.pay(amount);              // no if/else over payment types!
    }
}

// Usage
ShoppingCart cart = new ShoppingCart();
cart.setPaymentStrategy(new CreditCardPayment("1234-5678"));
cart.checkout(500);                     // Paid 500 via credit card 1234-5678

cart.setPaymentStrategy(new UpiPayment("alex@bank"));   // swap at runtime
cart.checkout(300);                     // Paid 300 via UPI alex@bank

from abc import ABC, abstractmethod

# --- Strategy: the common interface for every algorithm ---
class PaymentStrategy(ABC):
    @abstractmethod
    def pay(self, amount: int) -> None:
        ...

# --- Concrete Strategies: one class per algorithm ---
class CreditCardPayment(PaymentStrategy):
    def __init__(self, card_number: str):
        self.card_number = card_number
    def pay(self, amount: int) -> None:
        print(f"Paid {amount} via credit card {self.card_number}")

class PayPalPayment(PaymentStrategy):
    def __init__(self, email: str):
        self.email = email
    def pay(self, amount: int) -> None:
        print(f"Paid {amount} via PayPal {self.email}")

class UpiPayment(PaymentStrategy):
    def __init__(self, vpa: str):   # e.g. user@bank
        self.vpa = vpa
    def pay(self, amount: int) -> None:
        print(f"Paid {amount} via UPI {self.vpa}")

# --- Context: delegates to whatever strategy it holds ---
class ShoppingCart:
    def __init__(self, strategy: PaymentStrategy = None):
        self._strategy = strategy        # set or swapped at runtime

    def set_payment_strategy(self, strategy: PaymentStrategy) -> None:
        self._strategy = strategy
    def checkout(self, amount: int) -> None:
        self._strategy.pay(amount)         # no if/elif over payment types!

# Usage
cart = ShoppingCart()
cart.set_payment_strategy(CreditCardPayment("1234-5678"))
cart.checkout(500)                     # Paid 500 via credit card 1234-5678

cart.set_payment_strategy(UpiPayment("alex@bank"))   # swap at runtime
cart.checkout(300)                     # Paid 300 via UPI alex@bank

# Pythonic shortcut: a strategy can just be a plain function.
# Anything callable as pay(amount) works as a strategy here.
Pythonic note: Because functions are first-class objects in Python, a strategy is often just a function (or a lambda) passed in, with no interface or class hierarchy needed. Reach for the formal ABC-based version when strategies carry state or you want explicit, documented contracts.

When to Use It

  • You have several variants of an algorithm (sorting orders, pricing rules, compression schemes, routing) and want to pick one at runtime.
  • A class is cluttered with a large conditional that selects behavior based on a type or mode.
  • You want to isolate algorithm details from the code that uses them, so each can change independently.
  • You need to add new behaviors without modifying existing, tested code (Open/Closed Principle).

Trade-offs

✓ Pros

  • Swap algorithms at runtime.
  • Replaces sprawling conditionals with polymorphism.
  • Open/Closed: add strategies without editing the context.
  • Each algorithm is isolated and independently testable.

✗ Cons

  • More classes/objects to manage for simple cases.
  • The client must know the strategies to choose between them.
  • Overkill if behavior rarely changes — a plain conditional may be clearer.
Strategy vs State: The two patterns share the same structure — a context delegating to interchangeable objects — but differ in intent. In Strategy, the strategies are independent, interchangeable algorithms and the client chooses which one to plug in; they usually don't know about each other. In State, the objects represent stages of a lifecycle and drive internal transitions — one state decides which state comes next, with no client involvement. Rule of thumb: chosen from outside = Strategy; transitions from inside = State.

Related Patterns

State has the same shape but models internal transitions instead of client-chosen algorithms. Bridge uses a similar composition to separate an abstraction from its implementation, but it is a structural pattern decided at design time rather than swapped at runtime. Template Method solves a related problem with inheritance — fixing the skeleton of an algorithm and letting subclasses override steps — whereas Strategy varies the whole algorithm via composition.

Behavioral Pattern

Template Method

Define the skeleton of an algorithm in a base method, deferring some steps to subclasses — so subclasses can vary parts of the algorithm without changing its overall structure.

🍵 Real-World Analogy

Think of a recipe template for a hot drink: boil the water → add the main ingredient → steep/brew → pour into a cup → add condiments. The order of steps never changes. What changes is the detail: for tea you steep a tea bag and add lemon; for coffee you brew grounds and add sugar & milk.

The overall procedure is fixed (the "template"), but two specific steps differ per drink. That's exactly Template Method — like a building blueprint where the structure is set, but individual rooms can be customized.

The Problem It Solves

Sometimes several algorithms share the same overall structure but differ in a few specific steps. The naive approach is to copy-paste the whole procedure into each class and tweak the differing bits — which duplicates the structure everywhere:

  • A data import pipeline: open file → parse → validate → save. CSV and JSON differ only in how they parse.
  • A build/deploy script: checkout → compile → test → package → deploy. Each project differs only in the compile and test steps.
  • A game AI turn: collect resources → build structures → attack. Each faction differs only in how it builds.

Duplicating the skeleton is dangerous: if the structure changes (say you add a "log start" step), you must edit every copy and risk forgetting one. Template Method puts the structure in one place and lets subclasses fill in only the gaps.

How It Works

  1. An AbstractClass defines a templateMethod() that lays out the algorithm's steps in order. This method is usually final so subclasses can't break the structure.
  2. Some steps are concrete (shared by everyone) and live in the base class.
  3. The varying steps are abstract methods ("primitive operations") that subclasses must implement.
  4. Optional hooks are methods with an empty or default body that subclasses may override to plug into the algorithm.
  5. ConcreteClasses override the abstract steps (and optionally the hooks) — but never the template method itself.
The Hollywood Principle: "Don't call us, we'll call you." Your subclass never calls the high-level algorithm — the base class's templateMethod() calls your step implementations at the right moments. Control stays at the top; you just fill in the blanks.

Implementation

Below, a beverage maker fixes the brewing pipeline in the base class and lets Tea and Coffee customize only the steps that differ. Note the boilWater() / pourInCup() steps are shared, brew() / addCondiments() are abstract, and customerWantsCondiments() is an optional hook.

// --- AbstractClass: defines the skeleton ---
abstract class Beverage {

    // The TEMPLATE METHOD: final -> subclasses can't change the structure
    public final void prepare() {
        boilWater();          // shared step
        brew();               // varies -> abstract
        pourInCup();         // shared step
        if (customerWantsCondiments()) {  // hook decides
            addCondiments();   // varies -> abstract
        }
    }

    // Concrete steps shared by all beverages
    private void boilWater() { System.out.println("Boiling water"); }
    private void pourInCup() { System.out.println("Pouring into cup"); }

    // Primitive operations: subclasses MUST implement these
    protected abstract void brew();
    protected abstract void addCondiments();

    // Hook: default behaviour, subclasses MAY override
    protected boolean customerWantsCondiments() { return true; }
}

// --- ConcreteClass: fills in the gaps ---
class Tea extends Beverage {
    protected void brew() { System.out.println("Steeping the tea"); }
    protected void addCondiments() { System.out.println("Adding lemon"); }
}

class Coffee extends Beverage {
    protected void brew() { System.out.println("Brewing the coffee grounds"); }
    protected void addCondiments() { System.out.println("Adding sugar and milk"); }
    @Override protected boolean customerWantsCondiments() { return false; } // black coffee
}

// Usage
Beverage tea = new Tea();
tea.prepare();    // boil -> steep -> pour -> lemon
Beverage coffee = new Coffee();
coffee.prepare(); // boil -> brew -> pour (no condiments)

from abc import ABC, abstractmethod

# --- AbstractClass: defines the skeleton ---
class Beverage(ABC):

    # The TEMPLATE METHOD: lays out the fixed sequence of steps
    def prepare(self):
        self._boil_water()          # shared step
        self.brew()                 # varies -> abstract
        self._pour_in_cup()        # shared step
        if self.customer_wants_condiments():  # hook
            self.add_condiments()    # varies -> abstract

    # Concrete steps shared by all beverages
    def _boil_water(self): print("Boiling water")
    def _pour_in_cup(self): print("Pouring into cup")

    # Primitive operations: subclasses MUST implement
    @abstractmethod
    def brew(self): ...
    @abstractmethod
    def add_condiments(self): ...

    # Hook: default behaviour, subclasses MAY override
    def customer_wants_condiments(self):
        return True


# --- ConcreteClasses: fill in the gaps ---
class Tea(Beverage):
    def brew(self): print("Steeping the tea")
    def add_condiments(self): print("Adding lemon")

class Coffee(Beverage):
    def brew(self): print("Brewing the coffee grounds")
    def add_condiments(self): print("Adding sugar and milk")
    def customer_wants_condiments(self):
        return False   # black coffee


# Usage
Tea().prepare()      # boil -> steep -> pour -> lemon
Coffee().prepare()   # boil -> brew -> pour (no condiments)
Keep the template method closed. Mark it final in Java (or simply document "do not override" in Python). If subclasses can override the template method itself, they can reorder or skip steps — defeating the whole point of guaranteeing a fixed structure.

Template Method vs. Strategy

These two are easy to confuse — both let you vary part of an algorithm. The key difference is inheritance vs. composition:

Template MethodStrategy
MechanismInheritance — subclass overrides stepsComposition — inject a strategy object
What variesSelected steps within a fixed skeletonThe entire algorithm, swapped at runtime
BindingFixed at compile time (which subclass)Changeable at runtime (swap the object)

Rule of thumb: use Template Method when the structure is fixed and only a few steps differ; use Strategy when you want to swap whole interchangeable algorithms at runtime.

When to Use It

  • Several classes share an identical algorithm structure but differ in a handful of steps.
  • You want to localize the common structure in one place to avoid duplication.
  • You want to give subclasses controlled extension points (abstract steps + hooks) without letting them rewrite the whole flow.
  • You're building a framework and want to define the invariant parts once, letting users plug in the variable parts.

Trade-offs

✓ Pros

  • Eliminates duplicated structure — the skeleton lives in one place.
  • Subclasses only implement what actually differs.
  • Enforces a consistent algorithm flow across variants.
  • Hooks give flexible, optional extension points.

✗ Cons

  • Locks you into inheritance — one base class per hierarchy.
  • Behaviour is fixed at compile time; can't swap steps at runtime.
  • Can violate the Liskov principle if a subclass skips a required step.
  • Many hooks/steps can make the base class hard to follow.
Interview tip: When asked "how would you avoid duplicating this 5-step process across handlers?", reach for Template Method — then immediately contrast it with Strategy to show you understand inheritance vs. composition. Mention the Hollywood Principle by name; it signals you grasp inversion of control.

Related Patterns

Strategy solves a similar problem with composition instead of inheritance — swap whole algorithms at runtime. Facade also hides a multi-step process behind one call, but doesn't expose customizable steps. Factory Method is essentially a Template Method specialized for object creation — the base class calls an abstract "create" step that subclasses fill in.

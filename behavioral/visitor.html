Behavioral Pattern

Visitor

Represent an operation to be performed on the elements of an object structure. Visitor lets you add new operations without changing the classes of the elements on which they operate.

🏢 Real-World Analogy

Think of a tax auditor who visits different kinds of businesses — a restaurant, a factory, a software startup. The auditor knows how to compute taxes differently for each type of business, but the businesses themselves never change their day-to-day operations just because an auditor showed up. They simply let the auditor in ("accept" the visit), and the auditor does the type-specific work.

Next year a new kind of inspector arrives — say, a fire-safety inspector. The businesses don't change at all; you just send a different visitor through the same doors. That is exactly the Visitor pattern: the businesses are your element classes, and each inspector is a new operation you can add without touching the businesses.

The Problem It Solves

Imagine you have a stable set of element classes — for example geometric shapes (Circle, Rectangle, Triangle) or nodes in a file system. The set of types rarely changes, but you keep needing to run new operations across all of them:

  • Compute the area of every shape.
  • Export each shape to JSON, then later to XML, then to SVG.
  • Render, validate, price, serialize, optimize…

The naive approach is to add a new method to every element class each time a new operation appears. That means editing Circle, Rectangle, and Triangle for area, then editing all three again for export, and again for the next one. Each unrelated concern (geometry, export, rendering) gets smeared across every class, violating the Single Responsibility Principle and the Open/Closed Principle.

Visitor flips this around: it gathers one operation into a single class (the visitor). To add a new operation you write one new visitor class — and you never touch the element classes again.

The Roles

  • Visitor interface — declares one visit(...) method per concrete element type (visitCircle, visitRectangle, …).
  • ConcreteVisitor — implements those methods; one class per operation (e.g. AreaVisitor, ExportVisitor).
  • Element interface — declares a single accept(visitor) method.
  • ConcreteElement — implements accept by calling back the matching visit method on the visitor.
  • Object structure — the collection (list, tree) of elements that can be iterated and have a visitor applied to each.

The Key Idea: Double Dispatch

This is the heart of Visitor, and the part interviewers probe. In most object-oriented languages a method call is dispatched on one type — the runtime type of the object before the dot. But here we need to pick the right behavior based on two types at once: the concrete element (Circle vs. Rectangle) and the concrete visitor (Area vs. Export).

Visitor achieves this with two chained calls:

  1. First dispatch — you call element.accept(visitor). This resolves on the element's type, so the JVM/interpreter runs Circle's accept.
  2. Second dispatch — inside Circle.accept, the code calls visitor.visitCircle(this). Now this is statically known to be a Circle, and the call resolves on the visitor's type, so the right overload on the right visitor runs.

Why is the bounce necessary? Because if you only had visitor.visit(someElement), the language would only know someElement's declared type (often the base Shape), not its concrete subtype — so it couldn't pick visitCircle vs visitRectangle. The accept hop is what recovers the concrete element type, turning two single-dispatch calls into one double dispatch.

Implementation

Here we model two shapes — Circle and Rectangle — and two operations expressed as visitors: an AreaVisitor that computes area, and an ExportVisitor that produces a string. Notice that to add a third operation you only add a new visitor class; the shape classes stay frozen.

// --- Visitor interface: one method per concrete element ---
interface ShapeVisitor {
    double visitCircle(Circle c);
    double visitRectangle(Rectangle r);
}

// --- Element interface: the single accept() hook ---
interface Shape {
    double accept(ShapeVisitor v);
}

// --- Concrete elements: accept() bounces back to the visitor ---
class Circle implements Shape {
    final double radius;
    Circle(double radius) { this.radius = radius; }

    public double accept(ShapeVisitor v) {
        return v.visitCircle(this);   // 2nd dispatch: type of `this` is known
    }
}

class Rectangle implements Shape {
    final double w, h;
    Rectangle(double w, double h) { this.w = w; this.h = h; }

    public double accept(ShapeVisitor v) {
        return v.visitRectangle(this);
    }
}

// --- ConcreteVisitor #1: compute area ---
class AreaVisitor implements ShapeVisitor {
    public double visitCircle(Circle c) {
        return Math.PI * c.radius * c.radius;
    }
    public double visitRectangle(Rectangle r) {
        return r.w * r.h;
    }
}

// --- ConcreteVisitor #2: export to text (a NEW operation,
//     added WITHOUT editing Circle or Rectangle) ---
class ExportVisitor implements ShapeVisitor {
    StringBuilder out = new StringBuilder();
    public double visitCircle(Circle c) {
        out.append("<circle r=\"" + c.radius + "\"/>\n");
        return 0;
    }
    public double visitRectangle(Rectangle r) {
        out.append("<rect w=\"" + r.w + "\" h=\"" + r.h + "\"/>\n");
        return 0;
    }
}

// --- Usage: one structure, many operations ---
List<Shape> shapes = List.of(new Circle(2), new Rectangle(3, 4));

AreaVisitor area = new AreaVisitor();
double total = 0;
for (Shape s : shapes) total += s.accept(area);  // 1st dispatch
System.out.println("Total area: " + total);

ExportVisitor exp = new ExportVisitor();
for (Shape s : shapes) s.accept(exp);
System.out.print(exp.out);

from abc import ABC, abstractmethod
import math

# --- Element interface: the single accept() hook ---
class Shape(ABC):
    @abstractmethod
    def accept(self, visitor):
        ...

# --- Concrete elements: accept() bounces to the right visit method ---
class Circle(Shape):
    def __init__(self, radius):
        self.radius = radius
    def accept(self, visitor):
        return visitor.visit_circle(self)   # 2nd dispatch

class Rectangle(Shape):
    def __init__(self, w, h):
        self.w, self.h = w, h
    def accept(self, visitor):
        return visitor.visit_rectangle(self)

# --- Visitor interface ---
class ShapeVisitor(ABC):
    @abstractmethod
    def visit_circle(self, c): ...
    @abstractmethod
    def visit_rectangle(self, r): ...

# --- ConcreteVisitor #1: compute area ---
class AreaVisitor(ShapeVisitor):
    def visit_circle(self, c):
        return math.pi * c.radius ** 2
    def visit_rectangle(self, r):
        return r.w * r.h

# --- ConcreteVisitor #2: export to text (NEW op, shapes untouched) ---
class ExportVisitor(ShapeVisitor):
    def visit_circle(self, c):
        return f'<circle r="{c.radius}"/>'
    def visit_rectangle(self, r):
        return f'<rect w="{r.w}" h="{r.h}"/>'

# --- Usage ---
shapes = [Circle(2), Rectangle(3, 4)]

area = AreaVisitor()
print("Total area:", sum(s.accept(area) for s in shapes))

exp = ExportVisitor()
for s in shapes:
    print(s.accept(exp))


# --- Lighter Pythonic alternative: functools.singledispatch ---
# Python can dispatch on the argument type directly, so you often
# skip accept()/double dispatch entirely:
from functools import singledispatch

@singledispatch
def area(shape):
    raise NotImplementedError(shape)

@area.register
def _(shape: Circle):
    return math.pi * shape.radius ** 2

@area.register
def _(shape: Rectangle):
    return shape.w * shape.h

print(area(Circle(2)))   # dispatches on the runtime type
Tip: The phrase that wins interview points is "double dispatch via the accept callback." Be ready to explain that accept exists purely to recover the element's concrete type so the correct visit overload can be chosen. In Python, mention that functools.singledispatch (or singledispatchmethod) gives you single-dispatch on the argument type for free — a lighter alternative that avoids writing accept on every class.

When to Use It

  • You have a stable object structure (the element types rarely change) but you keep adding new operations over it.
  • You want to keep unrelated behaviors out of the element classes — e.g. pricing, export, and rendering each live in their own visitor.
  • You need to accumulate state while traversing a structure (a tree of AST nodes, a file-system hierarchy, a document model) and produce a single result.

The Big Trade-off

Visitor makes one direction easy and the opposite direction hard — the reverse of most object-oriented designs:

  • Adding a new operation is easy — write one new visitor class, touch nothing else.
  • Adding a new element type is hard — you must add a visit method to the visitor interface and every existing concrete visitor. A new Triangle forces edits to AreaVisitor, ExportVisitor, and any others.

So reach for Visitor when your types are stable and your operations grow. If it is the other way around — operations are fixed but you add types constantly — plain polymorphism (a method on each class) is the better fit.

Trade-offs

✓ Pros

  • Add new operations without changing element classes (Open/Closed).
  • Related behavior is gathered into one visitor, not scattered.
  • A visitor can accumulate state across a whole traversal.

✗ Cons

  • Adding a new element type forces changes to every visitor.
  • The accept boilerplate adds indirection that confuses newcomers.
  • Visitors may need access to element internals, breaking encapsulation.
  • Overkill for small or unstable hierarchies.

Related Patterns

Visitor is frequently applied to a Composite tree to run an operation over the whole hierarchy. An Iterator often supplies the elements that a visitor then processes. And like Strategy, a Visitor encapsulates an algorithm in its own object — the difference is that Strategy swaps one algorithm for one context, while Visitor applies one algorithm across many element types.

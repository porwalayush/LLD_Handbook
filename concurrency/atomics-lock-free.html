Atomics & Lock-Free

Do thread-safe updates without locks, using hardware-level atomic instructions so many threads can make progress at once.

🥤 Real-World Analogy

Picture a vending machine that does compare-and-swap. You read the price tag — it says $2. You go to insert your $2, but before you commit, the machine checks: "Does the tag still say $2?" If someone bumped it to $3 while you were fishing for coins, the machine rejects your attempt and you simply read the new price and try again. No one had to lock the whole machine while you decided.

It's the same as editing a shared Google Doc optimistically: you make your change assuming nothing shifted under you. If the document changed since you started, the editor detects the conflict and retries your edit on the fresh version. Nobody freezes the document for everyone else just so you can type.

The Problem It Solves

The obvious way to make shared data thread-safe is a lock: only one thread holds it at a time, so updates can't collide. That works, but it has real costs:

  • Blocking. While one thread holds the lock, every other thread that wants it goes to sleep and waits.
  • Context switches. Parking and waking threads is expensive — the OS has to save and restore state.
  • Deadlock. Two threads each waiting on a lock the other holds will wait forever.
  • Contention. Under heavy traffic a hot lock becomes a single-file bottleneck, even on a 32-core machine.

Atomics take a different route. They lean on a single CPU instruction that updates a value indivisibly — no other thread can observe a half-finished update. Because there's no lock to hold, threads never block each other; in the worst case they just retry. This is the foundation of lock-free programming, where the system as a whole always makes progress.

CAS: Compare-And-Swap

The single most important primitive is Compare-And-Swap (CAS). In one uninterruptible hardware step it says:

"If the value at this location still equals expected, replace it with newValue and report success. Otherwise change nothing and report failure."

That tiny atomic "if" is enough to build almost all lock-free code. The usual shape is a CAS retry loop: read the current value, compute the new value from it, then try to CAS it in. If someone else changed the value first, the CAS fails and you loop — re-reading the now-current value and recomputing. Eventually your swap lands. No thread is ever blocked; a thread only retries when it loses a race, which means some other thread won and made progress.

The ABA problem. CAS only checks that the value equals what you expected — not that it never changed. If a value goes from A → B → A while you weren't looking, your CAS sees "A" and happily succeeds, even though the world moved underneath you. This bites lock-free stacks/queues where a reused pointer looks identical but now points at recycled memory. The fix is a version stamp: pair the value with a counter (e.g. Java's AtomicStampedReference) so A-at-version-1 and A-at-version-3 are distinguishable.

The Tools

Java ships a rich java.util.concurrent.atomic toolkit built directly on CAS. Python is more limited — by design — so we are honest about what it actually offers.

  • Java AtomicInteger / AtomicLong — atomic counters. incrementAndGet() and compareAndSet(expected, new) never need a lock.
  • Java AtomicReference<T> — atomically swap an object reference; the basis for lock-free data structures.
  • Java LongAdder — a counter tuned for high contention. It spreads writes across multiple internal cells so threads rarely collide, then sums them on read. Prefer it over AtomicLong when many threads increment a shared counter (metrics, hit counts).
  • Python reality — Python has no rich public atomics library. The GIL makes some single bytecode operations effectively atomic, but compound operations like counter += 1 are not (they read, add, then store — three steps). Practical tools: a threading.Lock, itertools.count() for a thread-safe incrementing counter, the thread-safe queue.Queue, and multiprocessing.Value for shared state across processes. There are no public AtomicInteger-style classes.
import java.util.concurrent.atomic.AtomicInteger;
import java.util.concurrent.atomic.AtomicLong;
import java.util.concurrent.atomic.LongAdder;

// --- 1. AtomicInteger: a lock-free counter ---
AtomicInteger hits = new AtomicInteger(0);
int n = hits.incrementAndGet();   // atomic ++  -> returns new value
hits.addAndGet(5);                 // atomic += 5

// --- 2. A manual CAS retry loop: atomic "max" update ---
// There is no atomicMax in older JDKs, so we build it from compareAndSet.
static void updateMax(AtomicInteger max, int candidate) {
    while (true) {
        int current = max.get();            // read the latest value
        if (candidate <= current) return;  // nothing to do
        // try to swap current -> candidate; succeeds only if
        // nobody changed `max` since our read
        if (max.compareAndSet(current, candidate)) {
            return;                        // we won the race
        }
        // CAS failed -> someone else updated max; loop and retry
    }
}

// --- 3. AtomicReference: lock-free reference swap ---
AtomicReference<String> leader = new AtomicReference<>("none");
leader.compareAndSet("none", "node-A");  // claim leadership atomically

// --- 4. LongAdder: best for HIGH-contention counters ---
LongAdder requests = new LongAdder();
requests.increment();                  // many threads, almost no contention
long total = requests.sum();        // read the combined total

import threading
import itertools

# Python has NO rich public atomics library. `counter += 1` is NOT atomic:
# it READS, ADDs, then STOREs -- three bytecodes the scheduler can split.
# The GIL makes SOME single ops atomic, but never rely on that for += .

# --- Practical pattern 1: a Lock-guarded counter ---
class AtomicCounter:
    def __init__(self, start=0):
        self._value = start
        self._lock = threading.Lock()      # protects the read-modify-write

    def increment(self):
        with self._lock:                     # only one thread inside at a time
            self._value += 1
            return self._value

    def get(self):
        with self._lock:
            return self._value

counter = AtomicCounter()
counter.increment()                        # safe from many threads

# --- Practical pattern 2: itertools.count() for unique IDs ---
# count().__next__ is atomic at the C level -> handy lock-free ID source.
ids = itertools.count(1)
next_id = next(ids)                       # 1, then 2, then 3 ... thread-safe

# --- Note: Python 3.13+ free-threading (no-GIL) build ---
# Without the GIL, even "atomic-looking" ops lose their accidental safety,
# so explicit Locks (or queue.Queue) become REQUIRED, not optional.
# For CPU-bound shared counters across processes, use multiprocessing.Value.

When to Use It

  • Simple shared counters and flags — hit counts, sequence numbers, "started/stopped" booleans.
  • High-contention hot paths where lock blocking and context switches would dominate (use LongAdder).
  • Building lock-free data structures (stacks, queues) where the whole system must keep making progress even if one thread stalls.
  • Implementing optimistic "claim once" handoffs with compareAndSet (leader election, one-time initialization).

Trade-offs

✓ Pros

  • No blocking — threads never sleep waiting on a lock.
  • No deadlock (there's no lock to acquire in the wrong order).
  • Excellent throughput under contention.
  • System-wide progress guarantee (lock-freedom).

✗ Cons

  • Hard to reason about and easy to get subtly wrong.
  • The ABA problem needs version stamps to avoid.
  • Retry loops can livelock under extreme contention.
  • Only works for simple, single-variable state — not compound invariants.
Interview tip: The crisp rule is — atomics for single counters and flags; locks for compound invariants. If a correct update must touch two or more fields together (e.g. "transfer $10 from A to B, keeping the total constant"), a single CAS can't express that — reach for a lock. Mention CAS, the retry-loop shape, and the ABA problem and you've covered what most interviewers are listening for.

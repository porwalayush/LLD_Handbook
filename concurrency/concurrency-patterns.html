Concurrency

Concurrency Patterns

Battle-tested templates for writing correct concurrent code — reusable solutions to the recurring problems of sharing data and coordinating work across threads.

🚦 Real-World Analogy

Think about how a city stops thousands of independent drivers from crashing into each other. Engineers didn't reinvent a new rule for every intersection — they reach for a small toolbox of proven patterns: traffic lights let one direction proceed at a time, roundabouts let everyone keep moving without a central stop, one-way streets remove whole classes of head-on collisions, and roadworks signs warn drivers to slow before a hazard.

Concurrency patterns are exactly this kind of toolbox for code. Each one is a named, well-understood template that keeps many threads — your "drivers" — from colliding over shared memory. Learn the handful below and most multithreading problems become "which pattern fits?" instead of "how do I invent thread-safety from scratch?".

This page is a guided tour. For each pattern we cover what it is, why it helps, and when to reach for it — with focused code where the implementation detail matters. None of these are exotic; they are the everyday vocabulary of senior engineers and a frequent source of interview questions.

1. Thread-Safe Singleton

A Singleton guarantees exactly one instance of a class. In a single-threaded program that is trivial; under concurrency it is a classic trap, because two threads can both see "no instance yet" and each create one. The thread-safe Singleton patterns close that race.

When to use it: shared, expensive-to-build objects that must be unique — a configuration object, a connection pool, a registry, a logger.

The two idiomatic Java solutions are the Bill Pugh "initialization-on-demand holder" (a static nested class that the JVM loads lazily and initializes safely with zero locking) and the enum singleton. In Python, the simplest correct approach is a module-level object; when you need a class-based singleton created from multiple threads, guard creation with a threading.Lock.

// --- Bill Pugh: initialization-on-demand holder idiom ---
// The JVM guarantees a class is initialized exactly once, lazily,
// in a thread-safe way -> no synchronized, no volatile needed.
public class Config {

    private Config() {
        // expensive one-time setup runs only when getInstance() is first called
    }

    private static class Holder {
        // loaded only when Holder is first referenced (in getInstance)
        private static final Config INSTANCE = new Config();
    }

    public static Config getInstance() {
        return Holder.INSTANCE;   // triggers safe, lazy class init
    }
}

// --- The bullet-proof alternative: enum singleton ---
public enum Logger {
    INSTANCE;
    public void log(String msg) {
        System.out.println("[LOG] " + msg);
    }
}

// Usage
Config c = Config.getInstance();
Logger.INSTANCE.log("started");

import threading

# --- Lock-guarded, thread-safe class singleton ---
class Config:
    _instance = None
    _lock = threading.Lock()        # guards instance creation

    def __new__(cls):
        if cls._instance is None:           # fast path, no lock
            with cls._lock:                  # only racing threads pay the lock
                if cls._instance is None:   # re-check inside the lock
                    cls._instance = super().__new__(cls)
                    cls._instance.db_url = "postgresql://localhost/app"
        return cls._instance


# --- Most Pythonic: a module-level object ---
# config.py  ->  config = Config()   then  `from config import config`
# Module top-level code runs once under an import lock, so this is
# inherently thread-safe.

# Usage
a = Config()
b = Config()
assert a is b          # same object

2. Monitor Object

The Monitor Object pattern bundles together the shared state and the lock that protects it inside a single object, so that every method runs with mutual exclusion automatically. Callers never see the lock; they just call methods, and the object guarantees that only one thread is "inside" at a time.

Why it helps: it removes the biggest source of concurrency bugs — forgetting to acquire the right lock at every access site. Synchronization becomes a property of the object, not a discipline you must remember everywhere.

When to use it: any mutable object shared across threads where you want all access serialized — a bank account, a counter, a bounded buffer, an in-memory cache. In Java this is what synchronized methods give you; in Python you wrap a threading.Lock (or RLock) inside the class.

// Every public method is synchronized on `this`, so only one
// thread can mutate the balance at a time -> no lost updates.
public class BankAccount {
    private long balanceCents = 0;

    public synchronized void deposit(long cents) {
        balanceCents += cents;
    }

    public synchronized boolean withdraw(long cents) {
        if (cents > balanceCents) return false;
        balanceCents -= cents;
        return true;
    }

    public synchronized long getBalance() {
        return balanceCents;   // also synchronized -> sees latest value
    }
}

import threading

# The lock lives inside the object; callers just call methods.
class BankAccount:
    def __init__(self):
        self._balance = 0
        self._lock = threading.RLock()   # re-entrant, guards all access

    def deposit(self, cents):
        with self._lock:
            self._balance += cents

    def withdraw(self, cents):
        with self._lock:
            if cents > self._balance:
                return False
            self._balance -= cents
            return True

    def balance(self):
        with self._lock:
            return self._balance

3. Read-Copy-Update (RCU) / Copy-on-Write

When data is read far more often than it is written, locking on every read is wasteful — readers spend their time contending for a lock they almost never need to mutate behind. Copy-on-Write (and the closely related RCU technique) flips the cost: readers never block at all. A writer makes a private copy of the data, mutates the copy, then atomically swaps the shared reference to point at the new version. Readers already holding the old reference keep using a consistent, immutable snapshot until they finish.

When it shines: read-mostly, write-rarely structures — configuration that reloads occasionally, a set of event listeners, routing tables, feature flags. Java ships this as CopyOnWriteArrayList and CopyOnWriteArraySet; iterating them never throws ConcurrentModificationException because each iterator sees a frozen snapshot.

The trade-off: every write copies the whole structure, so writes are O(n) and produce garbage. Copy-on-Write is a poor choice for write-heavy data, but unbeatable for "many readers, occasional writer" scenarios where read latency is what you care about. 
import java.util.concurrent.CopyOnWriteArrayList;

// Read-mostly listener registry. Reads (notify) are lock-free;
// the rare add/remove copies the backing array internally.
public class EventBus {
    private final CopyOnWriteArrayList<Runnable> listeners
            = new CopyOnWriteArrayList<>();

    public void subscribe(Runnable r) { listeners.add(r); }      // writer: copies

    public void publish() {
        // reader: iterates a stable snapshot, never blocks, never throws
        for (Runnable r : listeners) r.run();
    }
}

import threading

# Writers build a new tuple and swap the reference under a lock.
# Readers just grab the current reference -> a stable snapshot.
class EventBus:
    def __init__(self):
        self._listeners = ()              # immutable snapshot
        self._write_lock = threading.Lock()

    def subscribe(self, fn):
        with self._write_lock:         # copy-on-write
            self._listeners = self._listeners + (fn,)

    def publish(self):
        for fn in self._listeners:      # lock-free read of a frozen tuple
            fn()

4. Double-Checked Locking

Double-checked locking is an optimization for lazy initialization: check whether the instance exists without a lock (the fast, common path), and only if it appears missing do you acquire the lock and check again before creating it. The second check prevents two threads that both passed the first check from each creating an instance.

The subtle, infamous part is that a naive version is broken. Without the volatile keyword on the instance field, the JVM (or CPU) is free to reorder the writes that happen during construction. A second thread can then observe a non-null but partially-constructed object — the reference is published before the object's fields are fully written — and use a half-built instance. Marking the field volatile forbids that reordering and guarantees a happens-before relationship, so any thread that sees the reference sees a fully-built object.

Prefer not to hand-roll it. Double-checked locking is correct only with volatile, and it is easy to get wrong. The Bill Pugh holder idiom shown in the Singleton section above achieves the same lazy, thread-safe initialization with no locking and no volatile — reach for it first. Know double-checked locking mainly so you can explain why the holder idiom is cleaner.

5. Producer-Consumer & Thread Pool

Two of the most important concurrency patterns get their own dedicated pages, but they belong on this map:

  • Producer-Consumer — decouple threads that create work from threads that process it using a shared, thread-safe BlockingQueue. Producers and consumers run at their own pace; the queue handles buffering and blocking when it is full or empty.
  • Thread Pool / Executor — reuse a fixed set of worker threads to run many short tasks, instead of paying the cost of creating a thread per task. The pool caps concurrency, queues overflow work, and is the foundation of nearly every server.

6. Future / Promise

A Future (also called a promise in some languages) is a placeholder for a result that will exist later. You hand off work to run asynchronously and immediately get back a Future; you can keep doing other things and later call get() to retrieve the result (blocking if it isn't ready), attach callbacks, or compose several Futures into a pipeline.

Why it helps: it lets you express "do this work in the background and let me continue" without manually managing threads, shared result variables, and the locks to read them safely. In Java, Future and the far richer CompletableFuture support chaining and combining; in Python, use concurrent.futures (thread/process pools) or asyncio for coroutine-based concurrency.

import java.util.concurrent.CompletableFuture;

// Run work asynchronously, then transform the result -- no
// manual threads, no shared mutable result variable to lock.
CompletableFuture<String> pipeline =
    CompletableFuture
        .supplyAsync(() -> fetchUserId())        // runs on a pool thread
        .thenApply(id -> "user-" + id)          // transform when ready
        .thenApply(String::toUpperCase);

String result = pipeline.join();              // block only when we need the value

from concurrent.futures import ThreadPoolExecutor

def fetch_user_id():
    return 42

with ThreadPoolExecutor() as pool:
    future = pool.submit(fetch_user_id)   # returns immediately
    # ... do other work here ...
    user_id = future.result()             # block only when the value is needed
    print(f"user-{user_id}")

# asyncio offers the same idea for coroutine-based concurrency:
#   result = await some_async_call()
The bugs these patterns exist to prevent. Concurrency goes wrong in a handful of classic ways:
  • Race condition — two threads touch shared data simultaneously and the result depends on timing (e.g. a lost update on counter++).
  • Deadlock — threads wait on each other in a cycle and none can proceed (acquire locks in a consistent global order to avoid).
  • Livelock — threads keep reacting to each other and changing state, but make no real progress.
  • Starvation — a thread is perpetually denied a resource because others keep winning it.
  • Visibility / stale reads — one thread's write is never seen by another due to caching/reordering (the bug volatile and proper synchronization fix).
Golden rule: prefer high-level constructs — BlockingQueue, ExecutorService, ConcurrentHashMap, atomics, CompletableFuture — over hand-rolled locks. They are correct, tested, and far harder to misuse.

When to Use Which

SituationRecommended Pattern
Exactly one shared, expensive instance neededThread-Safe Singleton (holder idiom / enum)
A mutable object accessed by many threadsMonitor Object (synchronized methods / lock-in-class)
Data read constantly, written rarelyCopy-on-Write / RCU (CopyOnWriteArrayList)
Lazy init of one field with minimal lockingBill Pugh holder (avoid hand-rolled double-checked locking)
Threads producing work faster/slower than consumersProducer-Consumer (BlockingQueue)
Many short tasks to run efficientlyThread Pool / Executor
A result computed asynchronously, needed laterFuture / Promise (CompletableFuture, asyncio)
A single counter or flag updated by many threadsAtomics / lock-free (AtomicLong, CAS)

Trade-offs

✓ Pros

  • Reusable, named solutions to recurring problems.
  • Encapsulate synchronization so callers can't misuse it.
  • Map directly to battle-tested library classes.
  • Shared vocabulary for design discussions and reviews.

✗ Cons

  • Picking the wrong pattern can hurt (e.g. Copy-on-Write on write-heavy data).
  • Still require understanding the memory model underneath.
  • Over-synchronizing kills the parallelism you wanted.
  • No pattern saves you from a bad lock-ordering design.
Interview tip: when asked "how would you make this thread-safe?", name the pattern before writing code — "this is a read-mostly structure, so I'd use Copy-on-Write" or "I'll make this a Monitor with a lock inside the class". Then mention you'd reach for java.util.concurrent primitives over raw synchronized, and call out the specific bug each choice prevents (lost update, stale read, deadlock). Showing you know why double-checked locking needs volatile — and that the holder idiom avoids the whole question — is a reliable way to stand out.

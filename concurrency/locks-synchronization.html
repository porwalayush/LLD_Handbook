Concurrency

Locks & Synchronization

Keeping shared data correct when many threads touch it at the same time.

🚻 Real-World Analogy

Picture a single-occupancy restroom with one key. Only the person holding the key may go in; everyone else waits in line outside. When that person comes out and hands back the key, the next person can enter. There is never any confusion about who is inside, because only one key exists.

A lock is exactly that key. The shared resource is the room. A thread must "grab the key" before touching the resource and "return the key" when done. Like a talking stick passed around a meeting — only the holder may speak; everyone else listens and waits their turn.

The Core Problem: Race Conditions

When two or more threads read and write the same data at the same time, and the final result depends on the exact timing of who got there first, you have a race condition. The bug is non-deterministic — it might pass a thousand tests and then corrupt data in production.

The classic example is the innocent-looking line count++. It looks like one step, but the CPU actually performs three:

  1. Read the current value of count from memory into a register.
  2. Modify it — add one in the register.
  3. Write the new value back to memory.

This is called a read-modify-write. Now imagine two threads both run count++ when count == 5, and their steps interleave:

TimeThread AThread Bcount
t1reads 5 5
t2 reads 55
t3computes 6, writes 6 6
t4 computes 6, writes 66

Two increments happened, but count ended at 6 instead of 7 — one update was silently lost. The fix is to make that read-modify-write atomic (indivisible) so no other thread can sneak in between the steps.

The Critical Section

The block of code that touches shared data and must not be run by more than one thread at a time is called a critical section. The whole job of locks is to enforce mutual exclusion (often shortened to "mutex"): at most one thread inside the critical section at any instant. Everyone else blocks until the holder leaves.

The Toolbox

Java

  • synchronized — every Java object has a built-in intrinsic monitor lock. A synchronized method locks on this (or the class for static methods); a synchronized (obj) { ... } block locks on a chosen object. The lock is released automatically when the block exits, even on exception. It is reentrant — the same thread can re-acquire a lock it already holds.
  • ReentrantLock — an explicit lock object. More flexible than synchronized: you must call lock() and unlock() yourself (always in a try/finally), but you also get tryLock() (give up instead of blocking forever), tryLock(timeout), interruptible locking, and an optional fairness policy (longest-waiting thread goes first).
  • ReadWriteLock — splits the lock into a read lock and a write lock. Many readers can hold the read lock at once (reading doesn't conflict), but the write lock is exclusive. Great when reads vastly outnumber writes.
  • volatile — guarantees visibility: a write by one thread is immediately seen by others (no stale cached copy). It does not provide atomicity — volatile count++ still races, because the increment is still three steps.

Python & the GIL

CPython has a Global Interpreter Lock (GIL): only one thread executes Python bytecode at a time. This means a single bytecode operation won't be interrupted, so some operations look safe. But count += 1 still compiles to multiple bytecodes (load, add, store), and the interpreter can switch threads between them — so it still races. The GIL protects the interpreter's internals, not your compound operations. Be honest about this in interviews: the GIL is not a substitute for locking.

  • threading.Lock — the basic mutex. Acquire with lock.acquire() / release with lock.release(), or far better, use with lock: which releases automatically.
  • threading.RLock — a reentrant lock; the same thread may acquire it multiple times (it tracks a count). Useful when a locked method calls another locked method.

Fixing the Unsafe Counter

Below: first the broken version, then the corrected one. In Java we show both synchronized and ReentrantLock (with the mandatory try/finally); in Python we use with lock:.

// --- UNSAFE: count++ is a read-modify-write, not atomic ---
class UnsafeCounter {
    private int count = 0;
    public void increment() { count++; }   // RACE: updates get lost
    public int get() { return count; }
}

// --- FIX 1: synchronized (intrinsic monitor lock) ---
class SyncCounter {
    private int count = 0;

    // only one thread can be inside this method at a time
    public synchronized void increment() { count++; }

    public int get() {
        synchronized (this) { return count; }   // block form
    }
}

// --- FIX 2: ReentrantLock (lock/unlock in try/finally) ---
import java.util.concurrent.locks.*;

class LockCounter {
    private final ReentrantLock lock = new ReentrantLock();  // new ReentrantLock(true) = fair
    private int count = 0;

    public void increment() {
        lock.lock();
        try {
            count++;                       // the critical section
        } finally {
            lock.unlock();              // ALWAYS release, even on exception
        }
    }

    public boolean tryIncrement() {
        if (!lock.tryLock()) return false;  // don't block — bail if busy
        try { count++; return true; }
        finally { lock.unlock(); }
    }
}

// --- ReadWriteLock: many readers OR one writer ---
class Cache {
    private final ReadWriteLock rw = new ReentrantReadWriteLock();
    private String value = "";

    public String read() {
        rw.readLock().lock();          // shared — many readers at once
        try { return value; } finally { rw.readLock().unlock(); }
    }
    public void write(String v) {
        rw.writeLock().lock();         // exclusive — one writer, no readers
        try { value = v; } finally { rw.writeLock().unlock(); }
    }
}

// --- volatile: visibility only (NOT atomicity) ---
private volatile boolean running = true;   // a flag other threads will see promptly

import threading

# --- UNSAFE: count += 1 is several bytecodes; the GIL won't save you ---
class UnsafeCounter:
    def __init__(self):
        self.count = 0
    def increment(self):
        self.count += 1          # RACE: load, add, store can interleave


# --- FIX: threading.Lock with the `with` context manager ---
class SafeCounter:
    def __init__(self):
        self.count = 0
        self._lock = threading.Lock()

    def increment(self):
        with self._lock:          # acquires on enter, releases on exit
            self.count += 1      # the critical section

    def get(self):
        with self._lock:
            return self.count


# --- RLock: reentrant — same thread can acquire it again ---
class Account:
    def __init__(self):
        self.balance = 0
        self._lock = threading.RLock()

    def deposit(self, amount):
        with self._lock:
            self.balance += amount

    def transfer_in(self, amount):
        with self._lock:        # already holding the lock here...
            self.deposit(amount)   # ...and deposit() grabs it again — OK with RLock


# --- tryLock equivalent: acquire(blocking=False) ---
lock = threading.Lock()
if lock.acquire(blocking=False):   # grab it only if free
    try:
        ...                          # do work
    finally:
        lock.release()
else:
    print("lock busy, skipping")
volatile is not a lock. A common mistake is thinking volatile count++ is thread-safe. It is not — volatile only guarantees that reads and writes are visible across threads, not that the read-modify-write happens atomically. For "make this number safe to increment", reach for a lock or an atomic type (see Atomics & Lock-Free), not volatile.

Deadlock

Locks solve race conditions but introduce a new hazard: deadlock — two or more threads each waiting forever for a lock the other holds. Nobody can proceed; the program hangs. Deadlock requires all four of the Coffman conditions to hold simultaneously:

  1. Mutual exclusion — at least one resource is held in a non-shareable way.
  2. Hold and wait — a thread holds one lock while waiting to acquire another.
  3. No preemption — a lock can't be forcibly taken away; it's released only voluntarily.
  4. Circular wait — a cycle exists: A waits on B, B waits on A (or a longer ring).

The classic trigger is two locks acquired in opposite order. Thread 1 grabs lock A then wants B; Thread 2 grabs lock B then wants A. Each holds what the other needs:

Object lockA = new Object(), lockB = new Object();

// Thread 1: A then B          // Thread 2: B then A  <-- opposite order = DEADLOCK
synchronized (lockA) {              synchronized (lockB) {
    synchronized (lockB) { ... }        synchronized (lockA) { ... }
}                                   }

// --- AVOIDANCE 1: global lock ordering (always A before B) ---
// Both threads acquire in the SAME order, so no cycle can form.
synchronized (lockA) {
    synchronized (lockB) { ... }
}

// --- AVOIDANCE 2: tryLock with timeout, back off and retry ---
if (lockA.tryLock(1, TimeUnit.SECONDS)) {
    try {
        if (lockB.tryLock(1, TimeUnit.SECONDS)) {
            try { ... } finally { lockB.unlock(); }
        }
    } finally { lockA.unlock(); }   // couldn't get both? release and retry
}

import threading

lock_a = threading.Lock()
lock_b = threading.Lock()

# Thread 1: A then B            # Thread 2: B then A  <-- opposite order = DEADLOCK
with lock_a:                     with lock_b:
    with lock_b:                      with lock_a:
        ...                              ...

# --- AVOIDANCE 1: always acquire in the SAME order ---
def acquire_in_order(first, second):
    with first:          # e.g. always lock_a before lock_b
        with second:
            ...

# --- AVOIDANCE 2: non-blocking acquire, back off and retry ---
def transfer():
    while True:
        if lock_a.acquire(timeout=1):
            try:
                if lock_b.acquire(timeout=1):
                    try:
                        ...
                        return
                    finally:
                        lock_b.release()
            finally:
                lock_a.release()   # release and loop to retry

Avoidance strategies break one of the four conditions. The most practical are: (1) lock ordering — every thread acquires locks in the same global order, breaking circular wait; and (2) tryLock with timeout — if you can't get all the locks you need, release what you hold and retry, breaking hold-and-wait. Keeping critical sections small and grabbing as few locks as possible also helps enormously.

When to Use It

  • Multiple threads read and write the same mutable state (counters, collections, caches, balances).
  • A sequence of operations must happen as one atomic, all-or-nothing unit (check-then-act, read-modify-write).
  • You need to coordinate access to a shared external resource (a file, a connection, a hardware device).
  • Use a ReadWriteLock specifically when reads heavily outnumber writes.

Trade-offs

✓ Pros

  • Simple, correct mutual exclusion for shared state.
  • Well understood; built into every language/runtime.
  • Flexible — fairness, timeouts, read/write split when you need them.

✗ Cons

  • Contention — threads queue up on a hot lock, serializing work and killing parallelism.
  • Deadlock (and livelock/starvation) risk from acquiring multiple locks.
  • Performance cost — acquiring/releasing and context-switching aren't free.
  • Easy to misuse: forgetting to unlock, locking on the wrong object, holding locks too long.
Interview tip: When asked "how do you make this thread-safe?", don't just say "add a lock." Name the race condition precisely (the read-modify-write on count++), define the critical section, then pick the lightest tool that works — synchronized or a single Lock for simple cases, ReadWriteLock for read-heavy data, atomics for a single counter. Always mention deadlock and that you'd prevent it with consistent lock ordering. Bonus points for noting that volatile gives visibility but not atomicity, and that Python's GIL does not make compound operations safe.

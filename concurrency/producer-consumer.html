Producer–Consumer

Decoupling threads that create work from threads that process it, via a shared buffer.

🍳 Real-World Analogy

Picture a busy restaurant kitchen. The cooks (producers) prepare dishes and set them on the pass — a counter with limited space. The waiters (consumers) pick up dishes from the pass and carry them to tables. The cooks and waiters never hand plates to each other directly; the pass sits between them so each side can work at its own pace.

Two natural rules emerge: if the pass is full, the cooks must pause until a waiter clears some space. If the pass is empty, the waiters must wait until a cook plates something. That waiting — and the shared counter in the middle — is the producer–consumer pattern.

The Bounded-Buffer Problem

This is the classic synchronization problem. You have one or more producer threads adding items to a shared, fixed-size buffer, and one or more consumer threads removing them. Three requirements must all hold at once:

  1. Producers block when the buffer is full — they can't overwrite or overflow it; they wait for space.
  2. Consumers block when the buffer is empty — there's nothing to take; they wait for an item.
  3. Mutual exclusion on the buffer — adding and removing both mutate shared state, so only one thread may touch the buffer at a time (otherwise you corrupt it — see Locks & Synchronization).

The bounded (fixed-size) buffer is what makes this interesting: it acts as a shock absorber. A burst of production gets buffered instead of lost, and a brief slowdown by consumers applies natural back-pressure on producers (they block), preventing unbounded memory growth.

Two Ways to Solve It

There are two solutions worth knowing. The first is what you should reach for in real code; the second is what you should be able to build from scratch in an interview.

1. Use a BlockingQueue (the idiomatic answer)

A thread-safe blocking queue already implements all three rules internally. You just call put/take and let it handle the locking and waiting.

  • Java: BlockingQueue — e.g. ArrayBlockingQueue (bounded). put() blocks when full; take() blocks when empty.
  • Python: queue.Queue(maxsize=N). put() blocks when full; get() blocks when empty.

2. Hand-roll it with a lock + condition variables

To show you understand what the queue does under the hood, you build it from a mutex plus condition variables. A condition variable lets a thread wait (releasing the lock and sleeping) until another thread signals that something changed (e.g. "an item was added"). In Java this is wait()/notify() on an object's monitor; in Python it's threading.Condition with wait()/notify().

Always wait inside a while loop, never an if. When a thread wakes from wait(), it must re-check the condition. Why? Spurious wakeups (the OS can wake a thread for no reason) and the fact that another thread may have grabbed the item first between the signal and your re-acquiring the lock. With if (empty) wait(); you might proceed on a buffer that's empty again and crash. With while (empty) wait(); you simply loop and wait again. This while-guard is the single most important detail of the hand-rolled version.

Implementation

Each tab shows both versions: the BlockingQueue / queue.Queue answer first, then the hand-rolled lock + condition version with the while-guard.

// ===== VERSION 1: BlockingQueue (idiomatic) =====
import java.util.concurrent.*;

BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(10);  // bounded, capacity 10

Runnable producer = () -> {
    try {
        for (int i = 0; i < 100; i++) {
            queue.put(i);            // blocks automatically if the queue is full
        }
    } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
};

Runnable consumer = () -> {
    try {
        while (true) {
            int item = queue.take();   // blocks automatically if the queue is empty
            process(item);
        }
    } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
};


// ===== VERSION 2: hand-rolled with lock + condition variables =====
class BoundedBuffer<T> {
    private final Queue<T> buf = new LinkedList<>();
    private final int capacity;

    public BoundedBuffer(int capacity) { this.capacity = capacity; }

    public synchronized void put(T item) throws InterruptedException {
        while (buf.size() == capacity) {   // WHILE, not if — re-check on wake
            wait();                       // release lock + sleep until notified
        }
        buf.add(item);
        notifyAll();                     // wake any waiting consumers
    }

    public synchronized T take() throws InterruptedException {
        while (buf.isEmpty()) {           // WHILE, not if
            wait();
        }
        T item = buf.poll();
        notifyAll();                     // wake any waiting producers
        return item;
    }
}

# ===== VERSION 1: queue.Queue (idiomatic) =====
import queue, threading

q = queue.Queue(maxsize=10)   # bounded, capacity 10

def producer():
    for i in range(100):
        q.put(i)              # blocks automatically if the queue is full

def consumer():
    while True:
        item = q.get()       # blocks automatically if the queue is empty
        process(item)
        q.task_done()         # optional: lets q.join() track completion


# ===== VERSION 2: hand-rolled with threading.Condition =====
from collections import deque

class BoundedBuffer:
    def __init__(self, capacity):
        self._buf = deque()
        self._capacity = capacity
        self._cond = threading.Condition()   # lock + wait/notify in one

    def put(self, item):
        with self._cond:                          # acquire the underlying lock
            while len(self._buf) == self._capacity:  # WHILE, not if
                self._cond.wait()                  # release lock + sleep until notified
            self._buf.append(item)
            self._cond.notify_all()                 # wake waiting consumers

    def take(self):
        with self._cond:
            while not self._buf:                   # WHILE, not if
                self._cond.wait()
            item = self._buf.popleft()
            self._cond.notify_all()                 # wake waiting producers
            return item
Why notifyAll over notify? With a single condition shared by both producers and consumers, notify() might wake the "wrong" kind of thread (a consumer that then sees an empty buffer and waits again, leaving a real waiter asleep — a lost-wakeup deadlock). notifyAll() wakes everyone; harmless threads simply re-check their while guard and go back to sleep. Use separate "not full" / "not empty" conditions if you want the precision of notify().

When to Use It

  • Work is generated and processed at different, variable rates and you want to smooth the mismatch with a buffer.
  • You want to decouple producers from consumers so each can scale and fail independently.
  • You need back-pressure — slow consumers should naturally throttle fast producers instead of exhausting memory.
  • Pipeline stages: the output of one stage feeds a queue that the next stage consumes.

Trade-offs

✓ Pros

  • Cleanly decouples producers and consumers.
  • Bounded buffer gives free back-pressure and bounded memory.
  • Easy to scale throughput — just add more consumer threads.
  • Buffer absorbs bursts and smooths uneven workloads.

✗ Cons

  • The shared buffer can become a contention point.
  • Hand-rolled versions are subtle — spurious wakeups, if-vs-while, lost notifications.
  • Adds latency (an item sits in the buffer) and tuning (what capacity?).
  • Needs a clean shutdown strategy (e.g. a "poison pill" sentinel to stop consumers).
Interview tip: Lead with "I'd use a BlockingQueue (ArrayBlockingQueue in Java, queue.Queue in Python) because it already handles the blocking and locking." Then offer to implement it by hand with a lock and condition variables to prove you understand the internals — and make a point of the while-loop guard against spurious wakeups. Finish by noting this pattern is the backbone of thread pools and work queues: an executor is just a fixed set of consumer threads draining a shared task queue that callers produce into.

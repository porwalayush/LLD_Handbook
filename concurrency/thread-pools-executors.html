Thread Pools & Executors

Reuse a fixed set of worker threads to run many tasks efficiently, instead of spawning a new thread for every job.

🏦 Real-World Analogy

Walk into a bank. There are, say, four tellers and a single queue of customers. When a teller finishes with one customer, they call the next person in line. The bank does not hire a brand-new teller for every customer who walks in and fire them at the counter — that would be absurdly slow and expensive. The same handful of tellers are reused all day long.

A thread pool is exactly that: a fixed crew of worker threads (the tellers) pulling tasks (the customers) off a shared queue. Tasks wait their turn; threads stay alive and get reused.

The Problem It Solves

The naive approach is to call new Thread() for every task. It works for a demo and falls apart in production:

  • Thread creation is expensive. Each thread costs a system call and a chunk of stack memory (often ~1 MB). Creating one per request burns time before any real work happens.
  • Unbounded threads exhaust the machine. If 10,000 requests arrive, you get 10,000 threads. The CPU thrashes between them (context-switch storm) and you run out of memory — frequently crashing the process.
  • No back-pressure. There's no natural limit; a traffic spike directly becomes a resource spike.

A thread pool fixes all three. It bounds concurrency to a sane number of workers, reuses those workers across many tasks, and holds the overflow in an internal task queue — giving you predictable resource use under load.

How It Works

In Java the abstraction is ExecutorService; the workhorse implementation is ThreadPoolExecutor. You hand it tasks and it schedules them onto its worker threads. The key tuning parameters of ThreadPoolExecutor are:

  • corePoolSize — the number of threads kept alive even when idle.
  • maximumPoolSize — the hard ceiling on threads, created only when the queue is full.
  • work queue — a BlockingQueue holding tasks waiting for a free worker.
  • keepAliveTime — how long threads above the core size linger idle before being reclaimed.
  • rejection policy — what to do when the queue is full and max threads are busy (e.g. AbortPolicy throws, CallerRunsPolicy runs it on the submitting thread).

You usually create one via the Executors factory — Executors.newFixedThreadPool(n) for a fixed crew. You submit work as a Runnable (returns nothing) or a Callable (returns a value and may throw). submit() immediately hands back a Future — a placeholder for the eventual result; call future.get() to block until it's ready. For composing async pipelines without blocking, CompletableFuture lets you chain steps (thenApply, thenCompose) and combine results.

When you're done you must shut the pool down, or its threads keep the JVM alive forever. shutdown() is graceful — it stops accepting new tasks but lets queued ones finish. shutdownNow() is abrupt — it tries to interrupt running tasks and returns the ones never started.

import java.util.concurrent.*;
import java.util.*;

// A fixed pool of 4 worker threads, reused for all tasks.
ExecutorService pool = Executors.newFixedThreadPool(4);

List<Future<Integer>> futures = new ArrayList<>();

for (int i = 1; i <= 8; i++) {
    final int id = i;
    // submit() takes a Callable and returns a Future right away
    Future<Integer> f = pool.submit(() -> {
        Thread.sleep(100);          // pretend to do real work
        return id * id;             // Callable returns a value
    });
    futures.add(f);
}

// Collect the results (get() blocks until each task finishes)
long total = 0;
for (Future<Integer> f : futures) {
    total += f.get();              // may throw if the task failed
}
System.out.println("sum of squares = " + total);

// --- Always shut the pool down (graceful) ---
pool.shutdown();                              // stop accepting new tasks
if (!pool.awaitTermination(10, TimeUnit.SECONDS)) {
    pool.shutdownNow();                       // force-stop stragglers
}

// --- Bonus: CompletableFuture chains work without blocking ---
CompletableFuture
    .supplyAsync(() -> 21, pool)
    .thenApply(x -> x * 2)
    .thenAccept(r -> System.out.println("result = " + r));

from concurrent.futures import ThreadPoolExecutor, as_completed
import time

def work(n):
    time.sleep(0.1)          # pretend to do real work
    return n * n

# The `with` block creates the pool and shuts it down cleanly on exit.
with ThreadPoolExecutor(max_workers=4) as pool:

    # --- submit(): returns a Future for each task ---
    futures = [pool.submit(work, i) for i in range(1, 9)]

    total = 0
    for f in as_completed(futures):  # results as they finish
        total += f.result()           # .result() blocks until ready
    print(f"sum of squares = {total}")

    # --- map(): simpler when you just want ordered results ---
    squares = list(pool.map(work, range(1, 5)))
    print(squares)               # [1, 4, 9, 16]

# Leaving the `with` block calls shutdown() and waits for tasks to finish.

# --- Note: threads share the GIL, so they help IO-bound work but NOT
# CPU-bound work. For CPU-bound jobs, swap in ProcessPoolExecutor --
# same API, but each worker is a separate process that sidesteps the GIL.

Designing Your Own Bounded Worker Pool

You won't build one in production — but interviewers love asking, because it ties the whole concurrency toolkit together. The recipe is small:

  1. Create a shared BlockingQueue of tasks with a fixed capacity. The capacity is your back-pressure: when it's full, submitters wait.
  2. Start a fixed number of worker threads. Each runs the same loop forever: take a task off the queue (blocking if empty), run it, repeat.
  3. To submit work, callers put a task on the queue (blocking if full).
  4. To shut down, push a sentinel "poison pill" per worker (or set a flag and interrupt) so each loop exits cleanly.

That blocking queue between producers (submitters) and consumers (workers) is exactly the Producer–Consumer pattern — the queue decouples how fast work arrives from how fast it's processed, and naturally throttles both sides.

When to Use It

  • You have many short-lived tasks (web requests, jobs, messages) and want to cap how many run at once.
  • You need predictable resource usage and back-pressure under load instead of unbounded thread growth.
  • You want to run work asynchronously and collect results later via Future / CompletableFuture.
  • Essentially any time you're tempted to write new Thread() inside a loop — use a pool instead.

Trade-offs

✓ Pros

  • Reuses threads — no per-task creation cost.
  • Bounds concurrency and memory; built-in back-pressure.
  • Clean async API with Future results.
  • Decouples task submission from execution.

✗ Cons

  • Pool sizing is tricky — too small starves, too big thrashes.
  • A blocking/long task can hog a worker and starve others.
  • Unbounded queues can hide overload and grow until OOM.
  • Forgetting to shut down leaks threads and hangs the JVM.
Interview tip: Know how to size the pool. For CPU-bound work, threads ≈ the number of cores (more just adds context-switch overhead). For IO-bound work (network, disk, DB), go higher — threads spend most of their time waiting, so you want many in flight to keep the CPU busy. And always state that you'd shut the pool down (and prefer a bounded queue with an explicit rejection policy over an unbounded one).

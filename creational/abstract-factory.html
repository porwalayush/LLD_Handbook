Creational Pattern

Abstract Factory

Provide an interface for creating families of related or dependent objects without specifying their concrete classes.

🖥️ Real-World Analogy

Think of a cross-platform UI toolkit. When your app runs on Windows it should show Windows-style buttons, checkboxes, and scrollbars; on macOS it should show Mac-style ones. Crucially, you never want to mix them — a Windows button sitting next to a Mac checkbox would look broken.

So you hire one of two "furniture makers" — a WindowsFactory or a MacFactory. You ask your chosen factory for "a button" and "a checkbox", and it always hands back a matching set in one consistent style. The rest of your app talks to plain "Button" and "Checkbox" — it never knows or cares which OS it's on.

The Problem It Solves

Sometimes objects are not created in isolation — they come in families that must be used together and stay visually or behaviourally consistent. If your code creates each piece with a bare new WindowsButton() here and new MacCheckbox() there, two things go wrong:

  • Inconsistent families. Nothing stops you from accidentally pairing a Windows button with a Mac checkbox. The "look" of your app drifts out of sync.
  • Concrete coupling everywhere. The new calls for a specific platform get scattered across the whole codebase. Adding a third theme (say, a Linux skin) means hunting down and editing every one of those sites.

Abstract Factory fixes both. It bundles the creation of a whole family behind a single interface. You pick one concrete factory at startup, and from then on every product it produces is guaranteed to belong to the same family. Client code depends only on the abstract products and the abstract factory — never on a concrete class.

How It Works

There are four moving parts:

  1. AbstractFactory — an interface declaring one creation method per product (e.g. createButton(), createCheckbox()).
  2. ConcreteFactories — one per family (WindowsFactory, MacFactory). Each implements the methods to return that family's variants.
  3. AbstractProducts — interfaces for each kind of product (Button, Checkbox) that the client codes against.
  4. ConcreteProducts — the actual variants (WindowsButton, MacButton, WindowsCheckbox, MacCheckbox). A given factory only ever produces products from its own family.

The client receives a factory (usually injected at startup), then calls the abstract creation methods. Because it only ever sees the abstract types, swapping the entire look-and-feel of the app is a one-line change: hand it a different factory.

How is this different from Factory Method?
  • Factory Method creates one product, and it varies that product through inheritance — a subclass overrides a single createX() method. It answers "which one object do I build?"
  • Abstract Factory creates a whole family of related products, and it varies them through composition — the client holds a factory object and is handed a different one to switch families. It answers "which matching set of objects do I build?"
  • In practice an Abstract Factory's individual createX() methods are often implemented as Factory Methods. Abstract Factory is essentially Factory Method scaled up to coordinate several related products at once.

Implementation

Below is the classic cross-platform UI example. Notice how the renderApp() client function takes any GUIFactory and is completely blind to the concrete classes — the only place the platform is decided is the one line that picks the factory.

// --- Abstract Products: interfaces the client codes against ---
interface Button   { String render(); }
interface Checkbox { String render(); }

// --- Concrete Products: the Windows family ---
class WindowsButton implements Button {
    public String render() { return "[ Windows Button ]"; }
}
class WindowsCheckbox implements Checkbox {
    public String render() { return "[x] Windows Checkbox"; }
}

// --- Concrete Products: the macOS family ---
class MacButton implements Button {
    public String render() { return "( macOS Button )"; }
}
class MacCheckbox implements Checkbox {
    public String render() { return "(✓) macOS Checkbox"; }
}

// --- Abstract Factory: one create method per product ---
interface GUIFactory {
    Button   createButton();
    Checkbox createCheckbox();
}

// --- Concrete Factories: each builds ONE consistent family ---
class WindowsFactory implements GUIFactory {
    public Button   createButton()   { return new WindowsButton(); }
    public Checkbox createCheckbox() { return new WindowsCheckbox(); }
}
class MacFactory implements GUIFactory {
    public Button   createButton()   { return new MacButton(); }
    public Checkbox createCheckbox() { return new MacCheckbox(); }
}

// --- Client: depends only on the abstractions ---
class Application {
    private final Button button;
    private final Checkbox checkbox;

    // the factory is injected -> the family is decided in one place
    Application(GUIFactory factory) {
        this.button   = factory.createButton();
        this.checkbox = factory.createCheckbox();
    }
    void render() {
        System.out.println(button.render() + "  " + checkbox.render());
    }
}

// --- Wiring: pick the factory once, at startup ---
GUIFactory factory = os.equals("mac")
        ? new MacFactory() : new WindowsFactory();
new Application(factory).render();   // always a matching set

from abc import ABC, abstractmethod

# --- Abstract Products: interfaces the client codes against ---
class Button(ABC):
    @abstractmethod
    def render(self) -> str: ...

class Checkbox(ABC):
    @abstractmethod
    def render(self) -> str: ...

# --- Concrete Products: the Windows family ---
class WindowsButton(Button):
    def render(self): return "[ Windows Button ]"

class WindowsCheckbox(Checkbox):
    def render(self): return "[x] Windows Checkbox"

# --- Concrete Products: the macOS family ---
class MacButton(Button):
    def render(self): return "( macOS Button )"

class MacCheckbox(Checkbox):
    def render(self): return "(✓) macOS Checkbox"

# --- Abstract Factory: one create method per product ---
class GUIFactory(ABC):
    @abstractmethod
    def create_button(self) -> Button: ...
    @abstractmethod
    def create_checkbox(self) -> Checkbox: ...

# --- Concrete Factories: each builds ONE consistent family ---
class WindowsFactory(GUIFactory):
    def create_button(self):   return WindowsButton()
    def create_checkbox(self): return WindowsCheckbox()

class MacFactory(GUIFactory):
    def create_button(self):   return MacButton()
    def create_checkbox(self): return MacCheckbox()

# --- Client: depends only on the abstractions ---
def render_app(factory: GUIFactory):
    button   = factory.create_button()
    checkbox = factory.create_checkbox()
    print(f"{button.render()}  {checkbox.render()}")

# --- Wiring: pick the factory once, at startup ---
os = "mac"
factory = MacFactory() if os == "mac" else WindowsFactory()
render_app(factory)          # always a matching set
Adding a new product is the painful part. Abstract Factory makes it cheap to add a whole new family (just write one new concrete factory). But adding a new kind of product — say a Slider — forces you to change the GUIFactory interface and every concrete factory that implements it. Lock down your set of product types before reaching for this pattern.

When to Use It

  • Your system must work with multiple families of related products (themes, OS skins, database backends, document formats) and you want to swap the whole family at once.
  • Products in a family must be used together and you need to enforce that they stay consistent — no mismatched mixing.
  • You want to isolate client code from concrete product classes so a new family can be plugged in without touching the client.
  • You can fix the set of product kinds up front (it changes far less often than the set of families).

Trade-offs

✓ Pros

  • Guarantees products from a family are mutually compatible.
  • Isolates concrete classes — clients depend only on interfaces (loose coupling).
  • Swapping an entire family is a one-line change.
  • Supports the Open/Closed Principle for adding new families.

✗ Cons

  • Lots of new interfaces and classes — significant boilerplate.
  • Adding a new product kind breaks every existing factory.
  • Can be overkill when you only have one family or one product.
  • Indirection makes the code harder to follow at first glance.
Interview tip: The single most common question is "how does Abstract Factory differ from Factory Method?" Nail it in one sentence: Factory Method makes one product and varies it through inheritance; Abstract Factory makes a family of products and varies it through composition (you swap the factory object). Bonus points for noting that each method inside an Abstract Factory is usually itself a Factory Method, and that the pattern's weakness is adding a new product type forces every factory to change.

Related Patterns

Factory Method is the building block — Abstract Factory's individual creation methods are usually Factory Methods, scaled up to coordinate a whole family. Builder focuses on constructing one complex object step-by-step, whereas Abstract Factory returns finished products immediately. Concrete factories are themselves frequently implemented as a Singleton, since you typically need just one per family.

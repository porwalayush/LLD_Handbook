Creational Pattern

Builder

Separate the construction of a complex object from its representation, so the same step-by-step construction process can create different representations.

🥪 Real-World Analogy

Think of ordering a custom sub at Subway. You don't shout the whole order in one breath. You walk down the line and build it step by step: first pick the bread, then the protein, then add cheese, pile on the veggies, and finish with sauces. Each step is optional and independent — skip the cheese, double the veggies, no sauce. At the end you say "that's it," and you get a finished sandwich.

The Builder pattern is the code version of that line: you add one part at a time through small, readable steps, and only at the final build() call do you receive the finished, complete object.

The Problem It Solves

Imagine a Pizza with lots of optional choices: size, crust, cheese, extra cheese, pepperoni, mushrooms, olives, stuffed crust. Without Builder, you reach for the telescoping constructor anti-pattern — a family of constructors with more and more parameters:

// Telescoping constructors — each one adds another parameter
new Pizza(12);
new Pizza(12, "thin");
new Pizza(12, "thin", true);
new Pizza(12, "thin", true, false, true, false, true);
//             size  crust  cheese  extra  pepperoni  mush  olives  ... wait, which is which?!

That last line is a disaster:

  • Unreadable. What does true, false, true mean? You have to count arguments against the signature to find out.
  • Error-prone. Swap two booleans and it still compiles — now your pizza silently has olives instead of mushrooms.
  • Explodes combinatorially. Every new optional field forces you to add yet another overloaded constructor (or pass a long line of nulls).
  • No required-vs-optional clarity. Nothing signals which arguments you actually have to provide.

Builder fixes all of this. Each value is set through a named, chainable step, optional parts are simply omitted, and the object is only handed over — fully assembled — when you call build().

How It Works

The classic GoF Builder has four roles. In everyday Java/Python code the Director is often skipped, so don't worry if you mostly see the other three.

  • Product — the complex object being assembled (e.g. Pizza). Often made immutable: all fields final, set once in the constructor, no setters.
  • Builder — defines the steps to build the product. In Java this is usually a static nested class with one fluent setter per field.
  • ConcreteBuilder — an actual implementation of those steps that accumulates state and produces the product in build(). (With a single nested builder, the Builder and ConcreteBuilder collapse into one class.)
  • Director (optional) — knows a recipe: it calls the builder's steps in a fixed order to produce a standard configuration (e.g. buildMargherita()). Useful when you reuse the same construction sequence.

The magic ingredient is the fluent interface: every step returns this (the builder itself), so calls can be chained:

new Pizza.Builder().setSize("L").addCheese().addPepperoni().build()

Each call mutates the builder and returns it, letting the next call hang right off the end. The product itself stays immutable — only the lightweight builder is mutable while you assemble.

Implementation

Java uses the classic static nested Builder: fluent setters return this, and build() produces an immutable Pizza whose private constructor only the builder can call. Python can do the same fluent style, but the most idiomatic Python alternative is a @dataclass with keyword arguments and defaults — shown at the bottom.

// --- Product: immutable, built only through its Builder ---
public final class Pizza {
    private final String size;        // required
    private final boolean cheese;       // optional parts
    private final boolean pepperoni;
    private final boolean mushrooms;

    // private: only Builder.build() can construct a Pizza
    private Pizza(Builder b) {
        this.size      = b.size;
        this.cheese    = b.cheese;
        this.pepperoni = b.pepperoni;
        this.mushrooms = b.mushrooms;
    }

    // no setters -> the object is immutable once built
    public String getSize() { return size; }

    // --- Static nested Builder ---
    public static class Builder {
        private final String size;     // required -> taken in builder ctor
        private boolean cheese    = false;
        private boolean pepperoni = false;
        private boolean mushrooms = false;

        public Builder(String size) {   // required field is mandatory
            this.size = size;
        }

        // each step returns `this` -> enables chaining (fluent interface)
        public Builder addCheese()    { this.cheese = true;    return this; }
        public Builder addPepperoni() { this.pepperoni = true; return this; }
        public Builder addMushrooms() { this.mushrooms = true; return this; }

        public Pizza build() {
            // validate invariants here before creating the product
            if (size == null) throw new IllegalStateException("size required");
            return new Pizza(this);   // hand back the finished, immutable object
        }
    }
}

// --- Optional Director: encapsulates a reusable recipe ---
public class PizzaDirector {
    public Pizza margherita() {
        return new Pizza.Builder("M").addCheese().build();
    }
}

// --- Usage: readable, order-independent, optional parts omitted ---
Pizza p = new Pizza.Builder("L")
        .addCheese()
        .addPepperoni()
        .build();
// vs. the awful  new Pizza("L", true, true, false, ...)

# --- Approach 1: a fluent Builder (mirrors the Java version) ---
class Pizza:
    # the immutable-ish product; only Builder should construct it
    def __init__(self, size, cheese, pepperoni, mushrooms):
        self.size      = size
        self.cheese    = cheese
        self.pepperoni = pepperoni
        self.mushrooms = mushrooms

    def __repr__(self):
        parts = [t for t, on in [
            ("cheese", self.cheese), ("pepperoni", self.pepperoni),
            ("mushrooms", self.mushrooms)] if on]
        return f"Pizza({self.size}, {parts})"


class PizzaBuilder:
    def __init__(self, size):       # required field
        self.size = size
        self.cheese = self.pepperoni = self.mushrooms = False

    # each step returns self -> enables chaining
    def add_cheese(self):    self.cheese = True;    return self
    def add_pepperoni(self): self.pepperoni = True; return self
    def add_mushrooms(self): self.mushrooms = True; return self

    def build(self):
        if not self.size:
            raise ValueError("size required")
        return Pizza(self.size, self.cheese, self.pepperoni, self.mushrooms)


# Usage
pizza = (PizzaBuilder("L")
         .add_cheese()
         .add_pepperoni()
         .build())
print(pizza)          # Pizza(L, ['cheese', 'pepperoni'])


# --- Approach 2 (most Pythonic): a @dataclass with kwargs + defaults ---
from dataclasses import dataclass

@dataclass(frozen=True)        # frozen -> immutable
class Pizza:
    size: str                   # required (no default)
    cheese: bool = False         # optional, named, self-documenting
    pepperoni: bool = False
    mushrooms: bool = False

# Keyword args make Python rarely *need* a builder class:
pizza = Pizza(size="L", cheese=True, pepperoni=True)
Why build() matters: The product stays incomplete (and unusable) while the builder accumulates parts. Only build() validates the invariants and produces the final, immutable object — so you can never accidentally pass around a half-constructed Pizza.

When to Use It

  • An object has many constructor parameters, especially when several are optional — the telescoping-constructor smell.
  • You want the resulting object to be immutable but still flexible to configure.
  • Construction needs to happen in multiple steps, or with validation that runs once at the end in build().
  • The same construction process should yield different representations (e.g. an HtmlReport vs PdfReport from one Director recipe).

Trade-offs

✓ Pros

  • Readable, self-documenting construction calls.
  • Kills the telescoping-constructor explosion.
  • Produces immutable, fully-valid objects.
  • Optional parameters are simply omitted.
  • Centralizes validation in build().

✗ Cons

  • More boilerplate — a whole extra Builder class per product.
  • Overkill for objects with only a few fields.
  • The builder is mutable while you assemble.
  • Two objects to maintain in sync as fields change.
Interview tip: Frame Builder as the cure for the telescoping constructor and as a way to get immutable objects with a fluent, readable API. Mention that the steps return this to enable chaining, and that the Director is optional. Big bonus points: note that in the Java ecosystem you rarely hand-write this — Lombok's @Builder generates the whole nested builder for you. In Python, mention that a @dataclass with keyword arguments and defaults usually makes a dedicated builder unnecessary.

Related Patterns

Builder is often confused with the factories. A Factory Method or Abstract Factory creates an object in one step and focuses on which concrete type to make; Builder constructs an object over many steps and focuses on how it's assembled. Builders (and factories) are themselves frequently implemented as a Singleton. Abstract Factory can use a Builder to construct its complex products.

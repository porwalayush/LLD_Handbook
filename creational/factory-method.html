Creational Pattern

Factory Method

Define an interface for creating an object, but let subclasses decide which class to instantiate — deferring instantiation to subclasses.

🚚 Real-World Analogy

Imagine a logistics company. Its core process is always the same: pick up the goods, plan the route, deliver, and bill the customer. But how a delivery happens depends on the situation — a road logistics branch dispatches a Truck, while a sea logistics branch dispatches a Ship.

The head office never says "build me a truck" directly. It just calls "create a transport" and trusts the right branch to hand back the right vehicle. The planning, dispatching, and billing logic stays identical — only the one step that decides which transport to build varies. That single overridable step is the factory method.

The Problem It Solves

You start with code that creates objects directly with new EmailNotification() scattered across your business logic. It works — until requirements grow:

  • A new channel arrives (SMS, then Push), and now every new EmailNotification() call site needs an if/else branch.
  • The high-level workflow (validate → render template → send → log) is now tightly coupled to concrete classes it shouldn't know about.
  • Adding a channel means editing existing, tested code — violating the Open/Closed Principle.

The root issue: the client code knows too much about which concrete class to build. Factory Method fixes this by pulling the new out of the client and hiding it behind a single overridable method. The client works only against an interface; subclasses supply the concrete product.

How It Works

The pattern has four roles working together:

  1. Product — an interface (e.g. Notification) that defines what every concrete object can do.
  2. Concrete Products — the real implementations (EmailNotification, SmsNotification, PushNotification).
  3. Creator — an abstract class declaring the factoryMethod() that returns a Product. Crucially, it also contains the shared business logic (send()) that uses the product without knowing its concrete type.
  4. Concrete Creators — subclasses that override factoryMethod() to return one specific product.

The magic is that the Creator's high-level code calls factoryMethod() as if it were a normal step. It never changes when you add a new product — you just add a new Concrete Creator.

Implementation

Both languages model the same four roles. Java uses an abstract Creator class whose concrete subclasses override createNotification(). Python expresses the same idea idiomatically with abc.ABC and an @abstractmethod; the shared send() workflow lives on the base creator and never touches a concrete class.

// --- Product: the interface every channel implements ---
public interface Notification {
    void notifyUser(String message);
}

// --- Concrete Products ---
public class EmailNotification implements Notification {
    public void notifyUser(String message) {
        System.out.println("[EMAIL] " + message);
    }
}
public class SmsNotification implements Notification {
    public void notifyUser(String message) {
        System.out.println("[SMS] " + message);
    }
}
public class PushNotification implements Notification {
    public void notifyUser(String message) {
        System.out.println("[PUSH] " + message);
    }
}

// --- Creator: declares the factory method + shared workflow ---
public abstract class NotificationCreator {
    // The factory method — subclasses decide the concrete class.
    protected abstract Notification createNotification();

    // High-level business logic that never knows the concrete type.
    public void send(String message) {
        Notification n = createNotification();   // deferred to subclass
        // ...validate, render template, retry, log... (shared)
        n.notifyUser(message);
    }
}

// --- Concrete Creators: each overrides only the factory method ---
public class EmailCreator extends NotificationCreator {
    protected Notification createNotification() { return new EmailNotification(); }
}
public class SmsCreator extends NotificationCreator {
    protected Notification createNotification() { return new SmsNotification(); }
}

// --- Client usage: works against the Creator, not concrete classes ---
NotificationCreator creator = new EmailCreator();   // chosen at runtime/config
creator.send("Your order has shipped!");
// To add Push: write PushCreator. No existing code changes.

from abc import ABC, abstractmethod

# --- Product: the interface every channel implements ---
class Notification(ABC):
    @abstractmethod
    def notify_user(self, message: str) -> None: ...

# --- Concrete Products ---
class EmailNotification(Notification):
    def notify_user(self, message):
        print(f"[EMAIL] {message}")

class SmsNotification(Notification):
    def notify_user(self, message):
        print(f"[SMS] {message}")

class PushNotification(Notification):
    def notify_user(self, message):
        print(f"[PUSH] {message}")

# --- Creator: declares the factory method + shared workflow ---
class NotificationCreator(ABC):
    @abstractmethod
    def create_notification(self) -> Notification:
        """Factory method: subclasses decide the concrete class."""

    def send(self, message):
        notification = self.create_notification()   # deferred to subclass
        # ...validate, render template, retry, log... (shared)
        notification.notify_user(message)

# --- Concrete Creators: each overrides only the factory method ---
class EmailCreator(NotificationCreator):
    def create_notification(self): return EmailNotification()

class SmsCreator(NotificationCreator):
    def create_notification(self): return SmsNotification()

# --- Client usage: works against the Creator, not concrete classes ---
creator: NotificationCreator = EmailCreator()   # chosen at runtime/config
creator.send("Your order has shipped!")
# To add Push: write PushCreator. No existing code changes.
Factory Method vs. Simple Factory. A "simple factory" is just one method with a big switch on a type string — handy, but not the GoF pattern and it still violates Open/Closed (you edit the switch for every new type). True Factory Method uses subclassing and polymorphism: a new product means a new subclass, with no edits to existing code.

When to Use It

  • You don't know the exact types your code will work with ahead of time, and you want it open to new ones.
  • A class wants its subclasses to specify the objects it creates (framework hooks like createButton()).
  • You want to localize the "which class to build" decision in one overridable spot instead of scattering new across the codebase.
  • Object creation involves shared setup (logging, caching, pooling) that you want to reuse around the varying step.

Trade-offs

✓ Pros

  • Decouples client code from concrete product classes.
  • Adds new products without touching existing code (Open/Closed).
  • Single Responsibility — creation logic lives in one place.
  • Easy to test: inject a creator that builds a fake/mock product.

✗ Cons

  • Introduces many subclasses — one creator per product can bloat the hierarchy.
  • Adds indirection that can feel like overkill for two stable types.
  • Requires a Creator hierarchy even if you only vary one method.
Interview tip: Lead with the intent — "let subclasses decide which class to instantiate" — and stress that the client depends only on the interface. Distinguish it from Abstract Factory (which creates families of related products) and from a plain switch-based simple factory. Bonus points: mention that Factory Method satisfies the Open/Closed Principle and is the classic answer to "how do I add a new type without editing existing code?"

Related Patterns

Abstract Factory is often implemented with a set of Factory Methods — use it when you need to create whole families of products. Builder focuses on constructing one complex object step by step rather than choosing among types. And factories themselves are frequently exposed as a Singleton so the whole app shares one creator.

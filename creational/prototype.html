Creational Pattern

Prototype

Create new objects by cloning an existing instance (the prototype) instead of constructing them from scratch.

🖨️ Real-World Analogy

Think of a rubber stamp (or a photocopier). You spend effort once to carve a detailed stamp — the logo, the address, the fine print. After that, you don't re-carve the design every time you need it. You just press the stamp and get an identical copy in an instant, then tweak the date or signature on each one.

Prototype works the same way: build one fully-configured object once, then clone it whenever you need another — far cheaper than rebuilding the whole thing each time.

The Problem It Solves

Sometimes creating an object from scratch is expensive or awkward:

  • The object's initial state comes from a slow source — a database query, a network call, or reading a large file.
  • Building it requires heavy computation (parsing, decompressing, laying out a scene).
  • You only know the concrete type at runtime and don't want a giant switch over every class just to call the right constructor.
  • You need many near-identical objects — like spawning 200 of the same configured game enemy — and re-running setup 200 times is wasteful.

Prototype sidesteps all of this: do the expensive setup once to produce a prototype, then ask the object to copy itself with clone(). The clone arrives pre-populated; you only adjust the few fields that differ.

How It Works

  1. Prototype interface — declares a clone() method that returns a copy of the object.
  2. Concrete prototypes — each class implements clone(), knowing how to copy its own fields correctly.
  3. Client — holds a reference to a prototype and produces new objects by calling clone() on it, never the constructor.
  4. Prototype registry (optional) — a small lookup (e.g. a Map<String, Prototype>) of ready-made prototypes. Ask the registry for "orc" or "dragon" and it returns a fresh clone of the matching template.

The Heart of It: Shallow vs Deep Copy

This is the part interviewers care about — and the part that bites people in production. When you clone an object, what happens to the fields inside it?

  • A shallow copy duplicates the top-level object, but nested objects are shared — both the original and the clone point at the same inner list/map/object in memory.
  • A deep copy recursively duplicates the nested objects too, so the clone owns its own independent copies.

For immutable fields (an int, a String) the distinction doesn't matter. But the moment a field is a mutable object — a list of tags, say — a shallow copy creates a hidden trap:

The shallow-copy bug. You clone a Document whose tags is ["draft"]. With a shallow copy, clone.tags and original.tags are the same list object. You call clone.tags.add("urgent") — and now original.tags is ["draft", "urgent"] too. You corrupted the original by mutating the copy. A deep copy gives the clone its own tags list, so the two never interfere.

Implementation

In Java you implement Cloneable and override clone() — the inherited Object.clone() does a shallow copy, so you must manually deep-copy any mutable field. In Python the standard library hands you both for free: copy.copy() is shallow, copy.deepcopy() is deep.

import java.util.*;

// A concrete prototype. Cloneable is the marker interface that
// lets Object.clone() succeed instead of throwing.
public class Document implements Cloneable {
    private String title;
    private List<String> tags;   // MUTABLE field -> needs care

    public Document(String title, List<String> tags) {
        this.title = title;
        this.tags = tags;
    }

    // --- DEEP copy: clone the list too, so clones are independent ---
    @Override
    public Document clone() {
        try {
            Document copy = (Document) super.clone();  // shallow: shares tags!
            copy.tags = new ArrayList<>(this.tags);   // fix it: own copy
            return copy;
        } catch (CloneNotSupportedException e) {
            throw new AssertionError(e);          // unreachable
        }
    }

    public List<String> getTags() { return tags; }
}

// --- Optional: a registry of ready-made prototypes ---
public class DocRegistry {
    private final Map<String, Document> protos = new HashMap<>();
    public void register(String key, Document p) { protos.put(key, p); }
    public Document create(String key) { return protos.get(key).clone(); }
}

// Usage
Document original = new Document("Report",
        new ArrayList<>(List.of("draft")));
Document copy = original.clone();
copy.getTags().add("urgent");
// original.getTags() == [draft]   (UNAFFECTED, thanks to deep copy)
// copy.getTags()     == [draft, urgent]

import copy

class Document:
    def __init__(self, title, tags):
        self.title = title
        self.tags = tags          # MUTABLE list


original = Document("Report", ["draft"])

# --- Shallow copy: copies the Document, but SHARES the tags list ---
shallow = copy.copy(original)
shallow.tags.append("urgent")
print(original.tags)   # ['draft', 'urgent']  <- BUG: original corrupted!
print(shallow.tags is original.tags)   # True - same list object

# --- Deep copy: recursively copies nested objects too ---
original = Document("Report", ["draft"])
deep = copy.deepcopy(original)
deep.tags.append("urgent")
print(original.tags)   # ['draft']            <- SAFE: untouched
print(deep.tags is original.tags)      # False - independent lists


# --- Customizing the clone + an optional prototype registry ---
class Enemy:
    def __init__(self, kind, loadout):
        self.kind = kind
        self.loadout = loadout    # mutable list

    def clone(self):
        return copy.deepcopy(self)   # safe, independent copy

registry = {"orc": Enemy("orc", ["axe"])}
goblin = registry["orc"].clone()   # cheap spawn from template
Java gotcha: the built-in Cloneable / Object.clone() mechanism is widely considered awkward (it's a marker interface, returns Object, and clones shallowly by default). Many teams skip it entirely and write a copy constructor new Document(other) or a static copyOf(...) factory instead — same Prototype intent, cleaner code.

When to Use It

  • Object creation is expensive (DB load, heavy parsing, network setup) and you need many similar instances.
  • You want to decouple the client from concrete classes — clone whatever prototype you're handed without knowing its exact type.
  • You need to snapshot an object's current configured state and stamp out variations of it.
  • The set of possible objects is configured at runtime (e.g. loaded from data) rather than hard-coded into a factory.

Trade-offs

✓ Pros

  • Skips costly initialization — clone instead of rebuild.
  • Adds/removes prototypes at runtime; no class explosion in a factory.
  • Client works against the prototype interface, not concrete types.
  • Great for producing variations of a complex, configured object.

✗ Cons

  • Deep-copying complex object graphs is tricky — especially with circular references, where a naive deep copy recurses forever.
  • Easy to forget a mutable field and ship a shallow-copy bug.
  • Every concrete class must implement clone() correctly.
  • Cloning objects with non-copyable resources (open sockets, file handles) needs special handling.
Interview tip: The instant you mention Prototype, expect "shallow or deep?" Have the bug ready: "A shallow copy shares the nested mutable fields, so mutating the clone corrupts the original — I'd deep-copy the mutable members." Bonus points for noting that deep-copying circular graphs needs an identity map of already-copied objects (which is exactly what Python's deepcopy does under the hood via its memo dict).

Related Patterns

Prototype is an alternative to the factory patterns: where Factory Method and Abstract Factory create objects by calling constructors, Prototype creates them by copying. A factory or registry is often implemented with prototypes, and that registry itself is frequently a Singleton.

Creational Pattern

Singleton

Ensure a class has exactly one instance, and provide a single, global point of access to it.

🌍 Real-World Analogy

Think of the government of a country. No matter how many times you ask "who governs here?", there is only ever one official government in charge at a time. You can't accidentally spin up a second one. Everyone — citizens, courts, businesses — refers to that same single authority.

A Singleton is the code version of that: the first time anyone asks for the object, it's created; every request after that hands back the exact same object.

The Problem It Solves

Some things should genuinely exist only once in an application. Creating more than one copy would be wasteful, confusing, or outright dangerous:

  • A database connection pool — you want one shared pool, not a new pool per call.
  • An application configuration object loaded from disk once at startup.
  • A logger writing to a single file — multiple writers would garble the output.
  • A hardware device driver or a print spooler managing one physical resource.

Two requirements fall out of this: (1) guarantee that only one instance ever exists, and (2) give everyone an easy, consistent way to reach it. A plain global variable solves access but not the "only one" guarantee — anyone could still call the constructor again. Singleton solves both by making the constructor private and exposing a controlled getInstance() accessor.

How It Works

  1. Make the constructor private so no outside code can call new on it.
  2. Hold the single instance in a static field inside the class.
  3. Expose a static getInstance() method that creates the instance on first call and returns the cached one thereafter.

Implementation

The two languages take slightly different routes. Java's idiomatic, bullet-proof Singleton is an enum (thread-safe and serialization-safe for free); we also show the classic thread-safe lazy version. Python rarely needs a strict Singleton — a module-level object is the Pythonic default — but the __new__ and metaclass approaches give you a true one-per-class guarantee.

// --- Thread-safe lazy Singleton (double-checked locking) ---
public class Config {
    // volatile guarantees other threads see the fully-built object
    private static volatile Config instance;

    private String dbUrl;

    // private constructor -> nobody outside can call new Config()
    private Config() {
        // expensive one-time setup: read file, open pool, etc.
        this.dbUrl = "jdbc:postgresql://localhost/app";
    }

    public static Config getInstance() {
        if (instance == null) {                 // 1st check (no lock, fast)
            synchronized (Config.class) {
                if (instance == null) {         // 2nd check (with lock)
                    instance = new Config();
                }
            }
        }
        return instance;
    }

    public String getDbUrl() { return dbUrl; }
}

// --- The bullet-proof idiom: enum Singleton (Effective Java) ---
public enum Logger {
    INSTANCE;                                // the one and only instance
    public void log(String msg) {
        System.out.println("[LOG] " + msg);
    }
}

// Usage
Config c1 = Config.getInstance();
Config c2 = Config.getInstance();
// c1 == c2  -> true  (same object)
Logger.INSTANCE.log("app started");

# --- Approach 1: override __new__ (true one-per-class) ---
class Config:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._init_once()       # one-time setup
        return cls._instance

    def _init_once(self):
        self.db_url = "postgresql://localhost/app"


# --- Approach 2: a metaclass (reusable across many classes) ---
class SingletonMeta(type):
    _instances = {}
    def __call__(cls, *args, **kwargs):
        if cls not in cls._instances:
            cls._instances[cls] = super().__call__(*args, **kwargs)
        return cls._instances[cls]

class Logger(metaclass=SingletonMeta):
    def log(self, msg):
        print(f"[LOG] {msg}")


# --- Approach 3 (most Pythonic): just use a module-level object ---
# config.py  ->  config = Config()   then  `from config import config`

# Usage
c1 = Config()
c2 = Config()
assert c1 is c2          # True - same object
Logger().log("app started")
Thread-safety matters. In a multithreaded app, a naive lazy Singleton can create two instances if two threads hit getInstance() at the same time. Use the volatile + double-checked locking shown above, an enum (Java), or eager initialization. In Python, the GIL helps but does not fully protect __new__ — guard with a threading.Lock if instances are created from multiple threads.

When to Use It

  • Exactly one instance must coordinate access to a shared resource (config, logger, connection pool, cache).
  • You need a single, well-known access point reachable from anywhere without passing it around.
  • The instance is expensive to create and should be built lazily, once.

Trade-offs

✓ Pros

  • Guarantees a single instance.
  • Global access point, lazily initialized.
  • Saves memory for heavy shared objects.

✗ Cons

  • Acts as a global — hidden dependencies, hard to track.
  • Makes unit testing harder (global state leaks between tests).
  • Can mask bad design / violate the Single Responsibility Principle.
  • Needs care to stay thread-safe.
Interview tip: Singleton is the most asked — and most criticized — pattern. Score points by mentioning its downsides (global state, testability) and that you'd prefer dependency injection to pass a single shared instance explicitly, getting the "one instance" benefit without the global. Then show you can still implement a thread-safe version when asked.

Related Patterns

Factories (Factory Method, Abstract Factory) and Builder objects are themselves often implemented as Singletons. Facade objects are frequently Singletons too.

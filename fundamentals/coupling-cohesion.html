Coupling, Cohesion & Composition

The forces that decide how maintainable your design is.

🧰 Real-World Analogy

Picture two workshops. In the first, every tool lives in a clearly labelled drawer — screwdrivers together, wrenches together. That's high cohesion: things that belong together are kept together. In the second workshop, someone has hot-glued the hammer to the drill to the saw. Pick up one and the whole tangled mess comes with it. That's tight coupling: you can't touch one part without disturbing the rest.

Good design aims for the first workshop: each class does one focused job (high cohesion) and connects to others through clean, snap-together joints rather than glue (loose coupling).

1. Cohesion — How Focused a Class Is

Cohesion measures how strongly the responsibilities inside a single class belong together. A class with high cohesion does one well-defined thing; every field and method serves that single purpose. A class with low cohesion is a junk drawer — it parses files, sends emails, and computes taxes all at once.

🧰 Think of it as

A well-organized toolbox: the "screwdriver tray" holds screwdrivers and nothing else. When you need a screwdriver, you know exactly where to look, and adding a new one doesn't disturb the wrenches.

  • High cohesion (good): an InvoiceCalculator that only calculates invoice totals. Easy to name, test, and reuse.
  • Low cohesion (bad): a Utils god-class that does string formatting, database access, and PDF generation. Hard to name precisely, hard to change safely.

A quick test: if you struggle to name a class without using "and" or "Manager"/"Utils", its cohesion is probably low.

2. Coupling — How Dependent Classes Are on Each Other

Coupling measures how much one class knows about and relies on another. Loose coupling (good) means classes interact through small, stable interfaces, so you can change one without breaking the other. Tight coupling (bad) means a change in one class forces changes in many others.

🧱 Think of it as

LEGO bricks vs. glued parts. LEGO bricks connect through a standard stud interface — you can swap any brick for another without touching its neighbours. Glued parts are fused: replace one and you must rebuild the whole assembly.

  • Tight: OrderService directly creates and calls a concrete MySqlOrderRepository. Switching to Postgres means editing OrderService.
  • Loose: OrderService depends on an OrderRepository interface. Any implementation — MySQL, Postgres, in-memory for tests — plugs in without touching the service.
The goal in one line: aim for high cohesion (each class focused) and loose coupling (classes weakly tied together). These two together are the single best predictor of how painful a codebase is to change.

3. Composition over Inheritance

This is the big one. Inheritance models an "is-a" relationship and is tempting because it reuses code for free. But deep inheritance trees create tight coupling to a parent: a subclass inherits everything, even things it shouldn't, and a change to the base class ripples into every descendant.

Consider modelling a car. A naive design says "a car needs an engine, so let Car extends Engine." But a car is not an engine — it has one. Inheritance here leaks the engine's whole API onto the car and makes it impossible to swap a petrol engine for an electric one at runtime.

Composition models a "has-a" relationship: the car holds a reference to an Engine object and delegates to it. Now you can swap engines freely, and the car only depends on the small Engine interface.

// BAD: inheritance abuse — a Car is NOT an Engine
// class Car extends Engine { ... }   // tightly coupled, can't swap engines

// GOOD: composition — a Car HAS-A Engine
interface Engine {
    void start();
}

class PetrolEngine implements Engine {
    public void start() { System.out.println("Vroom (petrol)"); }
}

class ElectricEngine implements Engine {
    public void start() { System.out.println("Hum (electric)"); }
}

class Car {
    private final Engine engine;   // HAS-A, not IS-A

    // engine injected -> loose coupling, easy to swap or mock
    Car(Engine engine) { this.engine = engine; }

    public void drive() {
        engine.start();           // delegate to the engine
        System.out.println("Driving...");
    }
}

// Usage: same Car, different engines — no Car code changes
Car petrol   = new Car(new PetrolEngine());
Car electric = new Car(new ElectricEngine());

from abc import ABC, abstractmethod

# GOOD: composition — a Car HAS-A Engine
class Engine(ABC):
    @abstractmethod
    def start(self): ...

class PetrolEngine(Engine):
    def start(self): print("Vroom (petrol)")

class ElectricEngine(Engine):
    def start(self): print("Hum (electric)")

class Car:
    def __init__(self, engine: Engine):
        self._engine = engine        # HAS-A, not IS-A

    def drive(self):
        self._engine.start()         # delegate to the engine
        print("Driving...")

# Usage: same Car, different engines — no Car code changes
petrol   = Car(PetrolEngine())
electric = Car(ElectricEngine())

✓ Composition

  • Swap behaviour at runtime (inject any Engine).
  • Loose coupling — Car only knows the small interface.
  • Easy to test (pass a mock engine).
  • No fragile base class ripples.

✗ Deep Inheritance

  • Tight coupling to the parent's internals.
  • Subclass inherits things it shouldn't.
  • Behaviour fixed at compile time.
  • One base-class change breaks many subclasses.

Rule of thumb: use inheritance only for a genuine, stable "is-a" relationship; reach for composition for almost everything else.

4. Association vs Aggregation vs Composition

These three UML relationships describe how objects connect, and they differ mainly in lifetime ownership — does the "whole" control whether the "part" lives or dies?

RelationshipMeaningLifetime / OwnershipExample
AssociationOne object uses or knows another ("uses-a")Independent — neither owns the otherA Teacher teaches Students; both exist on their own
AggregationA whole-part "has-a" where parts can exist aloneWeak ownership — parts outlive the wholeA Team has Players; delete the team, players still exist
CompositionA strong whole-part "owns-a"Strong ownership — parts die with the wholeA House has Rooms; destroy the house, the rooms go with it

In short: association is "I have your phone number," aggregation is "you're on my team for now," and composition is "you're a part of me and can't exist without me."

5. Law of Demeter — "Don't Talk to Strangers"

The Law of Demeter (LoD), or principle of least knowledge, says a method should only talk to its immediate friends — not to the friends of its friends. Concretely, avoid long reach-through chains like a.getB().getC().doX().

Each .get() in that chain ties your code to the internal structure of another object. If B stops holding a C, your call breaks even though you never touched B or C directly. That's tight coupling sneaking in through dot-chains.

  • Bad: order.getCustomer().getAddress().getZip() — Order now depends on Customer's and Address's internals.
  • Better: ask the object to do the work for you — order.getShippingZip() — and let Order figure out where the zip lives.

The fix is "tell, don't ask": tell an object what you want done rather than pulling out its guts and doing it yourself. This keeps coupling loose and cohesion high — the same two goals as everything above.

Interview tip: When asked to critique a design, reach for this vocabulary: "This class has low cohesion — it mixes persistence with business logic," or "These two classes are tightly coupled because the service news-up a concrete class; I'd depend on an interface instead." Naming high cohesion + loose coupling and citing composition over inheritance signals senior-level design instincts and almost always scores points.

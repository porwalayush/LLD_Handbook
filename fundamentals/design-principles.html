Core Design Principles

The rules of thumb that keep code clean beyond SOLID — DRY, KISS, YAGNI & GRASP.

🍳 Real-World Analogy

A great chef doesn't memorize a thousand recipes — they internalize a handful of cooking principles: taste as you go, don't crowd the pan, season in layers. Those rules of thumb let them handle any dish.

These design principles are the same idea for code. They aren't rigid laws like a compiler error; they're seasoned guidelines that, applied with judgment, keep your design simple, change-friendly, and free of needless repetition.

DRY — Don't Repeat Yourself

Every piece of knowledge should have a single, authoritative home in the code. When the same logic is copy-pasted into five places, a bug fix has to be made five times — and you'll inevitably miss one. Extract the duplication into one named method or class.

// BEFORE: the same tax math duplicated everywhere
double invoiceTotal = price + price * 0.18;     // 18% GST
double receiptTotal = amount + amount * 0.18;   // repeated!

// AFTER: one source of truth
double withTax(double base) {
    return base + base * TAX_RATE;   // change the rate in ONE place
}
double invoiceTotal = withTax(price);
double receiptTotal = withTax(amount);

# BEFORE: the same tax math duplicated everywhere
invoice_total = price + price * 0.18      # 18% GST
receipt_total = amount + amount * 0.18    # repeated!

# AFTER: one source of truth
def with_tax(base: float) -> float:
    return base + base * TAX_RATE   # change the rate in ONE place

invoice_total = with_tax(price)
receipt_total = with_tax(amount)

Beware over-applying it: don't merge two pieces of code that look alike but represent different concepts that may change for different reasons. DRY is about removing duplicated knowledge, not duplicated characters.

KISS — Keep It Simple

Prefer the simplest design that works. Clever one-liners, premature abstractions, and "look how smart I am" code are liabilities — the next person (often future-you) has to understand them. If a plain if/else reads clearly, don't replace it with a reflection-driven rule engine.

  • Write code for the reader, not the compiler.
  • A boring, obvious solution that ships beats a brilliant one nobody can maintain.
  • Complexity should be earned by a real requirement, never added speculatively.

YAGNI — You Aren't Gonna Need It

Don't build features, configuration knobs, or extension points for hypothetical future requirements. Speculative generality adds code, tests, and bugs today to serve a need that may never arrive. Build what's required now; add flexibility when a concrete second use case actually appears.

YAGNI is the natural partner of KISS: KISS says keep what you build simple, YAGNI says don't build it at all until you need it.

Program to an Interface, Not an Implementation

Depend on the most abstract type that does the job. Declaring a variable as List rather than ArrayList means you can swap the concrete class later — or substitute a test double — without rippling changes through your code. This is the practical engine behind loose coupling.

// BAD: tied to a concrete implementation
ArrayList<String> names = new ArrayList<>();
void printAll(ArrayList<String> items) { ... }  // only ArrayList allowed

// GOOD: depend on the List interface
List<String> names = new ArrayList<>();   // swap to LinkedList freely
void printAll(List<String> items) {        // accepts ANY List
    for (String s : items) System.out.println(s);
}

from collections.abc import Iterable, Sequence

# BAD: demands a concrete list
def print_all(items: list): ...    # tuples, generators rejected

# GOOD: depend on the abstract protocol
def print_all(items: Iterable):       # list, tuple, generator — all fine
    for s in items:
        print(s)

names: Sequence[str] = ["ann", "bob"]   # swap the backing type freely

Favor Composition over Inheritance

Reuse behaviour by holding objects ("has-a") rather than extending classes ("is-a"). Composition keeps coupling loose and lets you change behaviour at runtime, where deep inheritance trees become rigid and fragile. This principle is important enough to have its own deep dive — see Coupling, Cohesion & Composition for the full Car-has-a-Engine treatment.

Separation of Concerns

Split a system so each part addresses one distinct concern — UI, business rules, persistence — and changes to one don't disturb the others. A class that parses HTTP, applies business logic, and writes SQL has tangled three concerns; splitting them into a controller, a service, and a repository makes each independently testable and replaceable. This is cohesion applied at the architectural scale.

GRASP — Assigning Responsibilities

GRASP (General Responsibility Assignment Software Patterns) is a vocabulary for answering the recurring design question: which class should be responsible for this? Here are the core ones for interviews:

PrincipleAssign the responsibility to…
Information Expertthe class that already holds the data needed to do the job (e.g. Order computes its own total, since it owns the line items).
Creatorthe class that aggregates, contains, or closely uses the created object (e.g. Order creates its OrderItems).
Controllera dedicated coordinator class that receives a system request and delegates the work (keeps UI out of business logic).
Low Couplingwhichever assignment minimizes dependencies between classes.
High Cohesionwhichever assignment keeps each class focused on a single responsibility.

Low Coupling and High Cohesion show up again here precisely because they're the yardsticks every other design decision is measured against.

Interview tip: These principles are most powerful as justification language. Don't just write code — narrate it: "I'll extract this into one method to stay DRY," or "by Information Expert, the Order should own its total calculation since it has the line items." Citing the principle behind a choice shows the interviewer you're designing deliberately, not by reflex.

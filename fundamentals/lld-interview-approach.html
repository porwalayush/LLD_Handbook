The LLD Interview Approach

A repeatable framework for any machine-coding / LLD round.

🏗️ Real-World Analogy

Imagine an architect handed a plot of land and told "design a house." A weak one starts laying bricks immediately. A strong one asks: how many people will live here? What's the budget? One storey or two? — then sketches a plan, then builds.

An LLD interview rewards exactly that discipline. The interviewer cares less about whether you finish and more about how you think: clarify, model, structure, then code. This page is that process, step by step.

Step 1 — Clarify Requirements & Scope

Never start coding from the one-line prompt. Spend the first few minutes turning a vague problem ("design a parking lot") into a crisp, bounded one. Separate functional requirements (what the system does) from non-functional ones (scale, concurrency, persistence), and explicitly agree on what's in scope vs. out of scope.

Good clarifying questions:

  • "What are the core operations the system must support?"
  • "Single instance, or distributed? Do we need to persist state or keep it in memory?"
  • "Is concurrency in scope — multiple users acting at once?"
  • "Should I handle payments / pricing, or treat that as out of scope?"
  • "Any scale targets I should design for?"

Write the agreed scope down. It becomes your checklist and protects you from the interviewer later asking "but what about…?"

Step 2 — Identify Core Entities (Find the Nouns)

Read the requirements and underline the nouns — they're your candidate classes. For a parking lot: ParkingLot, Level, ParkingSpot, Vehicle, Ticket, Gate. Don't worry about getting it perfect; you're building a first vocabulary you'll refine. Group obvious variants (Car, Bike, Truck) under a common abstraction (Vehicle).

Step 3 — Define Relationships & Responsibilities (Find the Verbs)

Now underline the verbs — they become methods, and they reveal how entities relate. "A vehicle parks in a spot," "a lot issues a ticket." Assign each responsibility to the class best suited to own it (the one with the data — the Information Expert), and lean on the SOLID Principles so each class has a single, focused reason to change.

Step 4 — Sketch the Class Diagram / APIs

Put it on the whiteboard: boxes for classes, lines for relationships, and the key public method signatures. This is where you commit to interfaces before implementations. A quick diagram catches design mistakes far cheaper than code does and gives the interviewer a chance to nudge you. See UML & Class Diagrams for the notation.

Step 5 — Identify Which Design Patterns Fit

With the structure visible, spot where a known pattern cleanly solves a sub-problem. Don't force patterns — apply them where they genuinely reduce coupling or duplication:

  • Need exactly one shared instance (e.g. the lot's central registry)? Consider Singleton (and mention its testability downsides).
  • Bridging two incompatible interfaces (e.g. a third-party payment SDK)? Reach for the Adapter.
  • Interchangeable algorithms (e.g. different pricing or spot-allocation policies)? Use Strategy.

Step 6 — Code the Skeleton, Iterate, Handle Edge Cases

Only now do you write code — and you start with the skeleton: enums, interfaces, class shells with key method signatures, then fill in the most important method bodies. Iterate breadth-first rather than perfecting one class while others stay empty. As you go, call out edge cases (full lot, invalid ticket, vehicle not found) and, if concurrency is in scope, where you'd guard shared state — see Locks & Synchronization.

Step 7 — Discuss Trade-offs & Extensibility

Close strong. Walk through what you'd change to support new requirements ("adding a new vehicle type only needs a new enum value and no changes to existing classes — that's Open/Closed in action"), where the design could scale or break, and what you deliberately left out and why. This is where senior candidates separate themselves.

Common mistakes that sink LLD rounds:
  • Jumping straight to code without clarifying scope or sketching a design.
  • Over-engineering — adding patterns, layers, and config for requirements nobody asked for (violates YAGNI/KISS).
  • Ignoring the stated requirements and designing the system you find interesting.
  • Forcing premature patterns — sprinkling Factories and Singletons to look clever instead of solving the actual problem.
  • Going silent — the interviewer can't credit thinking they can't hear.

Worked Mini-Example — A Parking Lot

Applying the steps to a small slice of "design a parking lot":

  • Clarify: in-memory, single lot, multiple spot sizes; payments out of scope; basic concurrency on spot allocation.
  • Nouns (entities): ParkingLot, ParkingSpot, Vehicle (Car/Bike/Truck), Ticket.
  • Verbs (responsibilities): lot parks a vehicle and issues a ticket; lot frees a spot on exit.
  • Patterns: Strategy for "which spot to assign" (nearest, first-available); Vehicle as an abstract base for the size hierarchy.

Interview tip (time management): for a 45–60 minute round, budget roughly 5–8 min clarifying, 8–10 min entities + diagram, 20–25 min coding the skeleton and key methods, and 5 min on trade-offs. Don't burn the whole hour perfecting one class — get a working end-to-end skeleton first, then deepen. Talk continuously: a running narration of your reasoning is the single highest-signal thing the interviewer evaluates.

The Four Pillars of OOP

The four ideas that every object-oriented design rests on: encapsulation, abstraction, inheritance, and polymorphism.

🌍 Why "pillars"?

A building stands on its pillars. Pull one out and the whole thing wobbles. Object-oriented programming is the same: these four ideas hold up everything else — design patterns, SOLID principles, clean architecture. Master them and the rest of LLD suddenly makes sense.

The good news: each pillar is something you already understand from everyday life. We'll meet each one with a real-world picture first, then the code.

1. Encapsulation

Encapsulation means bundling data (fields) and the methods that operate on that data into one unit — a class — and hiding the internals behind a controlled interface. Outside code talks to the object through public methods (getters/setters), never by reaching in and grabbing raw fields.

💊 Analogy: the medicine capsule & the car pedals

A medicine capsule wraps the powder inside a shell — you swallow the capsule, you never handle the powder directly. The shell protects the contents and protects you.

Or think of a car: you press the accelerator and it goes faster. You don't reach into the engine to adjust fuel injection — the pedal hides that complexity and only lets you do safe, sensible things.

Why it matters: by hiding fields behind accessors you can validate input (reject a negative balance), change the internal representation later without breaking callers, and keep your object always in a valid state. The object owns its own rules.

public class BankAccount {
    // private -> the field is hidden from the outside world
    private double balance;

    public BankAccount(double opening) {
        if (opening < 0) throw new IllegalArgumentException("no negatives");
        this.balance = opening;
    }

    // controlled access: the only way to add money
    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("must be > 0");
        balance += amount;                // invariant stays safe
    }

    public double getBalance() { return balance; }   // getter, no setter
}

// Caller cannot do:  account.balance = -999;  // won't compile
BankAccount acct = new BankAccount(100);
acct.deposit(50);
System.out.println(acct.getBalance());   // 150.0

class BankAccount:
    def __init__(self, opening):
        if opening < 0:
            raise ValueError("no negatives")
        self.__balance = opening      # __ triggers name-mangling

    def deposit(self, amount):
        if amount <= 0:
            raise ValueError("must be > 0")
        self.__balance += amount

    # @property exposes a read-only "balance" attribute
    @property
    def balance(self):
        return self.__balance

# Python convention: _single = "internal, please don't touch"
#                    __double = name-mangled to _BankAccount__balance
acct = BankAccount(100)
acct.deposit(50)
print(acct.balance)          # 150  (read via property)
# acct.__balance = -999     # does NOT touch the real field
Python note: Python has no private keyword. By convention a leading underscore _value says "internal — leave it alone", and a double underscore __value triggers name-mangling (it becomes _ClassName__value), which discourages accidental access. It's "we're all adults here" enforced by etiquette, not the compiler.

2. Abstraction

Abstraction means exposing only the essential features of something while hiding the messy implementation details. You define what an object can do without committing to how it does it. In code this shows up as abstract classes and interfaces.

🚗 Analogy: driving without being a mechanic

You can drive a car perfectly well without knowing how combustion, transmission, or electronic fuel injection work. The steering wheel, pedals, and gear stick are the abstraction — a simple interface to a very complex machine. Swap a petrol engine for an electric motor and you still just press the pedal.

Encapsulation vs abstraction: they're cousins. Encapsulation is about hiding the data inside a class; abstraction is about hiding the complexity behind a simple, stable interface so callers depend on a concept, not a concrete implementation.

// interface = a pure contract: WHAT, never HOW
public interface PaymentMethod {
    void pay(double amount);     // no body, just the promise
}

// abstract class = a partial implementation + required gaps
public abstract class Notification {
    public void send(String msg) {        // shared behavior
        deliver("[App] " + msg);
    }
    // subclasses MUST fill this in
    protected abstract void deliver(String formatted);
}

public class EmailNotification extends Notification {
    @Override
    protected void deliver(String formatted) {
        System.out.println("Emailing: " + formatted);
    }
}

// Callers depend on the abstraction, not the concrete class:
Notification n = new EmailNotification();
n.send("Welcome!");

from abc import ABC, abstractmethod

# abc.ABC = Python's abstract base class
class Notification(ABC):
    def send(self, msg):              # shared, concrete behavior
        self.deliver(f"[App] {msg}")

    @abstractmethod
    def deliver(self, formatted):     # subclasses MUST implement
        ...

class EmailNotification(Notification):
    def deliver(self, formatted):
        print(f"Emailing: {formatted}")

# Notification()          # TypeError: can't instantiate abstract class
n = EmailNotification()
n.send("Welcome!")         # Emailing: [App] Welcome!

3. Inheritance

Inheritance lets one class (the child / subclass) reuse and extend the fields and methods of another (the parent / base class). It models an "is-a" relationship: a Dog is an Animal, so it inherits everything an animal can do and adds its own twist.

👨‍👩‍👧 Analogy: a child inheriting traits

A child inherits eye color and surname from their parents — they get those "for free" — and then develop their own personality on top. A subclass works the same way: it starts with everything the base class offers, then overrides or adds what makes it special.

public class Animal {
    protected String name;
    public Animal(String name) { this.name = name; }
    public String speak() { return name + " makes a sound"; }
}

// Dog IS-A Animal -> reuse + extend
public class Dog extends Animal {
    public Dog(String name) {
        super(name);                 // call the parent constructor
    }
    @Override
    public String speak() {        // override the inherited behavior
        return name + " says Woof!";
    }
}

Animal a = new Dog("Rex");
System.out.println(a.speak());     // Rex says Woof!

class Animal:
    def __init__(self, name):
        self.name = name
    def speak(self):
        return f"{self.name} makes a sound"

# Dog IS-A Animal
class Dog(Animal):
    def __init__(self, name):
        super().__init__(name)      # call the parent constructor
    def speak(self):                # override
        return f"{self.name} says Woof!"

a = Dog("Rex")
print(a.speak())               # Rex says Woof!
Don't over-use inheritance. Deep inheritance trees are rigid and brittle — a change in the base ripples through every subclass, and a subclass is locked to a single parent. The seasoned rule is "favor composition over inheritance": build behavior by holding objects (has-a) rather than extending them (is-a). Only reach for inheritance when the "is-a" relationship is genuine and stable. See Coupling & Cohesion for why looser relationships age better.

4. Polymorphism

Polymorphism (Greek for "many forms") lets a single interface stand for many underlying types. You call one method name and the right behavior runs depending on the actual object. There are two flavors:

  • Runtime polymorphism (overriding): a subclass replaces a parent method; which version runs is decided at runtime by the object's real type. This is the heart of OOP.
  • Compile-time polymorphism (overloading): several methods share a name but differ in parameters; the compiler picks the match. Java supports this directly; Python uses default/variadic args instead.
🎨 Analogy: "draw me" behaves per shape

Tell a circle, a square, and a triangle to draw(). Each obeys the same command but produces a different picture. You, the caller, don't care which shape it is — you just say "draw" and the object knows what to do. That is polymorphism: one message, many forms of response.

abstract class Shape {
    abstract double area();          // each shape computes its own

    // --- overloading: same name, different params (compile-time) ---
    String describe() { return "area = " + area(); }
    String describe(String unit) { return area() + " " + unit; }
}

class Circle extends Shape {
    double r;  Circle(double r) { this.r = r; }
    @Override double area() { return Math.PI * r * r; }
}
class Rectangle extends Shape {
    double w, h;  Rectangle(double w, double h) { this.w=w; this.h=h; }
    @Override double area() { return w * h; }
}

// runtime polymorphism: one loop, the right area() each time
Shape[] shapes = { new Circle(2), new Rectangle(3, 4) };
for (Shape s : shapes)
    System.out.println(s.area());   // 12.566..., then 12.0

from abc import ABC, abstractmethod
import math

class Shape(ABC):
    @abstractmethod
    def area(self): ...

    # Python "overloading" = default / optional args
    def describe(self, unit=""):
        return f"{self.area()} {unit}".strip()

class Circle(Shape):
    def __init__(self, r): self.r = r
    def area(self): return math.pi * self.r ** 2

class Rectangle(Shape):
    def __init__(self, w, h): self.w, self.h = w, h
    def area(self): return self.w * self.h

# runtime polymorphism: the right area() runs for each object
for s in [Circle(2), Rectangle(3, 4)]:
    print(s.area())        # 12.566..., then 12

The Four Pillars at a Glance

PillarWhat it doesKeyword / mechanism
Encapsulation Bundle data + methods; hide internals behind accessors private fields, getters/setters; Python _ / __
Abstraction Expose essentials, hide complexity; depend on contracts interface, abstract class; Python abc.ABC
Inheritance Reuse + extend a base class via an "is-a" relationship extends, super; Python class Child(Parent)
Polymorphism One interface, many forms; pick behavior by real type Overriding (@Override) & overloading
Interview tip: Interviewers love "explain the four pillars with an example." Don't just recite definitions — give a one-line real-world analogy for each (capsule, driving a car, child traits, "draw me"), then show a tiny code snippet. Bonus points: explain the difference between encapsulation (hiding data) and abstraction (hiding complexity), since candidates routinely blur them. Be ready to say overriding is runtime and overloading is compile-time.

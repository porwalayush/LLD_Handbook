SOLID Principles

The five rules for maintainable object-oriented design — the difference between code you can change happily and code you're afraid to touch.

🧱 What is SOLID?

SOLID is an acronym for five design principles, popularized by Robert C. Martin ("Uncle Bob"). Each addresses a common way that object-oriented code rots over time. Follow them and your classes stay small, focused, and easy to extend without breaking what already works.

Single Responsibility · Open/Closed · Liskov Substitution · Interface Segregation · Dependency Inversion. We'll take them one at a time, each with a "violation → fix".

S — Single Responsibility Principle

A class should have one, and only one, reason to change. Each class should do one job. If a class handles business logic and formatting and database access, then a change to any one of those forces you to touch — and risk breaking — the others.

🔧 Analogy: Swiss-army knife vs dedicated tools

A Swiss-army knife does ten jobs poorly and is awkward at all of them; if the scissors break you might have to replace the whole tool. A workshop of dedicated tools — one screwdriver, one good pair of scissors — each does one job well and can be swapped independently. SRP says build dedicated tools, not Swiss-army classes.

// VIOLATION: a "God class" with three reasons to change
class Invoice {
    void calculateTotal() { /* business logic */ }
    void printToPdf()    { /* formatting */ }
    void saveToDb()      { /* persistence */ }
}

// FIX: split by responsibility — one reason to change each
class Invoice {
    double calculateTotal() { /* only business logic */ return 0; }
}
class InvoicePrinter {
    void printToPdf(Invoice inv) { /* only formatting */ }
}
class InvoiceRepository {
    void save(Invoice inv) { /* only persistence */ }
}

# VIOLATION: one class doing three unrelated jobs
class Invoice:
    def calculate_total(self): ...   # business logic
    def print_to_pdf(self):    ...   # formatting
    def save_to_db(self):      ...   # persistence

# FIX: one responsibility per class
class Invoice:
    def calculate_total(self): ...   # only business logic

class InvoicePrinter:
    def print_to_pdf(self, invoice): ...

class InvoiceRepository:
    def save(self, invoice): ...

O — Open/Closed Principle

Software entities should be open for extension, but closed for modification. You should be able to add new behavior by writing new code, not by editing existing, tested code. The classic smell is a growing if/else or switch on a type field — every new type forces you back into the same method.

🔌 Analogy: a power strip

A power strip is "open for extension" — plug in a new device any time — but "closed for modification": you never rewire the strip to add a device. New behavior plugs in; the existing wiring stays untouched. Polymorphism is your power strip.

// VIOLATION: every new shape edits this method
double area(Object shape) {
    if (shape instanceof Circle c)    return Math.PI * c.r * c.r;
    else if (shape instanceof Square s) return s.side * s.side;
    // ...add Triangle here? edit again. Risky.
    throw new IllegalArgumentException();
}

// FIX: extend via new classes; the caller never changes
interface Shape { double area(); }
class Circle   implements Shape { double r;    public double area(){return Math.PI*r*r;} }
class Square   implements Shape { double side; public double area(){return side*side;} }
class Triangle implements Shape { /* new type, zero edits elsewhere */ }

double area(Shape shape) { return shape.area(); }   // closed for modification

from abc import ABC, abstractmethod
import math

# VIOLATION: a type-switch you must edit for every new shape
def area(shape):
    if shape.kind == "circle": return math.pi * shape.r ** 2
    elif shape.kind == "square": return shape.side ** 2
    # add elif for every new shape... fragile

# FIX: each shape owns its area(); add new ones freely
class Shape(ABC):
    @abstractmethod
    def area(self): ...

class Circle(Shape):
    def __init__(self, r): self.r = r
    def area(self): return math.pi * self.r ** 2

class Square(Shape):
    def __init__(self, side): self.side = side
    def area(self): return self.side ** 2

def area(shape): return shape.area()   # never changes again

L — Liskov Substitution Principle

Subtypes must be substitutable for their base type without breaking the program. If code works with a Bird, handing it any subclass of Bird must not produce surprises. A subclass that can't honor the parent's contract is not really an "is-a".

🐧 Analogy: the penguin that can't fly

If Bird promises a fly() method, then Penguin extends Bird is a trap — a penguin can't fly, so any code expecting "all birds fly" breaks the moment it meets a penguin. The fix: don't model the wrong "is-a". Split capabilities so only flying birds promise fly().

// VIOLATION: Penguin breaks the "every Bird can fly" assumption
class Bird { void fly() { /* ... */ } }
class Penguin extends Bird {
    @Override void fly() {
        throw new UnsupportedOperationException(); // surprise!
    }
}

// FIX: separate the ability that not all birds have
interface Bird { void eat(); }
interface FlyingBird extends Bird { void fly(); }

class Sparrow implements FlyingBird {
    public void eat() {} public void fly() {}
}
class Penguin implements Bird {   // no fly() to violate
    public void eat() {}
}

# VIOLATION: substituting a Penguin breaks fly-expecting code
class Bird:
    def fly(self): ...

class Penguin(Bird):
    def fly(self):
        raise NotImplementedError("penguins can't fly")  # surprise!

# FIX: model only the abilities each bird truly has
class Bird:
    def eat(self): ...

class FlyingBird(Bird):
    def fly(self): ...

class Sparrow(FlyingBird):   # can be used wherever fly() is expected
    def fly(self): print("flap flap")

class Penguin(Bird):       # never promises fly()
    def eat(self): print("munch")

I — Interface Segregation Principle

Clients should not be forced to depend on methods they do not use. Many small, focused interfaces beat one big "fat" interface. A class implementing a fat interface is dragged into implementing methods that make no sense for it.

🤖 Analogy: a robot forced to eat()

Suppose one fat Worker interface declares work() and eat(). A human worker implements both happily. But a RobotWorker must also implement eat() — which is nonsense. Split the interface so robots implement only Workable and skip Feedable.

// VIOLATION: a fat interface forces robots to eat()
interface Worker {
    void work();
    void eat();            // robots don't eat!
}
class Robot implements Worker {
    public void work() {}
    public void eat() { /* awkward no-op / throws */ }
}

// FIX: small, client-specific interfaces
interface Workable { void work(); }
interface Feedable { void eat(); }

class Human implements Workable, Feedable {
    public void work() {} public void eat() {}
}
class Robot implements Workable {   // only what it needs
    public void work() {}
}

from abc import ABC, abstractmethod

# VIOLATION: one fat interface for everyone
class Worker(ABC):
    @abstractmethod
    def work(self): ...
    @abstractmethod
    def eat(self): ...        # robots forced to implement this

# FIX: split into focused, role-based interfaces
class Workable(ABC):
    @abstractmethod
    def work(self): ...

class Feedable(ABC):
    @abstractmethod
    def eat(self): ...

class Human(Workable, Feedable):
    def work(self): ...
    def eat(self): ...

class Robot(Workable):       # implements only Workable
    def work(self): ...

D — Dependency Inversion Principle

Depend on abstractions, not on concretions. High-level modules (your business logic) should not depend on low-level modules (a specific database, a specific email service) — both should depend on an interface. The trick in practice is constructor injection: pass the dependency in, typed as an interface, instead of hard-coding new inside.

🔌 Analogy: a wall socket, not soldered wires

Your lamp doesn't have its wires soldered directly into the building's electrical system. It ends in a plug that fits a standard socket — the abstraction. You can move the lamp to any socket, and the building doesn't care what you plug in. Code should depend on the "socket" (interface), not be soldered to one concrete device.

// VIOLATION: high-level class soldered to a concrete dependency
class OrderService {
    private MySqlDatabase db = new MySqlDatabase(); // hard-coded!
    void place(Order o) { db.save(o); }       // can't swap or test
}

// FIX: depend on an abstraction, inject via the constructor
interface OrderRepository { void save(Order o); }

class MySqlRepository  implements OrderRepository { public void save(Order o){} }
class InMemoryRepository implements OrderRepository { public void save(Order o){} }

class OrderService {
    private final OrderRepository repo;
    public OrderService(OrderRepository repo) {  // injected
        this.repo = repo;
    }
    void place(Order o) { repo.save(o); }
}

// Production wires MySql; tests wire InMemory — same OrderService.
new OrderService(new MySqlRepository());

from abc import ABC, abstractmethod

# VIOLATION: business logic hard-wired to one concrete class
class OrderService:
    def __init__(self):
        self.db = MySqlDatabase()        # hard-coded; can't swap
    def place(self, order):
        self.db.save(order)

# FIX: depend on an abstraction; inject it in the constructor
class OrderRepository(ABC):
    @abstractmethod
    def save(self, order): ...

class MySqlRepository(OrderRepository):
    def save(self, order): ...

class InMemoryRepository(OrderRepository):
    def save(self, order): ...

class OrderService:
    def __init__(self, repo: OrderRepository):  # injected
        self.repo = repo
    def place(self, order):
        self.repo.save(order)

# Production: OrderService(MySqlRepository())
# Tests:      OrderService(InMemoryRepository())

SOLID at a Glance

LetterPrincipleOne-line takeaway
SSingle ResponsibilityOne class, one reason to change.
OOpen/ClosedExtend with new code, don't edit old code.
LLiskov SubstitutionSubtypes must work wherever the base works.
IInterface SegregationMany small interfaces beat one fat one.
DDependency InversionDepend on abstractions, inject concretions.
Interview tip: SOLID is the most-asked fundamentals topic in LLD interviews. Don't just expand the acronym — be ready to name a violation and fix it live. Keep one crisp example per letter in your back pocket (God-class split, replace a type-switch with polymorphism, the penguin, the fat Worker interface, constructor injection). Mentioning that the Open/Closed fix is really just the Factory/Strategy pattern in disguise shows you see how principles connect to patterns.

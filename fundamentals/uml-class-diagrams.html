UML & Class Diagrams

Sketching designs you can draw on a whiteboard in interviews.

🗺️ Real-World Analogy

A UML class diagram is the blueprint of a building before construction starts. An architect doesn't pour concrete and then figure out where the walls go — they sketch a plan everyone can read: how many rooms, how they connect, which is load-bearing.

UML is that shared visual language for software. It lets you (and your interviewer) agree on what classes exist and how they relate before a single line of code is written.

1. Class Diagram Basics

A class is drawn as a box split into three compartments: the class name on top, its attributes (fields) in the middle, and its methods (behaviour) at the bottom. Each member gets a visibility marker:

  • + public — visible everywhere
  • - private — visible only inside the class
  • # protected — visible to the class and its subclasses
  • ~ package / default — visible within the same package

Here is a single class drawn in ASCII, the way you'd sketch it on a whiteboard:

+---------------------------------+
|         BankAccount             |   <- name
+---------------------------------+
| - id: String                    |
| - balance: double               |   <- attributes
| # ownerName: String             |
+---------------------------------+
| + deposit(amount: double): void |
| + withdraw(amount: double): void|   <- methods
| + getBalance(): double          |
| - validate(amount: double): void|
+---------------------------------+

Attributes and methods are written as name: Type (and name(param: Type): ReturnType for methods), matching most UML conventions regardless of language.

2. Relationships

The real power of UML is showing how classes connect. Each relationship has a distinct line/arrowhead so the diagram reads at a glance.

RelationshipNotationMeaningExample
AssociationPlain solid line ———One class knows about / uses anotherTeacher — Student
AggregationSolid line + hollow diamond ◇——— at the wholeWhole-part where parts can live independentlyTeam ◇— Player
CompositionSolid line + filled diamond ◆——— at the wholeStrong whole-part; parts die with the wholeHouse ◆— Room
Inheritance (generalization)Solid line + hollow triangle ———▷ at the parent"is-a"; subclass extends superclassDog ——▷ Animal
Realization (implements)Dashed line + hollow triangle - - ▷Class implements an interfaceArrayList - -▷ List
DependencyDashed arrow - - >Uses transiently (e.g. a method parameter)OrderService - -> EmailClient

A handy memory hook: diamonds = ownership (hollow = weak, filled = strong), triangles = type hierarchy (solid = extends, dashed = implements), and dashed lines = "just uses, doesn't hold."

3. Multiplicity

Numbers at the ends of a relationship line state how many objects participate. Read them as "each X is related to this many Y".

MultiplicityMeansExample
1Exactly oneAn Order has exactly 1 Customer
0..1Zero or one (optional)A Person has 0..1 Spouse
*Zero or manyA Shelf holds * Books
1..*One or many (at least one)An Order has 1..* LineItems

You write multiplicity at both ends. For example, Customer "1" ——— "*" Order reads "one customer places many orders, and each order belongs to exactly one customer."

4. Worked Example — Order / Customer / Product

Let's model a tiny ordering system. A Customer places Orders; each Order contains many OrderItems; each OrderItem refers to a Product. The order owns its items (composition — kill the order and its items vanish), but only references products (association — products exist in a catalog independently).

   +------------+   1      *   +-----------+
   |  Customer  |------------>|   Order   |
   +------------+ places      +-----------+
   | - name     |             | - id      |
   +------------+             | - date    |
                              +-----------+
                                   | 1
                                   | owns  (composition)
                                   ◆
                                   | *
                              +-------------+      *   1  +-----------+
                              |  OrderItem  |----------->|  Product  |
                              +-------------+ refers to   +-----------+
                              | - quantity  |             | - name    |
                              +-------------+             | - price   |
                              +-------------+             +-----------+

And the corresponding class skeletons:

class Customer {
    private String name;
    private List<Order> orders = new ArrayList<>();  // 1 -> *
    public void placeOrder(Order o) { orders.add(o); }
}

class Order {
    private String id;
    private LocalDate date;
    // composition: the order owns its items
    private List<OrderItem> items = new ArrayList<>();
}

class OrderItem {
    private int quantity;
    private Product product;   // association: just a reference
}

class Product {
    private String name;
    private double price;
}

from datetime import date

class Customer:
    def __init__(self, name: str):
        self.name = name
        self.orders: list["Order"] = []      # 1 -> *

    def place_order(self, order: "Order"):
        self.orders.append(order)

class Order:
    def __init__(self, id: str, on: date):
        self.id = id
        self.date = on
        # composition: the order owns its items
        self.items: list["OrderItem"] = []

class OrderItem:
    def __init__(self, quantity: int, product: "Product"):
        self.quantity = quantity
        self.product = product       # association: just a reference

class Product:
    def __init__(self, name: str, price: float):
        self.name = name
        self.price = price

5. A Note on Sequence Diagrams

Where a class diagram shows the static structure ("what exists"), a sequence diagram shows dynamic interaction over time ("what happens when"). Each participating object gets a vertical lifeline running top to bottom, and horizontal arrows between them represent messages (method calls), drawn in time order from top to bottom.

For example, "Customer → Order: addItem()" then "Order → Product: getPrice()". In an LLD interview you usually only need a sequence diagram to clarify a tricky flow (like a checkout or a payment retry); the class diagram does most of the heavy lifting.

Interview tip: Draw the class diagram before you start coding. Identifying the entities, their attributes, and the relationship arrows on the whiteboard demonstrates structured thinking, gives the interviewer a chance to course-correct you early, and earns explicit design points. Even a rough box-and-line sketch beats jumping straight into code.

LLD Problem Easy

Design an ATM Machine

A close cousin of the vending machine — driven by a State machine. The card moves through clear stages (inserted, PIN entered, transaction chosen, dispensing), and each stage only permits certain actions.

📋 Problem Statement

Design an ATM. A customer inserts a card, enters a PIN, picks a transaction (withdraw, balance, deposit), and the machine dispenses cash using the fewest notes, then ejects the card. Each step is only valid in the right stage — you can't withdraw before authenticating — and the cash dispenser must hand out an exact amount from its note inventory.

Step 1 — Gather Requirements

Start by pinning down scope. Say your assumptions out loud — interviewers want to see you separate the must-haves from the nice-to-haves.

✓ Functional

  • Insert card → enter PIN → select transaction.
  • Withdraw, check balance, deposit.
  • Dispense cash using the fewest notes.
  • Eject card and reset at the end (or on cancel).
  • Reject withdrawals over balance or over machine cash.

◆ Non-Functional

  • Safety: no action allowed out of sequence.
  • Extensibility: add a transaction type or state without rewrites.
  • Atomicity: debit account only if cash actually dispenses.
  • Clear prompts and graceful cancellation.

Step 2 — Identify the Core Entities

Find the nouns in the problem. These become your classes:

  • ATM — the context; holds the current state, the active card/account, and the dispenser.
  • ATMState (interface) — insertCard, enterPin, selectTransaction, ejectCard.
  • IdleState, CardInsertedState, PinEnteredState, TransactionSelectedState, DispensingState.
  • Card — card number + linked account; Account — balance + PIN.
  • CashDispenser — note inventory + the fewest-notes algorithm.
  • Transaction (Strategy) — WithdrawTransaction, BalanceTransaction, DepositTransaction.

Step 3 — Class Diagram

Two patterns combine here: State drives the workflow stages, while Strategy models the interchangeable transaction types.

Step 4 — Patterns in Play

Name the patterns as you design — it signals maturity. The ATM is a classic State + Strategy combination:

State — workflow stages Strategy — transaction types Singleton — one ATM controller Locking — atomic dispense

Step 5 — Implementation

The State machine enforces the order of operations; the CashDispenser shows the fewest-notes greedy algorithm; and Transaction is a Strategy so withdraw/balance/deposit are interchangeable.

import java.util.*;

class Account {
    final String pin; int balance;
    Account(String pin, int balance) { this.pin = pin; this.balance = balance; }
}
class Card {
    final String number; final Account account;
    Card(String number, Account account) { this.number = number; this.account = account; }
}

// Dispenses an amount with the FEWEST notes (greedy, largest first)
class CashDispenser {
    private final TreeMap<Integer, Integer> notes = new TreeMap<>(Collections.reverseOrder());
    CashDispenser() { notes.put(2000, 5); notes.put(500, 10); notes.put(100, 20); }

    synchronized Map<Integer, Integer> dispense(int amount) {
        Map<Integer, Integer> plan = new LinkedHashMap<>();
        int remaining = amount;
        for (Map.Entry<Integer, Integer> e : notes.entrySet()) {
            int denom = e.getKey(), available = e.getValue();
            int use = Math.min(remaining / denom, available);
            if (use > 0) { plan.put(denom, use); remaining -= use * denom; }
        }
        if (remaining != 0) throw new IllegalStateException("Cannot dispense exact amount");
        plan.forEach((d, c) -> notes.put(d, notes.get(d) - c));   // commit
        return plan;
    }
}

// Strategy: interchangeable transaction types
interface Transaction { void execute(ATM atm); }
class WithdrawTransaction implements Transaction {
    final int amount;
    WithdrawTransaction(int amount) { this.amount = amount; }
    public void execute(ATM atm) {
        Account acc = atm.card.account;
        if (amount > acc.balance) { System.out.println("Insufficient funds"); return; }
        atm.setState(new DispensingState());            // -> Dispensing
        Map<Integer, Integer> notes = atm.dispenser.dispense(amount);
        acc.balance -= amount;                                // debit only after cash leaves
        System.out.println("Dispensed " + amount + " as " + notes);
    }
}
class BalanceTransaction implements Transaction {
    public void execute(ATM atm) { System.out.println("Balance: " + atm.card.account.balance); }
}

// State: each stage only permits certain actions
interface ATMState {
    default void insertCard(ATM a, Card c) { System.out.println("Invalid here"); }
    default void enterPin(ATM a, String pin) { System.out.println("Invalid here"); }
    default void selectTransaction(ATM a, Transaction t) { System.out.println("Invalid here"); }
    default void ejectCard(ATM a) { a.card = null; a.setState(new IdleState()); System.out.println("Card ejected"); }
}
class IdleState implements ATMState {
    public void insertCard(ATM a, Card c) { a.card = c; a.setState(new CardInsertedState()); }
}
class CardInsertedState implements ATMState {
    public void enterPin(ATM a, String pin) {
        if (pin.equals(a.card.account.pin)) a.setState(new PinEnteredState());
        else System.out.println("Wrong PIN");
    }
}
class PinEnteredState implements ATMState {
    public void selectTransaction(ATM a, Transaction t) {
        a.setState(new TransactionSelectedState());
        t.execute(a);
        a.ejectCard();                                     // finish -> Idle
    }
}
class TransactionSelectedState implements ATMState {}
class DispensingState implements ATMState {}

class ATM {                                          // context
    ATMState state = new IdleState();
    Card card;
    final CashDispenser dispenser = new CashDispenser();
    void setState(ATMState s) { state = s; }
    void insertCard(Card c) { state.insertCard(this, c); }
    void enterPin(String p) { state.enterPin(this, p); }
    void selectTransaction(Transaction t) { state.selectTransaction(this, t); }
    void ejectCard() { state.ejectCard(this); }
}

from abc import ABC, abstractmethod
import threading

class Account:
    def __init__(self, pin, balance):
        self.pin, self.balance = pin, balance

class Card:
    def __init__(self, number, account):
        self.number, self.account = number, account

# Dispenses an amount with the FEWEST notes (greedy, largest first)
class CashDispenser:
    def __init__(self):
        self.notes = {2000: 5, 500: 10, 100: 20}
        self.lock = threading.Lock()
    def dispense(self, amount):
        with self.lock:                                  # atomic
            plan, remaining = {}, amount
            for denom in sorted(self.notes, reverse=True):
                use = min(remaining // denom, self.notes[denom])
                if use:
                    plan[denom] = use; remaining -= use * denom
            if remaining != 0:
                raise RuntimeError("Cannot dispense exact amount")
            for denom, cnt in plan.items():
                self.notes[denom] -= cnt                  # commit
            return plan

# Strategy: interchangeable transaction types
class Transaction(ABC):
    @abstractmethod
    def execute(self, atm): ...

class WithdrawTransaction(Transaction):
    def __init__(self, amount): self.amount = amount
    def execute(self, atm):
        acc = atm.card.account
        if self.amount > acc.balance:
            print("Insufficient funds"); return
        atm.state = DispensingState()                    # -> Dispensing
        notes = atm.dispenser.dispense(self.amount)
        acc.balance -= self.amount                        # debit after cash leaves
        print(f"Dispensed {self.amount} as {notes}")

class BalanceTransaction(Transaction):
    def execute(self, atm): print(f"Balance: {atm.card.account.balance}")

# State: each stage only permits certain actions
class ATMState(ABC):
    def insert_card(self, a, c): print("Invalid here")
    def enter_pin(self, a, pin): print("Invalid here")
    def select_transaction(self, a, t): print("Invalid here")
    def eject_card(self, a):
        a.card = None; a.state = IdleState(); print("Card ejected")

class IdleState(ATMState):
    def insert_card(self, a, c):
        a.card = c; a.state = CardInsertedState()

class CardInsertedState(ATMState):
    def enter_pin(self, a, pin):
        if pin == a.card.account.pin: a.state = PinEnteredState()
        else: print("Wrong PIN")

class PinEnteredState(ATMState):
    def select_transaction(self, a, t):
        a.state = TransactionSelectedState()
        t.execute(a)
        a.eject_card()                                   # finish -> Idle

class TransactionSelectedState(ATMState): pass
class DispensingState(ATMState): pass

class ATM:                                          # context
    def __init__(self):
        self.state = IdleState()
        self.card = None
        self.dispenser = CashDispenser()
    def insert_card(self, c): self.state.insert_card(self, c)
    def enter_pin(self, p): self.state.enter_pin(self, p)
    def select_transaction(self, t): self.state.select_transaction(self, t)
    def eject_card(self): self.state.eject_card(self)

# --- demo ---
atm = ATM()
card = Card("1234", Account("4321", 5000))
atm.insert_card(card)              # Idle -> CardInserted
atm.enter_pin("4321")            # -> PinEntered
atm.select_transaction(WithdrawTransaction(2600))  # 1x2000 + 1x500 + 1x100
Why this design scores well: the State machine makes illegal flows impossible — calling enterPin while Idle just prints "Invalid here", because only CardInsertedState implements it. The fewest-notes dispenser is a greedy walk from the largest denomination, and it commits the inventory only after a full plan is found, so a failed dispense leaves the drawer untouched. Debiting the account after cash leaves keeps money and balance consistent. Transaction as a Strategy means a new transaction type is a new class, never an edit to the states.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • PIN retry limit: track attempts in CardInsertedState and capture the card after 3 failures.
  • Atomic withdraw: wrap "check balance → reserve cash → debit" in one lock so two sessions can't overdraw a shared account.
  • Bank network: route authentication/debit to a BankService instead of a local account.
  • Greedy isn't always optimal: with odd denominations, fall back to a DP coin-change to guarantee a valid combination.
  • Receipt & deposit: add a cash-acceptor and a printer; model deposit as another Strategy.
Common mistakes: using a single enum + switch instead of State objects (lets illegal transitions slip through); debiting the account before confirming the cash dispensed; mutating the note inventory mid-loop before knowing the full plan succeeds; and hardcoding the three transaction types with if/else rather than a Strategy. Separate "what stage am I in" (State) from "what am I doing" (Strategy) and the design stays clean.

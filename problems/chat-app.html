LLD Problem Hard

Design a Chat / Messaging App

A WhatsApp-style messaging core. Users send messages into one-to-one and group conversations, a central server routes them to the right recipients, and delivery/read receipts plus online presence flow back to senders.

📋 Problem Statement

Design a chat system. A User sends a Message into a Conversation — either one-to-one or a Group. All messages pass through a central ChatServer that decides who receives them (fan-out to every group member except the sender). The server tracks presence (online/offline) and each message carries a MessageStatus moving SENT → DELIVERED → READ. Receipts and incoming messages are delivered to recipients via notifications.

Step 1 — Gather Requirements

Scope the problem out loud before coding. Separate the essentials from the extras.

✓ Functional

  • One-to-one and group conversations.
  • Send a message; route it to the right recipients.
  • Fan-out a group message to all members but the sender.
  • Delivery and read receipts per message.
  • Online / offline presence per user.

◆ Non-Functional

  • Decoupling: users never reference each other directly — only the server.
  • Extensibility: add conversation types or notification channels easily.
  • Offline handling: queue messages for offline users.
  • Ordering: messages within a conversation stay in order.

Step 2 — Identify the Core Entities

The nouns of the problem map to classes:

  • User — sends and receives messages; has a presence flag.
  • Message — text + sender + timestamp + a mutable MessageStatus.
  • MessageStatus — enum SENT / DELIVERED / READ.
  • Conversation (abstract) → DirectConversation and GroupConversation — hold members and history.
  • ChatServer — the Mediator: the single hub all messages route through; knows membership and presence.
  • NotificationObserver — the Observer: listeners notified when a user gets a message or a receipt.

Step 3 — Class Diagram

The ChatServer is a Mediator — users talk to it, not to each other. It owns routing, presence, and fan-out. Observers carry events back to clients.

Step 4 — Patterns in Play

Name the patterns as you design — it shows you recognize the shape of the problem:

Mediator — ChatServer routes all messages Observer — push receipts & messages to clients Factory — create conversation types Producer-Consumer — offline message queue

Step 5 — Implementation

A runnable core. Users only ever call server.send(...); the Mediator looks up the conversation, fans out to members, flips statuses, and notifies via observers.

import java.util.*;

enum MessageStatus { SENT, DELIVERED, READ }

class Message {
    final String id, text; final User sender; final long ts;
    MessageStatus status = MessageStatus.SENT;
    Message(String id, User sender, String text) {
        this.id = id; this.sender = sender; this.text = text;
        this.ts = System.currentTimeMillis();
    }
}

// ---- Observer: how a client is notified ----
interface NotificationObserver {
    void onMessage(User to, Message m);
    void onReceipt(Message m);
}
class PushNotifier implements NotificationObserver {
    public void onMessage(User to, Message m) {
        System.out.println("[push -> " + to.name + "] " + m.sender.name + ": " + m.text);
    }
    public void onReceipt(Message m) {
        System.out.println("  receipt: " + m.id + " -> " + m.status);
    }
}

// ---- Conversations ----
abstract class Conversation {
    final String id;
    final List<User> members = new ArrayList<>();
    final List<Message> history = new ArrayList<>();
    Conversation(String id) { this.id = id; }
    List<User> recipients(User sender) {
        List<User> r = new ArrayList<>(members);
        r.remove(sender);                // everyone but the sender
        return r;
    }
}
class DirectConversation extends Conversation {
    DirectConversation(String id, User a, User b) { super(id); members.add(a); members.add(b); }
}
class GroupConversation extends Conversation {
    GroupConversation(String id, List<User> m) { super(id); members.addAll(m); }
}

// ---- Mediator: the single hub all messages route through ----
class ChatServer {
    private final Map<String, Conversation> convs = new HashMap<>();
    private final Map<User, Boolean> online = new HashMap<>();
    private final Map<User, Queue<Message>> pending = new HashMap<>();
    private final NotificationObserver notifier;
    ChatServer(NotificationObserver n) { notifier = n; }

    void register(User u) { online.put(u, true); pending.put(u, new LinkedList<>()); u.server = this; }
    void addConversation(Conversation c) { convs.put(c.id, c); }

    void setOnline(User u, boolean up) {
        online.put(u, up);
        if (up) flush(u);          // deliver queued messages on reconnect
    }

    // the core routing method — users call only this
    synchronized void send(User from, String convId, String text) {
        Conversation c = convs.get(convId);
        Message m = new Message(UUID.randomUUID().toString().substring(0, 4), from, text);
        c.history.add(m);
        for (User to : c.recipients(from)) {       // fan-out
            if (Boolean.TRUE.equals(online.get(to))) deliver(to, m);
            else pending.get(to).add(m);          // queue for offline user
        }
    }

    private void deliver(User to, Message m) {
        notifier.onMessage(to, m);
        m.status = MessageStatus.DELIVERED;
        notifier.onReceipt(m);
    }
    private void flush(User u) {
        Queue<Message> q = pending.get(u);
        while (!q.isEmpty()) deliver(u, q.poll());
    }
    void markRead(Message m) { m.status = MessageStatus.READ; notifier.onReceipt(m); }
}

class User {
    final String name; ChatServer server;
    User(String name) { this.name = name; }
    void send(String convId, String text) { server.send(this, convId, text); }
}

public class Demo {
    public static void main(String[] a) {
        ChatServer server = new ChatServer(new PushNotifier());
        User alice = new User("Alice"), bob = new User("Bob"), eve = new User("Eve");
        server.register(alice); server.register(bob); server.register(eve);
        server.addConversation(new GroupConversation("g1", List.of(alice, bob, eve)));

        alice.send("g1", "Hi team!");     // fans out to Bob & Eve
        server.setOnline(eve, false);
        bob.send("g1", "On my way");      // queued for offline Eve
        server.setOnline(eve, true);       // Eve reconnects -> queued msg flushed
    }
}

from abc import ABC, abstractmethod
from collections import deque
from enum import Enum
import time, uuid, threading

class MessageStatus(Enum):
    SENT = 1; DELIVERED = 2; READ = 3

class Message:
    def __init__(self, id, sender, text):
        self.id, self.sender, self.text = id, sender, text
        self.ts = time.time()
        self.status = MessageStatus.SENT

# ---- Observer: how a client is notified ----
class NotificationObserver(ABC):
    @abstractmethod
    def on_message(self, to, m): ...
    @abstractmethod
    def on_receipt(self, m): ...

class PushNotifier(NotificationObserver):
    def on_message(self, to, m):
        print(f"[push -> {to.name}] {m.sender.name}: {m.text}")
    def on_receipt(self, m):
        print(f"  receipt: {m.id} -> {m.status.name}")

# ---- Conversations ----
class Conversation:
    def __init__(self, id, members):
        self.id, self.members, self.history = id, list(members), []
    def recipients(self, sender):
        return [u for u in self.members if u is not sender]

class DirectConversation(Conversation):
    def __init__(self, id, a, b): super().__init__(id, [a, b])
class GroupConversation(Conversation):
    def __init__(self, id, members): super().__init__(id, members)

# ---- Mediator: the single hub all messages route through ----
class ChatServer:
    def __init__(self, notifier):
        self.notifier = notifier
        self.convs, self.online, self.pending = {}, {}, {}
        self._lock = threading.Lock()

    def register(self, u):
        self.online[u], self.pending[u] = True, deque()
        u.server = self
    def add_conversation(self, c): self.convs[c.id] = c

    def set_online(self, u, up):
        self.online[u] = up
        if up: self._flush(u)        # deliver queued messages on reconnect

    # the core routing method — users call only this
    def send(self, frm, conv_id, text):
        with self._lock:
            c = self.convs[conv_id]
            m = Message(str(uuid.uuid4())[:4], frm, text)
            c.history.append(m)
            for to in c.recipients(frm):    # fan-out
                if self.online.get(to):
                    self._deliver(to, m)
                else:
                    self.pending[to].append(m)   # queue for offline user

    def _deliver(self, to, m):
        self.notifier.on_message(to, m)
        m.status = MessageStatus.DELIVERED
        self.notifier.on_receipt(m)
    def _flush(self, u):
        q = self.pending[u]
        while q: self._deliver(u, q.popleft())
    def mark_read(self, m):
        m.status = MessageStatus.READ
        self.notifier.on_receipt(m)

class User:
    def __init__(self, name):
        self.name, self.server = name, None
    def send(self, conv_id, text): self.server.send(self, conv_id, text)

if __name__ == "__main__":
    server = ChatServer(PushNotifier())
    alice, bob, eve = User("Alice"), User("Bob"), User("Eve")
    for u in (alice, bob, eve): server.register(u)
    server.add_conversation(GroupConversation("g1", [alice, bob, eve]))

    alice.send("g1", "Hi team!")      # fans out to Bob & Eve
    server.set_online(eve, False)
    bob.send("g1", "On my way")      # queued for offline Eve
    server.set_online(eve, True)        # Eve reconnects -> queued msg flushed
Why this design scores well: the ChatServer Mediator means a User never holds references to other users — it just calls server.send(...). Adding a user, a group, or a moderation hook touches only the server, not every client (this is exactly the N-to-N coupling Mediator dissolves). The Observer (NotificationObserver) decouples how a client is reached — push, websocket, badge — from the routing logic. Queuing for offline users is a natural producer-consumer touchpoint the interviewer will appreciate.

Step 6 — Follow-up Extensions

Anticipate where the interviewer pushes next:

  • Read receipts at scale: in a group, track a per-member status map instead of one status on the message.
  • Ordering & dedupe: attach a monotonic sequence number per conversation; clients ack by sequence.
  • Real delivery: swap the in-process observer for websockets / a message broker (Kafka) behind the same interface.
  • Persistence: store history in a DB partitioned by conversation; the in-memory list becomes a recent-cache.
  • Typing & presence: presence becomes its own pub/sub topic through the same Mediator.
Common mistakes: letting users hold direct references to each other (N×N coupling) instead of routing through the Mediator; baking the notification channel into the server instead of an Observer interface; forgetting offline users so messages are silently dropped; storing a single status on a group message instead of per-recipient. Avoid these and you're in the top tier.

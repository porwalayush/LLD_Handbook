LLD Problem Medium

Design Chess

A polymorphism showcase. Model an 8×8 board where six kinds of pieces each move by their own rules, players alternate turns, and every move is validated before it lands.

📋 Problem Statement

Design the engine for a two-player game of chess. The board is an 8×8 grid of squares. There are six piece types — King, Queen, Rook, Bishop, Knight, Pawn — and each moves differently. Players alternate turns; a move is only allowed if it is legal for that piece and the path is valid. The core of the design is modeling the piece hierarchy so each type owns its own movement logic (polymorphism), plus a validator that checks moves before applying them.

Step 1 — Gather Requirements

Always start by pinning down scope. State your assumptions out loud; interviewers want to see you separate the must-haves from the nice-to-haves.

✓ Functional

  • 8×8 board of squares, each possibly holding a piece.
  • Six piece types, each with its own movement rule.
  • Two players alternate turns (white, then black).
  • Validate a move: legal for the piece, path clear, not capturing your own color.
  • Apply a move, capturing any opponent piece on the destination.

◆ Non-Functional

  • Extensibility: adding a piece variant means one new class, not edits everywhere.
  • Correctness: illegal moves must be rejected cleanly.
  • Undo: support taking back a move (snapshot the state).
  • Clear separation of board state from move rules.

Step 2 — Identify the Core Entities

Find the nouns in the problem. These become your classes:

  • Game — orchestrates turns, holds the board and players, and drives the move loop.
  • Board — the 8×8 grid; exposes the square at a coordinate.
  • Cell / Square — one position on the board; may hold a piece.
  • Piece (abstract) → King, Queen, Rook, Bishop, Knight, Pawn — each overrides canMove / validMoves.
  • Player — owns a color (white/black).
  • Move — from-square, to-square, the moving piece, and any captured piece.
  • MoveValidator — checks a proposed move against the rules before it's applied.

Step 5 — Implementation

A focused skeleton showing the Piece hierarchy with per-piece canMove, plus the validation that the Board runs before applying a move. In an interview you'd build this up incrementally; here's a clean, runnable core.

enum Color { WHITE, BLACK }

class Cell {
    final int row, col;
    Piece piece;                 // null = empty
    Cell(int r, int c) { row = r; col = c; }
}

// Abstract base: each subclass encapsulates its own movement rule
abstract class Piece {
    final Color color;
    Piece(Color color) { this.color = color; }
    // Is this geometric move legal for THIS piece type?
    abstract boolean canMove(Board b, Cell from, Cell to);
}

class Rook extends Piece {
    Rook(Color c) { super(c); }
    boolean canMove(Board b, Cell f, Cell t) {
        if (f.row != t.row && f.col != t.col) return false;  // straight lines only
        return b.pathClear(f, t);                                // nothing blocking
    }
}
class Bishop extends Piece {
    Bishop(Color c) { super(c); }
    boolean canMove(Board b, Cell f, Cell t) {
        if (Math.abs(f.row - t.row) != Math.abs(f.col - t.col)) return false; // diagonals
        return b.pathClear(f, t);
    }
}
class Knight extends Piece {
    Knight(Color c) { super(c); }
    boolean canMove(Board b, Cell f, Cell t) {
        int dr = Math.abs(f.row - t.row), dc = Math.abs(f.col - t.col);
        return (dr == 2 && dc == 1) || (dr == 1 && dc == 2);   // L-shape, jumps over pieces
    }
}
// Queen = Rook + Bishop, King = one step any direction, Pawn = forward/diagonal-capture (omitted for brevity)

class Move {
    final Cell from, to;
    final Piece moved, captured;
    Move(Cell f, Cell t, Piece m, Piece c) { from = f; to = t; moved = m; captured = c; }
}

class Board {
    final Cell[][] grid = new Cell[8][8];
    Board() { for (int r = 0; r < 8; r++) for (int c = 0; c < 8; c++) grid[r][c] = new Cell(r, c); }

    // Are all squares strictly between from and to empty? (knights skip this)
    boolean pathClear(Cell f, Cell t) {
        int dr = Integer.signum(t.row - f.row), dc = Integer.signum(t.col - f.col);
        int r = f.row + dr, c = f.col + dc;
        while (r != t.row || c != t.col) {
            if (grid[r][c].piece != null) return false;
            r += dr; c += dc;
        }
        return true;
    }

    // Validate then apply. The validator + polymorphic canMove do the heavy lifting.
    Move makeMove(Cell from, Cell to, Color turn) {
        Piece p = from.piece;
        if (p == null || p.color != turn) throw new IllegalArgumentException("Not your piece");
        if (to.piece != null && to.piece.color == turn) throw new IllegalArgumentException("Can't capture own");
        if (!p.canMove(this, from, to)) throw new IllegalArgumentException("Illegal for this piece");
        Move m = new Move(from, to, p, to.piece);
        to.piece = p; from.piece = null;     // apply
        return m;                          // push onto history for undo (Memento)
    }
}
from abc import ABC, abstractmethod
from enum import Enum

class Color(Enum):
    WHITE = 0; BLACK = 1

class Cell:
    def __init__(self, row, col):
        self.row, self.col, self.piece = row, col, None   # piece = None means empty

# Abstract base: each subclass owns its movement rule (polymorphism)
class Piece(ABC):
    def __init__(self, color):
        self.color = color
    @abstractmethod
    def can_move(self, board, frm, to): ...

class Rook(Piece):
    def can_move(self, board, frm, to):
        if frm.row != to.row and frm.col != to.col:    # straight lines only
            return False
        return board.path_clear(frm, to)

class Bishop(Piece):
    def can_move(self, board, frm, to):
        if abs(frm.row - to.row) != abs(frm.col - to.col):  # diagonals only
            return False
        return board.path_clear(frm, to)

class Knight(Piece):
    def can_move(self, board, frm, to):
        dr, dc = abs(frm.row - to.row), abs(frm.col - to.col)
        return (dr, dc) in ((2, 1), (1, 2))    # L-shape; jumps over pieces

class Queen(Piece):
    def can_move(self, board, frm, to):       # Rook OR Bishop move
        straight = frm.row == to.row or frm.col == to.col
        diagonal = abs(frm.row - to.row) == abs(frm.col - to.col)
        return (straight or diagonal) and board.path_clear(frm, to)

class Move:
    def __init__(self, frm, to, moved, captured):
        self.frm, self.to, self.moved, self.captured = frm, to, moved, captured

class Board:
    def __init__(self):
        self.grid = [[Cell(r, c) for c in range(8)] for r in range(8)]
        self.history = []                  # stack of Moves -> undo (Memento)

    # All squares strictly between frm and to must be empty (knights skip this)
    def path_clear(self, frm, to):
        dr = (to.row > frm.row) - (to.row < frm.row)   # sign: -1, 0, or +1
        dc = (to.col > frm.col) - (to.col < frm.col)
        r, c = frm.row + dr, frm.col + dc
        while (r, c) != (to.row, to.col):
            if self.grid[r][c].piece is not None:
                return False
            r, c = r + dr, c + dc
        return True

    # Validate, then apply. Polymorphic can_move does the per-piece check.
    def make_move(self, frm, to, turn):
        p = frm.piece
        if p is None or p.color != turn:
            raise ValueError("Not your piece")
        if to.piece and to.piece.color == turn:
            raise ValueError("Can't capture own piece")
        if not p.can_move(self, frm, to):
            raise ValueError("Illegal move for this piece")
        m = Move(frm, to, p, to.piece)
        to.piece, frm.piece = p, None     # apply
        self.history.append(m)               # remember for undo
        return m

    def undo(self):                          # Memento: restore last state
        m = self.history.pop()
        m.frm.piece, m.to.piece = m.moved, m.captured
Why this design scores well: the abstract Piece.canMove is the whole trick — adding a piece is one new subclass, and the Board never needs a giant switch on piece type (Open/Closed via polymorphism). Validation is layered: ownership check → not-capturing-own check → piece-specific geometry → path clear. Keeping a history stack of Move objects (Memento) makes undo a two-line operation.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Check / checkmate detection: after a move, scan whether the mover's own king is attacked (illegal) and whether the opponent's king has any legal escape (checkmate).
  • Special moves: castling (king + rook, neither moved, path clear, not through check), en passant (capture a pawn that just double-stepped), and promotion (a pawn reaching the back rank becomes a Queen via a Factory).
  • Pawn nuance: forward-only movement, two-square first move, diagonal-only capture — the one piece whose move depends on direction and history.
  • Move generation: add validMoves() per piece for an AI or hint feature.
  • Persistence / replay: serialize the move history to PGN; the Memento stack already gives you full undo/redo.
Common mistakes: writing one massive isValidMove with a switch on piece type instead of polymorphic canMove; forgetting that the knight jumps (so it must skip the path-clear check); allowing a move that leaves your own king in check; modeling the board as a flat array of ints instead of Cell objects holding pieces; and ignoring whose turn it is. Avoid these and you're in the top tier.

LLD Problem Medium

Design an Elevator System

A classic systems-thinking problem. Model a building with multiple elevator cars that respond to button presses, decide which car answers a call, and move floor-by-floor while doors open and close.

📋 Problem Statement

Design the control software for a building with multiple elevator cars. People press hall buttons (up/down) on a floor, or car buttons (a target floor) inside a car. The system must dispatch the best car to each hall call, move each car efficiently through its pending stops, and open the doors when it arrives. The heart of the problem is the scheduling strategy — which car should serve which request.

Step 1 — Gather Requirements

Always start by pinning down scope. State your assumptions out loud; interviewers want to see you separate the must-haves from the nice-to-haves.

✓ Functional

  • Multiple elevator cars serving N floors.
  • Two request types: internal (car button, a target floor) and external (hall call: floor + direction).
  • Dispatch a hall call to the most suitable car.
  • Each car moves up/down through its pending stops and opens doors on arrival.
  • Track each car's current floor, direction, and state.

◆ Non-Functional

  • Concurrency: requests arrive from many floors at once and must be queued safely.
  • Extensibility: swap scheduling algorithms (FCFS, nearest-car, SCAN) without touching the cars.
  • Efficiency: minimize wait time and wasted travel.
  • Responsive: never starve a waiting passenger.

Step 2 — Identify the Core Entities

Find the nouns in the problem. These become your classes:

  • ElevatorSystem — the controller; owns all cars and routes incoming requests through a scheduling strategy.
  • Elevator (the car) — has a current floor, a direction, a state, and a queue of stops it must visit.
  • Request — either an internal car-button press (target floor) or an external hall call (floor + desired direction).
  • Direction / ElevatorState — enums for UP/DOWN/IDLE and MOVING/IDLE/DOORS_OPEN.
  • SchedulingStrategy — pluggable algorithm that picks the best car for a hall call.
  • Door — opens/closes when the car stops.
  • Display — shows the current floor and direction.

Step 4 — Patterns in Play

Name the patterns as you design — it signals maturity. This problem naturally uses:

Strategy — pluggable dispatch (FCFS / nearest / SCAN) State — Idle / Moving / DoorsOpen Singleton — one ElevatorSystem Producer-Consumer — request queue Locking — thread-safe queues

Step 5 — Implementation

A focused skeleton showing the controller dispatching a hall call to the best car, and a car processing one step of its stop queue. In an interview you'd build this up incrementally; here's a clean, runnable core.

enum Direction { UP, DOWN, IDLE }
enum ElevatorState { IDLE, MOVING, DOORS_OPEN }

class Elevator {
    final int id;
    int currentFloor = 0;
    Direction direction = Direction.IDLE;
    ElevatorState state = ElevatorState.IDLE;
    // TreeSet keeps pending stops sorted so we sweep in order (LOOK)
    final TreeSet<Integer> stops = new TreeSet<>();

    Elevator(int id) { this.id = id; }

    synchronized void addStop(int floor) { stops.add(floor); }

    // Process exactly one step: move one floor toward the next stop,
    // or open the doors if we've arrived. Called in a loop by the controller.
    synchronized void step() {
        if (stops.isEmpty()) { direction = Direction.IDLE; state = ElevatorState.IDLE; return; }
        if (stops.contains(currentFloor)) {            // arrived
            state = ElevatorState.DOORS_OPEN;
            System.out.println("Car " + id + " doors open at " + currentFloor);
            stops.remove(currentFloor);
            return;
        }
        int next = direction == Direction.DOWN
                ? stops.floor(currentFloor) // nearest stop at/below
                : stops.ceiling(currentFloor);
        if (next == null) next = stops.first();   // reverse sweep
        direction = next > currentFloor ? Direction.UP : Direction.DOWN;
        state = ElevatorState.MOVING;
        currentFloor += direction == Direction.UP ? 1 : -1;
    }
}

// Strategy: pick the best car for a hall call
interface SchedulingStrategy {
    Elevator selectCar(List<Elevator> cars, int floor, Direction dir);
}
// Nearest-car: choose the closest car (a simple, effective heuristic)
class NearestCarStrategy implements SchedulingStrategy {
    public Elevator selectCar(List<Elevator> cars, int floor, Direction dir) {
        return cars.stream()
            .min(Comparator.comparingInt(c -> Math.abs(c.currentFloor - floor)))
            .orElseThrow();
    }
}

class ElevatorSystem {
    private final List<Elevator> cars = new ArrayList<>();
    private SchedulingStrategy strategy = new NearestCarStrategy();

    ElevatorSystem(int numCars) {
        for (int i = 0; i < numCars; i++) cars.add(new Elevator(i));
    }
    void setStrategy(SchedulingStrategy s) { this.strategy = s; }

    // External hall call: strategy picks the car, then we enqueue the stop
    void requestHallCall(int floor, Direction dir) {
        Elevator car = strategy.selectCar(cars, floor, dir);
        car.addStop(floor);
        System.out.println("Hall call @" + floor + " -> car " + car.id);
    }
    // Internal car button: that specific car gets the target floor
    void pressFloor(int carId, int target) { cars.get(carId).addStop(target); }

    void tick() { cars.forEach(Elevator::step); }  // advance all cars one step
}
from enum import Enum
from sortedcontainers import SortedSet   # or use a sorted list
import threading

class Direction(Enum):
    UP = 1; DOWN = -1; IDLE = 0

class ElevatorState(Enum):
    IDLE = 0; MOVING = 1; DOORS_OPEN = 2

class Elevator:
    def __init__(self, id):
        self.id = id
        self.current_floor = 0
        self.direction = Direction.IDLE
        self.state = ElevatorState.IDLE
        self.stops = SortedSet()        # pending stops, kept in order
        self.lock = threading.Lock()

    def add_stop(self, floor):
        with self.lock:               # requests arrive from many threads
            self.stops.add(floor)

    # One step: arrive + open doors, or move a single floor toward next stop
    def step(self):
        with self.lock:
            if not self.stops:
                self.direction = Direction.IDLE
                self.state = ElevatorState.IDLE
                return
            if self.current_floor in self.stops:        # arrived
                self.state = ElevatorState.DOORS_OPEN
                print(f"Car {self.id} doors open at {self.current_floor}")
                self.stops.discard(self.current_floor)
                return
            target = self.stops[0] if self.current_floor < self.stops[0] else self.stops[-1]
            self.direction = Direction.UP if target > self.current_floor else Direction.DOWN
            self.state = ElevatorState.MOVING
            self.current_floor += self.direction.value

# Strategy: how to choose the car for a hall call
class NearestCarStrategy:
    def select_car(self, cars, floor, direction):
        return min(cars, key=lambda c: abs(c.current_floor - floor))

class FCFSStrategy:
    def select_car(self, cars, floor, direction):
        return cars[0]               # simplest possible: always car 0

class ElevatorSystem:
    def __init__(self, num_cars, strategy=None):
        self.cars = [Elevator(i) for i in range(num_cars)]
        self.strategy = strategy or NearestCarStrategy()

    def request_hall_call(self, floor, direction):   # external
        car = self.strategy.select_car(self.cars, floor, direction)
        car.add_stop(floor)
        print(f"Hall call @{floor} -> car {car.id}")

    def press_floor(self, car_id, target):           # internal
        self.cars[car_id].add_stop(target)

    def tick(self):
        for c in self.cars:
            c.step()                # advance every car one step

# --- demo ---
system = ElevatorSystem(2)
system.request_hall_call(5, Direction.UP)   # someone waiting on floor 5
system.press_floor(0, 8)                    # rider in car 0 picks floor 8
for _ in range(12):
    system.tick()
Why this design scores well: the SchedulingStrategy interface is the centerpiece — you can ship FCFS first, then drop in NearestCarStrategy or a full ScanStrategy without touching Elevator or ElevatorSystem (Open/Closed). The car's behavior is driven by an ElevatorState enum so it never "moves" while doors are open. Keeping stops in a sorted set lets the car sweep like a real elevator (the SCAN/LOOK algorithm) instead of bouncing around.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Smarter scheduling: implement full SCAN/LOOK — keep sweeping in one direction, serving all same-direction calls, then reverse. Score cars by estimated wait time, not just distance.
  • Real concurrency: give each car its own thread that blocks on a BlockingQueue of stops (producer-consumer); the controller is the producer.
  • Capacity & weight limits: refuse new car-button presses when full.
  • Express / restricted cars: some cars only serve certain floors.
  • Emergency & maintenance: a State for OUT_OF_SERVICE; fire mode that sends all cars to the ground floor.
  • Displays & doors as observers: notify the Display and Door on each state change.
Common mistakes: hardcoding one dispatch rule with if/else instead of a Strategy; conflating internal (target floor) and external (floor + direction) requests; letting a car change floors while in the DOORS_OPEN state; ignoring concurrency so two threads mutate the same stop queue; and making each car re-decide globally instead of letting the controller dispatch. Avoid these and you're in the top tier.

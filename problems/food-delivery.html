LLD Problem Hard

Design a Food Delivery System

A Swiggy/Zomato-style platform that connects customers, restaurants, and delivery partners. Customers browse menus, place orders, restaurants accept and cook, and a delivery partner is dispatched — all while the order moves through a strict lifecycle and everyone gets notified.

📋 Problem Statement

Design the core of a food delivery service. A Customer browses nearby Restaurants, adds MenuItems to a Cart, and places an Order. The restaurant accepts and prepares it; the system assigns the nearest free DeliveryPartner; the order then flows PLACED → CONFIRMED → PREPARING → OUT_FOR_DELIVERY → DELIVERED. At every transition, the customer (and other subscribers) must be notified. Pricing/assignment rules must be swappable.

Step 1 — Gather Requirements

Pin down scope first. Call out the must-haves versus the nice-to-haves so the interviewer sees you can prioritize.

✓ Functional

  • Browse restaurants and their menus.
  • Build a cart and place an order with payment.
  • Restaurant accepts/rejects; order is prepared.
  • Assign the nearest available delivery partner.
  • Track order through a fixed lifecycle of states.
  • Notify the customer on every status change.

◆ Non-Functional

  • Valid transitions only: an order can't skip from PLACED to DELIVERED.
  • Extensibility: add new assignment / pricing rules without edits.
  • Decoupling: order logic shouldn't know who is listening.
  • Concurrency: one partner is never double-assigned.
  • Low-latency partner lookup at scale.

Step 2 — Identify the Core Entities

Pull the nouns out of the statement — these become your classes:

  • Customer — places orders; subscribes to updates.
  • Restaurant — owns a menu; accepts/rejects orders.
  • MenuItem — a dish with a price and availability.
  • Cart — a customer's in-progress selection of items.
  • Order — the central entity; holds a state and line items.
  • OrderState (interface) — the State pattern: each lifecycle phase is a class controlling what comes next.
  • DeliveryPartner — a rider with a location and availability.
  • DeliveryAssignmentStrategy — pluggable rule to pick a partner (e.g. nearest).
  • Payment — captures the order amount.
  • Observer / Notification — listeners notified on state changes.

Step 3 — Class Diagram

Three patterns hang off the Order: State drives its lifecycle, Strategy picks the rider, and Observer fans out notifications.

Step 4 — Patterns in Play

Name the patterns as you go — it signals design maturity:

State — order lifecycle transitions Strategy — pluggable partner assignment Observer — fan-out status notifications Singleton — central dispatch service Locking — safe partner assignment

Step 5 — Implementation

A runnable core showing the three patterns working together: the order moves through states, Strategy picks a rider, and observers get pinged on every change.

import java.util.*;

// ---- Domain ----
class MenuItem {
    final String name; final double price;
    MenuItem(String n, double p) { name = n; price = p; }
}

class DeliveryPartner {
    final String name; int x, y; boolean free = true;
    DeliveryPartner(String n, int x, int y) { name = n; this.x = x; this.y = y; }
}

// ---- Observer pattern ----
interface OrderObserver { void onStatusChange(Order o, String status); }
class CustomerNotifier implements OrderObserver {
    public void onStatusChange(Order o, String status) {
        System.out.println("[SMS] Order " + o.id + " is now " + status);
    }
}

// ---- Strategy pattern: how to choose a delivery partner ----
interface DeliveryAssignmentStrategy {
    DeliveryPartner assign(List<DeliveryPartner> pool, int rx, int ry);
}
class NearestPartnerStrategy implements DeliveryAssignmentStrategy {
    public synchronized DeliveryPartner assign(List<DeliveryPartner> pool, int rx, int ry) {
        DeliveryPartner best = null; long bestD = Long.MAX_VALUE;
        for (DeliveryPartner p : pool) {
            if (!p.free) continue;
            long d = (long)(p.x-rx)*(p.x-rx) + (long)(p.y-ry)*(p.y-ry);
            if (d < bestD) { bestD = d; best = p; }
        }
        if (best != null) best.free = false;   // reserve atomically
        return best;
    }
}

// ---- State pattern: each phase decides the legal next() ----
interface OrderState {
    String name();
    OrderState next(Order o);   // advance; throws if terminal
}
class Placed implements OrderState {
    public String name() { return "PLACED"; }
    public OrderState next(Order o) { return new Confirmed(); }
}
class Confirmed implements OrderState {
    public String name() { return "CONFIRMED"; }
    public OrderState next(Order o) { return new Preparing(); }
}
class Preparing implements OrderState {
    public String name() { return "PREPARING"; }
    public OrderState next(Order o) {
        o.assignPartner();              // dispatch on leaving the kitchen
        return new OutForDelivery();
    }
}
class OutForDelivery implements OrderState {
    public String name() { return "OUT_FOR_DELIVERY"; }
    public OrderState next(Order o) {
        if (o.partner != null) o.partner.free = true;   // release rider
        return new Delivered();
    }
}
class Delivered implements OrderState {
    public String name() { return "DELIVERED"; }
    public OrderState next(Order o) {
        throw new IllegalStateException("Order already delivered");
    }
}

class Order {
    final String id; final int rx, ry;   // restaurant location
    final List<MenuItem> items;
    final List<DeliveryPartner> pool;
    final DeliveryAssignmentStrategy strategy;
    private final List<OrderObserver> observers = new ArrayList<>();
    OrderState state = new Placed();
    DeliveryPartner partner;

    Order(String id, int rx, int ry, List<MenuItem> items,
          List<DeliveryPartner> pool, DeliveryAssignmentStrategy s) {
        this.id = id; this.rx = rx; this.ry = ry;
        this.items = items; this.pool = pool; this.strategy = s;
    }
    void subscribe(OrderObserver o) { observers.add(o); }
    double total() { return items.stream().mapToDouble(i -> i.price).sum(); }

    void assignPartner() {
        partner = strategy.assign(pool, rx, ry);
        if (partner == null) throw new IllegalStateException("No riders free");
        System.out.println("  -> assigned " + partner.name);
    }

    void advance() {                       // delegate to current state
        state = state.next(this);
        for (OrderObserver o : observers) o.onStatusChange(this, state.name());
    }
}

public class Demo {
    public static void main(String[] a) {
        List<DeliveryPartner> pool = List.of(
            new DeliveryPartner("Asha", 10, 10),
            new DeliveryPartner("Ravi", 2, 3));
        Order o = new Order("OD1", 1, 1,
            List.of(new MenuItem("Pizza", 299)), pool, new NearestPartnerStrategy());
        o.subscribe(new CustomerNotifier());
        System.out.println("Total: " + o.total());
        o.advance();  // CONFIRMED
        o.advance();  // PREPARING
        o.advance();  // OUT_FOR_DELIVERY (picks Ravi, nearest to 1,1)
        o.advance();  // DELIVERED
    }
}

from abc import ABC, abstractmethod
import threading

# ---- Domain ----
class MenuItem:
    def __init__(self, name, price):
        self.name, self.price = name, price

class DeliveryPartner:
    def __init__(self, name, x, y):
        self.name, self.x, self.y, self.free = name, x, y, True

# ---- Observer pattern ----
class OrderObserver(ABC):
    @abstractmethod
    def on_status_change(self, order, status): ...

class CustomerNotifier(OrderObserver):
    def on_status_change(self, order, status):
        print(f"[SMS] Order {order.id} is now {status}")

# ---- Strategy pattern: how to choose a delivery partner ----
class DeliveryAssignmentStrategy(ABC):
    @abstractmethod
    def assign(self, pool, rx, ry): ...

class NearestPartnerStrategy(DeliveryAssignmentStrategy):
    def __init__(self):
        self._lock = threading.Lock()
    def assign(self, pool, rx, ry):
        with self._lock:                 # reserve atomically
            free = [p for p in pool if p.free]
            if not free: return None
            best = min(free, key=lambda p: (p.x-rx)**2 + (p.y-ry)**2)
            best.free = False
            return best

# ---- State pattern: each phase decides the legal next() ----
class OrderState(ABC):
    name = "?"
    @abstractmethod
    def next(self, order): ...

class Placed(OrderState):
    name = "PLACED"
    def next(self, order): return Confirmed()
class Confirmed(OrderState):
    name = "CONFIRMED"
    def next(self, order): return Preparing()
class Preparing(OrderState):
    name = "PREPARING"
    def next(self, order):
        order.assign_partner()          # dispatch on leaving the kitchen
        return OutForDelivery()
class OutForDelivery(OrderState):
    name = "OUT_FOR_DELIVERY"
    def next(self, order):
        if order.partner: order.partner.free = True   # release rider
        return Delivered()
class Delivered(OrderState):
    name = "DELIVERED"
    def next(self, order):
        raise RuntimeError("Order already delivered")

class Order:
    def __init__(self, id, rx, ry, items, pool, strategy):
        self.id, self.rx, self.ry = id, rx, ry
        self.items, self.pool, self.strategy = items, pool, strategy
        self.state = Placed()
        self.partner = None
        self._observers = []

    def subscribe(self, obs): self._observers.append(obs)
    def total(self): return sum(i.price for i in self.items)

    def assign_partner(self):
        self.partner = self.strategy.assign(self.pool, self.rx, self.ry)
        if self.partner is None: raise RuntimeError("No riders free")
        print(f"  -> assigned {self.partner.name}")

    def advance(self):                  # delegate to current state
        self.state = self.state.next(self)
        for o in self._observers:
            o.on_status_change(self, self.state.name)

if __name__ == "__main__":
    pool = [DeliveryPartner("Asha", 10, 10),
            DeliveryPartner("Ravi", 2, 3)]
    o = Order("OD1", 1, 1, [MenuItem("Pizza", 299)],
              pool, NearestPartnerStrategy())
    o.subscribe(CustomerNotifier())
    print("Total:", o.total())
    o.advance()   # CONFIRMED
    o.advance()   # PREPARING
    o.advance()   # OUT_FOR_DELIVERY (picks Ravi, nearest to 1,1)
    o.advance()   # DELIVERED
Why this design scores well: the OrderState classes make illegal transitions impossible — there's no way to call next() on a Delivered order without an exception, and the lifecycle logic lives in the states, not a giant switch on the Order. The DeliveryAssignmentStrategy lets you swap "nearest" for "highest-rated" or "load-balanced" without touching Order (Open/Closed). And the Observer list means Order publishes status changes without knowing whether an SMS, push, or restaurant dashboard is listening.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Restaurant rejection / cancellation: add a Cancelled state and transitions from Placed/Confirmed.
  • Fast nearest lookup: index riders in a geo-grid / quadtree instead of scanning the pool — O(1) bucket lookup.
  • Pricing: add a PricingStrategy (surge, coupons, delivery fee) alongside the assignment strategy.
  • Payment flow: a PaymentProcessor with card/UPI/wallet via Strategy; block Confirmed until payment succeeds.
  • Reliability: persist state transitions to an event log so a crash can resume the order; retry rider assignment on rejection.
Common mistakes: modeling the lifecycle as a single mutable String status with scattered if checks instead of the State pattern; hardcoding "nearest" so a new rule means editing Order; having Order call the SMS/push services directly instead of publishing to observers; forgetting to lock the partner pool so two orders grab the same rider. Avoid all four and you're in the top tier.

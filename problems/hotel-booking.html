LLD Problem Hard

Design a Hotel Booking System

Search rooms for a date range, reserve without double-booking, and run the reservation through its lifecycle. The hard part is the interval-overlap check done safely under concurrency.

📋 Problem Statement

Design a hotel booking system. A hotel has rooms grouped by room type (Standard, Deluxe, Suite). A guest searches for available rooms over a check-in / check-out date range, reserves one, pays, then checks in and later checks out. The system must never double-book a room for overlapping dates — even when two guests reserve the last room at the same instant — and must support pluggable pricing.

Step 1 — Gather Requirements

Booking systems live or die on the availability check and concurrency. State the scope and the consistency guarantee up front.

✓ Functional

  • Search available rooms for a [checkIn, checkOut) range.
  • Reserve a room; create a Reservation.
  • Process payment for the reservation.
  • Check-in and check-out flows.
  • Cancel a reservation and release the dates.
  • Room types with different base prices.

◆ Non-Functional

  • No double-booking: overlapping dates on one room must be impossible.
  • Concurrency: simultaneous reserves of the last room must serialize.
  • Extensible pricing: seasonal / weekend / dynamic rates.
  • Auditable lifecycle: valid reservation state transitions only.

Step 2 — Identify the Core Entities

Find the nouns; the interesting ones are Reservation (a state machine) and the availability check:

  • Hotel — owns room types and rooms.
  • RoomType — Standard / Deluxe / Suite; carries a base price.
  • Room — a physical room of a type; holds its booked date ranges.
  • Guest — the person booking.
  • Reservation — a booking with a lifecycle (PENDING → CONFIRMED → CHECKED_IN → CHECKED_OUT, or CANCELLED).
  • DateRange — [checkIn, checkOut); the overlap test lives here.
  • PricingStrategy — pluggable rate calculation.
  • Payment — captures money for a reservation.
  • BookingService — Facade / Singleton orchestrating search + reserve.

Step 5 — Implementation

The heart of the problem: the interval-overlap test and a per-room lock so two guests can never confirm overlapping stays. Two ranges [a,b) and [c,d) overlap iff a < d and c < b.

import java.time.LocalDate;
import java.util.*;
import java.util.concurrent.locks.ReentrantLock;

enum RoomType { STANDARD, DELUXE, SUITE }
enum ReservationState { PENDING, CONFIRMED, CHECKED_IN, CHECKED_OUT, CANCELLED }

// Half-open interval [checkIn, checkOut); the overlap test lives here
record DateRange(LocalDate checkIn, LocalDate checkOut) {
    boolean overlaps(DateRange o) {
        return checkIn.isBefore(o.checkOut) && o.checkIn.isBefore(checkOut);
    }
    long nights() { return checkOut.toEpochDay() - checkIn.toEpochDay(); }
}

// Strategy: price varies independently of the booking flow
interface PricingStrategy { double price(RoomType type, DateRange range); }
class FlatPricing implements PricingStrategy {
    private static final Map<RoomType, Double> BASE =
        Map.of(RoomType.STANDARD, 100.0, RoomType.DELUXE, 180.0, RoomType.SUITE, 320.0);
    public double price(RoomType t, DateRange r) { return BASE.get(t) * r.nights(); }
}

class Room {
    final String id; final RoomType type;
    private final List<DateRange> booked = new ArrayList<>();
    final ReentrantLock lock = new ReentrantLock();   // one lock per room

    Room(String id, RoomType type) { this.id = id; this.type = type; }

    boolean isAvailable(DateRange r) {
        for (DateRange b : booked) if (b.overlaps(r)) return false;
        return true;
    }
    void book(DateRange r) { booked.add(r); }
    void release(DateRange r) { booked.remove(r); }
}

class Reservation {
    final Room room; final String guest; final DateRange range; final double amount;
    ReservationState state = ReservationState.PENDING;
    Reservation(Room room, String guest, DateRange r, double amt) {
        this.room = room; this.guest = guest; this.range = r; this.amount = amt;
    }
    // State pattern: only legal transitions allowed
    void confirm()   { require(ReservationState.PENDING);    state = ReservationState.CONFIRMED; }
    void checkIn()   { require(ReservationState.CONFIRMED);  state = ReservationState.CHECKED_IN; }
    void checkOut()  { require(ReservationState.CHECKED_IN); state = ReservationState.CHECKED_OUT; room.release(range); }
    private void require(ReservationState s) {
        if (state != s) throw new IllegalStateException("Bad transition from " + state);
    }
}

// Facade + Singleton: the one entry point for booking
class BookingService {
    private static final BookingService INSTANCE = new BookingService();
    public static BookingService get() { return INSTANCE; }

    private final List<Room> rooms = new ArrayList<>();
    private PricingStrategy pricing = new FlatPricing();

    public void addRoom(Room r) { rooms.add(r); }

    // Availability search: interval-overlap filter
    public List<Room> search(RoomType type, DateRange range) {
        List<Room> out = new ArrayList<>();
        for (Room r : rooms)
            if (r.type == type && r.isAvailable(range)) out.add(r);
        return out;
    }

    // Reserve: lock the room, re-check, then book — no double-booking
    public Reservation reserve(Room room, String guest, DateRange range) {
        room.lock.lock();
        try {
            if (!room.isAvailable(range))      // re-check INSIDE the lock
                throw new IllegalStateException("Room just got booked");
            double amt = pricing.price(room.type, range);
            room.book(range);
            Reservation res = new Reservation(room, guest, range, amt);
            res.confirm();                     // assume payment captured
            return res;
        } finally { room.lock.unlock(); }
    }
}

class Demo {
    public static void main(String[] a) {
        BookingService svc = BookingService.get();
        svc.addRoom(new Room("101", RoomType.DELUXE));
        DateRange stay = new DateRange(LocalDate.of(2026,7,1), LocalDate.of(2026,7,4));
        Room room = svc.search(RoomType.DELUXE, stay).get(0);
        Reservation r = svc.reserve(room, "Ann", stay);
        System.out.println("Booked $" + r.amount + ", state " + r.state);
    }
}

import threading
from dataclasses import dataclass, field
from datetime import date
from enum import Enum

class RoomType(Enum):
    STANDARD = 1; DELUXE = 2; SUITE = 3

class State(Enum):
    PENDING = 1; CONFIRMED = 2; CHECKED_IN = 3; CHECKED_OUT = 4; CANCELLED = 5

@dataclass(frozen=True)                       # [check_in, check_out)
class DateRange:
    check_in: date
    check_out: date
    def overlaps(self, o):
        return self.check_in < o.check_out and o.check_in < self.check_out
    def nights(self): return (self.check_out - self.check_in).days

# Strategy: pricing is pluggable
class FlatPricing:
    BASE = {RoomType.STANDARD: 100, RoomType.DELUXE: 180, RoomType.SUITE: 320}
    def price(self, room_type, rng):
        return self.BASE[room_type] * rng.nights()

class Room:
    def __init__(self, id, room_type):
        self.id, self.type = id, room_type
        self.booked = []
        self.lock = threading.Lock()           # one lock per room
    def is_available(self, rng):
        return not any(b.overlaps(rng) for b in self.booked)
    def book(self, rng): self.booked.append(rng)
    def release(self, rng): self.booked.remove(rng)

class Reservation:
    def __init__(self, room, guest, rng, amount):
        self.room, self.guest, self.range, self.amount = room, guest, rng, amount
        self.state = State.PENDING
    # State pattern: only legal transitions allowed
    def confirm(self):   self._require(State.PENDING);    self.state = State.CONFIRMED
    def check_in(self):  self._require(State.CONFIRMED);  self.state = State.CHECKED_IN
    def check_out(self):
        self._require(State.CHECKED_IN)
        self.state = State.CHECKED_OUT
        self.room.release(self.range)
    def _require(self, s):
        if self.state != s:
            raise RuntimeError(f"Bad transition from {self.state}")

# Facade + Singleton: one entry point for booking
class BookingService:
    _instance = None
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.rooms = []
            cls._instance.pricing = FlatPricing()
        return cls._instance

    def add_room(self, room): self.rooms.append(room)

    # Availability search: interval-overlap filter
    def search(self, room_type, rng):
        return [r for r in self.rooms
                if r.type == room_type and r.is_available(rng)]

    # Reserve: lock room, re-check inside lock, then book
    def reserve(self, room, guest, rng):
        with room.lock:
            if not room.is_available(rng):    # re-check INSIDE the lock
                raise RuntimeError("Room just got booked")
            amt = self.pricing.price(room.type, rng)
            room.book(rng)
            res = Reservation(room, guest, rng, amt)
            res.confirm()                      # assume payment captured
            return res

if __name__ == "__main__":
    svc = BookingService()
    svc.add_room(Room("101", RoomType.DELUXE))
    stay = DateRange(date(2026,7,1), date(2026,7,4))
    room = svc.search(RoomType.DELUXE, stay)[0]
    r = svc.reserve(room, "Ann", stay)
    print(f"Booked ${r.amount}, state {r.state}")
Why this design scores well: the overlap test a < d && c < b on half-open intervals is the crux — it correctly treats a same-day checkout/check-in as not overlapping. The per-room lock with a re-check inside the lock is the concurrency detail interviewers hunt for: checking availability and then booking must be one atomic step, or two threads both pass the check and double-book. PricingStrategy (Open/Closed) and the guarded Reservation state transitions round it out.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Faster availability: replace the linear scan with an interval tree or a per-room sorted set of ranges for O(log n) overlap checks.
  • Distributed locking: across many servers, the in-process lock fails — use a DB unique constraint on (room_id, date) or an optimistic-version / SELECT … FOR UPDATE.
  • Hold & expire: a PENDING reservation should auto-release after N minutes if payment never completes (timeout / scheduled job).
  • Dynamic pricing: add SeasonalPricing / OccupancyPricing as new Strategy classes.
  • Overbooking policy: deliberately allow a small over-sell with a waitlist, mirroring real hotels.
  • Payment states: integrate a payment gateway and tie capture/refund to the reservation lifecycle.
Common mistakes: getting the overlap condition wrong (using <= and rejecting valid same-day turnovers); checking availability outside the lock then booking inside it (the classic double-book race); storing dates as strings so comparisons break; and letting any reservation jump to any state instead of guarding transitions.

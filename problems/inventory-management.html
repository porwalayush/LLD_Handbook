LLD Problem Hard

Design an Inventory Management System

Track stock for thousands of products across many warehouses, reserve units for orders, and auto-replenish from suppliers — all while concurrent gates mutate the same quantities.

📋 Problem Statement

Design the inventory engine behind an e-commerce backend. The system manages products (identified by SKU) stored as stock across multiple warehouses. It must support adding and removing stock, reserving units for an order (so two customers never buy the last item), and automatically raising a purchase order to a supplier when a product drops below its reorder level. Interested parties — dashboards, alerting, suppliers — should be notified of low-stock events. Many warehouse gates and order workers mutate the same quantities at once, so updates must be thread-safe.

Step 1 — Gather Requirements

Always start by pinning down scope. State your assumptions out loud; interviewers want to see you separate the must-haves from the nice-to-haves.

✓ Functional

  • Register products (SKU) and warehouses.
  • addStock / removeStock for a product at a warehouse.
  • Reserve units for an order, then commit or release them.
  • Trigger replenishment when available stock falls below the reorder level.
  • Notify observers (alerts, dashboards) on low-stock and reorder events.
  • Query total available quantity for a SKU across all warehouses.

◆ Non-Functional

  • Concurrency: many gates/orders mutate the same SKU — no oversell, no lost updates.
  • Extensibility: pluggable reorder policy and notification channels.
  • Low contention: a hot SKU shouldn't block unrelated SKUs.
  • Consistency: reserved + available must always equal on-hand.

Step 2 — Identify the Core Entities

Find the nouns in the problem. These become your classes:

  • Product — a catalog item identified by an immutable SKU, with a reorder level and reorder quantity.
  • Warehouse — a physical location holding stock items.
  • StockItem — the (product, warehouse) pair holding available and reserved counts. This is the unit that gets locked.
  • InventoryManager — the façade orchestrating add/remove/reserve and replenishment.
  • Supplier — a vendor that fulfils purchase orders for products.
  • PurchaseOrder — a replenishment request raised against a supplier.
  • ReplenishmentStrategy — pluggable rule deciding when and how much to reorder (Strategy).
  • StockObserver — subscribers notified of low-stock / reorder events (Observer).

Step 4 — Patterns in Play

Name the patterns as you design — it signals maturity. This problem naturally uses:

Step 5 — Implementation

A focused skeleton showing the key classes and the reserve / replenish flow. The crux is the per-StockItem lock: it makes each SKU's update atomic without serializing the whole warehouse.

import java.util.*;
import java.util.concurrent.*;
import java.util.concurrent.locks.ReentrantLock;

class Product {
    final String sku;
    final int reorderLevel, reorderQty;   // reorder when available < reorderLevel
    Product(String sku, int level, int qty) { this.sku = sku; reorderLevel = level; reorderQty = qty; }
}

// One StockItem per (SKU, warehouse). It owns the lock that guards its counters.
class StockItem {
    final Product product; final String warehouseId;
    private int available = 0, reserved = 0;
    final ReentrantLock lock = new ReentrantLock();
    StockItem(Product p, String w) { product = p; warehouseId = w; }
    int available() { return available; }
    void add(int n)     { available += n; }
    void reserve(int n) { available -= n; reserved += n; }   // on-hand unchanged
    void commit(int n)  { reserved  -= n; }                  // units shipped out
    void release(int n) { reserved -= n; available += n; }   // order cancelled
}

// Strategy: decide WHEN and HOW MUCH to reorder.
interface ReplenishmentStrategy {
    // returns qty to order, or 0 if no reorder needed
    int reorderQty(Product p, int availableAcrossWarehouses);
}
class ThresholdReorder implements ReplenishmentStrategy {
    public int reorderQty(Product p, int avail) {
        return avail < p.reorderLevel ? p.reorderQty : 0;
    }
}

// Observer: fan out low-stock / reorder events.
interface StockObserver { void onLowStock(Product p, int avail, int ordered); }
class LowStockAlert implements StockObserver {
    public void onLowStock(Product p, int avail, int ordered) {
        System.out.println("ALERT " + p.sku + ": avail=" + avail + ", reordered " + ordered);
    }
}

class Supplier { final String name; Supplier(String n) { name = n; } }
class PurchaseOrder {
    final String sku; final int qty; final Supplier supplier;
    PurchaseOrder(String sku, int qty, Supplier s) { this.sku = sku; this.qty = qty; supplier = s; }
}

class InventoryManager {
    // SKU -> (warehouseId -> StockItem). ConcurrentHashMap = safe map structure.
    private final Map<String, Map<String, StockItem>> stock = new ConcurrentHashMap<>();
    private final Map<String, Product> products = new ConcurrentHashMap<>();
    private final List<StockObserver> observers = new CopyOnWriteArrayList<>();
    private final ReplenishmentStrategy strategy;
    private final Supplier defaultSupplier = new Supplier("AcmeCorp");

    InventoryManager(ReplenishmentStrategy s) { strategy = s; }
    void register(Product p) { products.put(p.sku, p); }
    void subscribe(StockObserver o) { observers.add(o); }

    private StockItem item(String sku, String wh) {
        return stock.computeIfAbsent(sku, k -> new ConcurrentHashMap<>())
                    .computeIfAbsent(wh, k -> new StockItem(products.get(sku), wh));
    }

    void addStock(String sku, String wh, int n) {
        StockItem it = item(sku, wh);
        it.lock.lock();
        try { it.add(n); } finally { it.lock.unlock(); }
    }

    // Reserve n units for an order. Returns true only if the full quantity was held.
    boolean reserve(String sku, String wh, int n) {
        StockItem it = item(sku, wh);
        it.lock.lock();                          // guard check-then-act → no oversell
        try {
            if (it.available() < n) return false;
            it.reserve(n);
        } finally { it.lock.unlock(); }
        maybeReplenish(sku);                  // check threshold after the mutation
        return true;
    }

    // Sum availability across warehouses; reorder if the strategy says so.
    private void maybeReplenish(String sku) {
        Product p = products.get(sku);
        int total = stock.getOrDefault(sku, Map.of()).values().stream()
                        .mapToInt(StockItem::available).sum();
        int qty = strategy.reorderQty(p, total);
        if (qty > 0) {
            new PurchaseOrder(sku, qty, defaultSupplier);   // raise PO to supplier
            for (StockObserver o : observers) o.onLowStock(p, total, qty);
        }
    }

    public static void main(String[] args) {
        InventoryManager inv = new InventoryManager(new ThresholdReorder());
        inv.subscribe(new LowStockAlert());
        inv.register(new Product("SKU-1", 5, 50));   // reorder when avail < 5
        inv.addStock("SKU-1", "WH-A", 8);
        System.out.println(inv.reserve("SKU-1", "WH-A", 4));  // true → avail drops to 4 → reorder fires
    }
}

import threading
from dataclasses import dataclass, field

@dataclass
class Product:
    sku: str
    reorder_level: int      # reorder when available < reorder_level
    reorder_qty: int

# One StockItem per (SKU, warehouse). It owns the lock guarding its counters.
class StockItem:
    def __init__(self, product, warehouse_id):
        self.product, self.warehouse_id = product, warehouse_id
        self.available, self.reserved = 0, 0
        self.lock = threading.Lock()
    def add(self, n):     self.available += n
    def reserve(self, n): self.available -= n; self.reserved += n   # on-hand unchanged
    def commit(self, n):  self.reserved  -= n                     # units shipped out
    def release(self, n): self.reserved -= n; self.available += n   # order cancelled

# Strategy: decide WHEN and HOW MUCH to reorder.
class ThresholdReorder:
    def reorder_qty(self, product, available):
        return product.reorder_qty if available < product.reorder_level else 0

# Observer: fan out low-stock / reorder events.
class LowStockAlert:
    def on_low_stock(self, product, available, ordered):
        print(f"ALERT {product.sku}: avail={available}, reordered {ordered}")

@dataclass
class Supplier:
    name: str

@dataclass
class PurchaseOrder:
    sku: str
    qty: int
    supplier: Supplier

class InventoryManager:
    def __init__(self, strategy):
        self.stock = {}            # sku -> {warehouse_id: StockItem}
        self.products = {}
        self.observers = []
        self.strategy = strategy
        self.supplier = Supplier("AcmeCorp")
        self._map_lock = threading.Lock()   # guards lazy creation of items

    def register(self, product): self.products[product.sku] = product
    def subscribe(self, obs):    self.observers.append(obs)

    def _item(self, sku, wh):
        with self._map_lock:                       # create-once, safe map structure
            return self.stock.setdefault(sku, {}).setdefault(
                wh, StockItem(self.products[sku], wh))

    def add_stock(self, sku, wh, n):
        it = self._item(sku, wh)
        with it.lock:
            it.add(n)

    # Reserve n units for an order. True only if the full quantity was held.
    def reserve(self, sku, wh, n):
        it = self._item(sku, wh)
        with it.lock:                              # guard check-then-act → no oversell
            if it.available < n:
                return False
            it.reserve(n)
        self._maybe_replenish(sku)                # check threshold after mutation
        return True

    # Sum availability across warehouses; reorder if the strategy says so.
    def _maybe_replenish(self, sku):
        product = self.products[sku]
        total = sum(i.available for i in self.stock.get(sku, {}).values())
        qty = self.strategy.reorder_qty(product, total)
        if qty > 0:
            PurchaseOrder(sku, qty, self.supplier)    # raise PO to supplier
            for o in self.observers:
                o.on_low_stock(product, total, qty)

if __name__ == "__main__":
    inv = InventoryManager(ThresholdReorder())
    inv.subscribe(LowStockAlert())
    inv.register(Product("SKU-1", reorder_level=5, reorder_qty=50))
    inv.add_stock("SKU-1", "WH-A", 8)
    print(inv.reserve("SKU-1", "WH-A", 4))   # True → avail 4 → reorder fires
Why this design scores well: the per-StockItem lock makes each reserve a single atomic check-then-act, so two order workers can never both grab the last unit (no oversell) — yet mutating SKU-A never blocks SKU-B, keeping contention low. Splitting available from reserved keeps the invariant on-hand = available + reserved true at all times. The ReplenishmentStrategy means a new policy (forecast-based, seasonal) is a new class — never an edit to InventoryManager (Open/Closed). And the StockObserver list lets you add alerting or a supplier feed without touching the reserve path.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Lock-free counters: if you only ever add/subtract (no check-then-act), an AtomicInteger / CAS loop beats a lock — see atomics. Reserve still needs the lock because it reads and writes together.
  • Multi-warehouse reserve: fulfil one order from several warehouses — acquire locks in a fixed SKU order to avoid deadlock.
  • Reservation expiry: a cart hold times out and auto-releases via a background sweeper.
  • Async replenishment: push reorder events onto a queue so the reserve call returns immediately.
  • Persistence & idempotency: back counters with a database using optimistic locking (version column); make PO creation idempotent.
  • Distributed inventory: shard by SKU; use a distributed lock or per-shard owner to keep the in-memory model valid at scale.
Common mistakes: a global lock around the whole manager (kills throughput — lock per StockItem instead); doing the availability check and the decrement as two separate steps without a lock (classic oversell race); conflating available and reserved into one counter so you can't tell held stock from sellable stock; hardcoding the reorder rule with if/else instead of a Strategy; and notifying observers inside the lock, which lets a slow listener stall every other order.

LLD Problem Easy

Design a Parking Lot

The single most-asked LLD interview problem. Model a multi-floor parking lot that admits vehicles, assigns spots, issues tickets, and charges on exit.

📋 Problem Statement

Design a parking lot system for a mall. The lot has multiple floors, each with a fixed number of parking spots. Spots come in sizes — motorcycle, compact, and large. A vehicle entering is assigned the nearest available spot that fits, and receives a ticket. On exit, the system computes the fee based on duration and vehicle type, and frees the spot.

Step 1 — Gather Requirements

Always start by pinning down scope. State your assumptions out loud; interviewers want to see you separate the must-haves from the nice-to-haves.

✓ Functional

  • Multiple floors, each with many spots of different sizes.
  • Park a vehicle → assign a fitting spot, issue a ticket.
  • Unpark → free the spot, compute the fee.
  • Support vehicle types: motorcycle, car, bus/truck.
  • Track real-time availability per floor.

◆ Non-Functional

  • Concurrency: many entry/exit gates act at once — no double-assigning a spot.
  • Extensibility: easy to add spot sizes or pricing rules.
  • Fast lookups for the nearest available spot.
  • Pluggable payment / pricing strategy.

Step 2 — Identify the Core Entities

Find the nouns in the problem. These become your classes:

  • ParkingLot — the top-level system (often a Singleton); holds floors.
  • ParkingFloor — holds a collection of spots; tracks availability.
  • ParkingSpot — a single spot with a size; knows if it's free and which vehicle occupies it.
  • Vehicle (abstract) → Motorcycle, Car, Truck — each maps to a required spot size.
  • Ticket — issued on entry; records spot, vehicle, and entry time.
  • FeeStrategy — pluggable pricing (per-hour, flat, vehicle-based).
  • EntryGate / ExitGate — orchestrate park/unpark.

Step 4 — Patterns in Play

Name the patterns as you design — it signals maturity. This problem naturally uses:

Singleton — one ParkingLot Strategy — pluggable fee rules Factory — create vehicles/spots Locking — safe spot assignment

Step 5 — Implementation

A focused skeleton showing the key classes and the park/unpark flow. In an interview you'd build this up incrementally; here's a clean, runnable core.

enum SpotSize { MOTO, COMPACT, LARGE }

abstract class Vehicle {
    final String plate;
    final SpotSize size;
    Vehicle(String plate, SpotSize size) { this.plate = plate; this.size = size; }
}
class Car extends Vehicle { Car(String p) { super(p, SpotSize.COMPACT); } }
class Truck extends Vehicle { Truck(String p) { super(p, SpotSize.LARGE); } }

class ParkingSpot {
    final String id; final SpotSize size;
    private Vehicle vehicle;          // null = free
    ParkingSpot(String id, SpotSize size) { this.id = id; this.size = size; }
    boolean isFree() { return vehicle == null; }
    void assign(Vehicle v) { this.vehicle = v; }
    void free() { this.vehicle = null; }
}

// Strategy pattern: pricing varies independently of the lot
interface FeeStrategy { double calculate(Ticket t, long exitMs); }
class HourlyFeeStrategy implements FeeStrategy {
    public double calculate(Ticket t, long exitMs) {
        long hours = Math.max(1, (exitMs - t.entryMs) / 3600000);
        return hours * (t.vehicle.size == SpotSize.LARGE ? 30 : 20);
    }
}

class Ticket {
    final Vehicle vehicle; final ParkingSpot spot; final long entryMs;
    Ticket(Vehicle v, ParkingSpot s, long t) { vehicle = v; spot = s; entryMs = t; }
}

class ParkingLot {
    private static final ParkingLot INSTANCE = new ParkingLot();  // Singleton
    public static ParkingLot get() { return INSTANCE; }

    private final List<ParkingSpot> spots = new ArrayList<>();
    private final FeeStrategy feeStrategy = new HourlyFeeStrategy();

    // synchronized so two gates never grab the same spot
    public synchronized Ticket park(Vehicle v) {
        for (ParkingSpot s : spots) {
            if (s.isFree() && s.size.ordinal() >= v.size.ordinal()) {
                s.assign(v);
                return new Ticket(v, s, System.currentTimeMillis());
            }
        }
        throw new IllegalStateException("Parking full");
    }

    public synchronized double unpark(Ticket t) {
        double fee = feeStrategy.calculate(t, System.currentTimeMillis());
        t.spot.free();
        return fee;
    }
}

from enum import IntEnum
from abc import ABC
import time, threading

class SpotSize(IntEnum):
    MOTO = 0; COMPACT = 1; LARGE = 2

class Vehicle(ABC):
    def __init__(self, plate, size):
        self.plate, self.size = plate, size

class Car(Vehicle):
    def __init__(self, plate): super().__init__(plate, SpotSize.COMPACT)
class Truck(Vehicle):
    def __init__(self, plate): super().__init__(plate, SpotSize.LARGE)

class ParkingSpot:
    def __init__(self, id, size):
        self.id, self.size, self.vehicle = id, size, None
    def is_free(self): return self.vehicle is None

# Strategy pattern: pricing is pluggable
class HourlyFeeStrategy:
    def calculate(self, ticket, exit_s):
        hours = max(1, int((exit_s - ticket.entry_s) // 3600))
        rate = 30 if ticket.vehicle.size == SpotSize.LARGE else 20
        return hours * rate

class Ticket:
    def __init__(self, vehicle, spot, entry_s):
        self.vehicle, self.spot, self.entry_s = vehicle, spot, entry_s

class ParkingLot:
    _instance = None
    def __new__(cls):                      # Singleton
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.spots = []
            cls._instance.fee = HourlyFeeStrategy()
            cls._instance.lock = threading.Lock()
        return cls._instance

    def park(self, v):
        with self.lock:                       # no double-assignment
            for s in self.spots:
                if s.is_free() and s.size >= v.size:
                    s.vehicle = v
                    return Ticket(v, s, time.time())
            raise RuntimeError("Parking full")

    def unpark(self, ticket):
        with self.lock:
            fee = self.fee.calculate(ticket, time.time())
            ticket.spot.vehicle = None
            return fee
Why this design scores well: the FeeStrategy interface means new pricing (weekend rates, EV surcharge) is a new class — never an edit to ParkingLot (Open/Closed). The SpotSize ordinal check lets a bigger spot hold a smaller vehicle. And the synchronized/Lock guard is the detail interviewers probe: without it, two gates can assign the same spot.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Nearest-spot optimization: keep a per-size PriorityQueue/heap of free spots instead of scanning a list — O(log n) assignment.
  • Multiple gates & floors: partition locks per floor to reduce contention.
  • Payment: add a PaymentProcessor with card/cash/UPI via Strategy.
  • Reservations / EV charging spots: new SpotSize or a features set on spots.
  • Persistence: back availability with a database; the in-memory maps become a cache.
Common mistakes: jumping straight to code without listing requirements; forgetting concurrency on spot assignment; hardcoding pricing with if/else instead of a Strategy; modeling spot size as a free-form string instead of an enum. Avoid all four and you're in the top tier.

LLD Problem Medium

Design a Rate Limiter

A classic systems-flavored LLD problem. Build a component that decides, per client, whether a request is allowed right now — and stays correct under heavy concurrency.

📋 Problem Statement

Design a rate limiter that caps how many requests each client may make in a time window — for example, 100 requests per minute per API key. The core operation is allowRequest(clientId), which returns true if the request is permitted and false if the client should be throttled. The limiter must support multiple algorithms (token bucket, leaky bucket, fixed window, sliding window) and remain thread-safe when many threads hit the same client at once.

Step 1 — Gather Requirements

Rate limiting looks simple until you push on the edges — bursts, clock boundaries, and concurrency. Pin the scope down first.

✓ Functional

  • allowRequest(clientId) returns a boolean verdict.
  • Per-client limits (each API key / IP tracked separately).
  • Configurable capacity and window (e.g. 100 / minute).
  • Pluggable algorithm: token bucket, leaky bucket, fixed window, sliding window.
  • Allow controlled bursts where the algorithm permits.

◆ Non-Functional

  • Thread-safety: concurrent requests for one client must not over-admit.
  • Low latency: the check sits on the hot path — O(1) per call.
  • Memory bounded: per-client state must stay small.
  • Extensible: add an algorithm without touching callers.

Step 2 — Identify the Core Entities

Find the nouns and the one verb that matters:

  • RateLimiter (interface) — exposes allowRequest(clientId).
  • RateLimitAlgorithm — the Strategy: TokenBucket, LeakyBucket, FixedWindowCounter, SlidingWindowLog.
  • TokenBucket — capacity + refill rate; one bucket holds the per-client state.
  • ClientState — per-client bucket / counter / log, stored in a concurrent map.
  • RateLimiterConfig — capacity, window size, refill rate.

Step 5 — Implementation

We implement Token Bucket fully (it is the most-asked variant) and outline the other three. Token bucket holds up to capacity tokens, refills at a steady rate, and admits a request only if a token is available — naturally allowing short bursts.

import java.util.concurrent.*;

// Strategy interface: every algorithm answers one question
interface RateLimitAlgorithm { boolean tryConsume(); }

// ---- Token Bucket (fully implemented, thread-safe) ----
class TokenBucket implements RateLimitAlgorithm {
    private final long capacity;          // max tokens (burst size)
    private final double refillPerMs;     // tokens added per millisecond
    private double tokens;
    private long lastRefillMs;

    TokenBucket(long capacity, double refillPerSec) {
        this.capacity = capacity;
        this.refillPerMs = refillPerSec / 1000.0;
        this.tokens = capacity;
        this.lastRefillMs = System.currentTimeMillis();
    }

    // synchronized: refill + consume must be one atomic step
    public synchronized boolean tryConsume() {
        refill();
        if (tokens >= 1) { tokens -= 1; return true; }
        return false;
    }

    private void refill() {
        long now = System.currentTimeMillis();
        double added = (now - lastRefillMs) * refillPerMs;
        if (added > 0) {
            tokens = Math.min(capacity, tokens + added);
            lastRefillMs = now;
        }
    }
}

// ---- The limiter: one bucket per client, looked up concurrently ----
class StrategyRateLimiter {
    private final ConcurrentHashMap<String, RateLimitAlgorithm> clients = new ConcurrentHashMap<>();
    private final long capacity; private final double refillPerSec;

    StrategyRateLimiter(long capacity, double refillPerSec) {
        this.capacity = capacity; this.refillPerSec = refillPerSec;
    }

    public boolean allowRequest(String clientId) {
        // computeIfAbsent is atomic — no duplicate bucket on first hit
        RateLimitAlgorithm bucket = clients.computeIfAbsent(
            clientId, k -> new TokenBucket(capacity, refillPerSec));
        return bucket.tryConsume();
    }
}

// ---- Outline of the other three strategies ----
// LeakyBucket:        queue of requests drained at a fixed rate;
//                     admit if queue size < capacity. Smooths bursts.
// FixedWindowCounter: AtomicLong count keyed by (window = now/size);
//                     reset on new window. Cheap, but spikes at edges.
// SlidingWindowLog:   keep request timestamps; drop those older than
//                     the window; admit if remaining count < limit.

class Demo {
    public static void main(String[] args) {
        StrategyRateLimiter rl = new StrategyRateLimiter(5, 1);  // burst 5, 1/sec
        for (int i = 0; i < 7; i++)
            System.out.println("req " + i + " -> " + rl.allowRequest("keyA"));
    }
}

import time, threading

# ---- Token Bucket (fully implemented, thread-safe) ----
class TokenBucket:
    def __init__(self, capacity, refill_per_sec):
        self.capacity = capacity
        self.refill_per_sec = refill_per_sec
        self.tokens = float(capacity)
        self.last = time.monotonic()
        self.lock = threading.Lock()          # guard refill + consume

    def try_consume(self):
        with self.lock:
            self._refill()
            if self.tokens >= 1:
                self.tokens -= 1
                return True
            return False

    def _refill(self):
        now = time.monotonic()
        added = (now - self.last) * self.refill_per_sec
        if added > 0:
            self.tokens = min(self.capacity, self.tokens + added)
            self.last = now

# ---- The limiter: one bucket per client ----
class StrategyRateLimiter:
    def __init__(self, capacity, refill_per_sec):
        self.capacity = capacity
        self.refill_per_sec = refill_per_sec
        self.clients = {}
        self.map_lock = threading.Lock()       # guard first-time creation

    def allow_request(self, client_id):
        bucket = self.clients.get(client_id)
        if bucket is None:
            with self.map_lock:                  # double-checked creation
                bucket = self.clients.setdefault(
                    client_id,
                    TokenBucket(self.capacity, self.refill_per_sec))
        return bucket.try_consume()

# ---- Outline of the other three strategies ----
# LeakyBucket:        deque drained at a fixed rate; admit if not full.
# FixedWindowCounter: count keyed by (now // window); reset each window.
# SlidingWindowLog:   keep timestamps; evict < now - window; admit if
#                     remaining count < limit.

if __name__ == "__main__":
    rl = StrategyRateLimiter(5, 1)            # burst 5, 1/sec
    for i in range(7):
        print(f"req {i} -> {rl.allow_request('keyA')}")
Why this design scores well: the RateLimitAlgorithm Strategy interface lets you swap token bucket for sliding window without touching callers (Open/Closed). computeIfAbsent / setdefault makes per-client bucket creation atomic, and the bucket's own lock makes refill-then-consume a single indivisible step — the detail interviewers probe, since without it two threads can both see the last token.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Distributed limiting: move per-client state to Redis with an atomic Lua script (INCR + EXPIRE) so many app servers share one limit.
  • Lock-free token bucket: replace synchronized with an AtomicLong CAS loop on a packed (tokens, timestamp) value for higher throughput.
  • Memory eviction: expire idle client buckets via an LRU / TTL cache so the map stays bounded.
  • Response semantics: return Retry-After / X-RateLimit-Remaining headers instead of a bare boolean.
  • Tiered limits: different capacities per plan (free vs. premium) selected by config.
Common mistakes: doing refill and consume as two separate operations (a race that over-admits); using System.currentTimeMillis() where a monotonic clock is needed; an unbounded client map that leaks memory; and the fixed-window edge spike — 2× the limit straddling the boundary — which is exactly why sliding window exists.

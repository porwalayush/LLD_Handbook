LLD Problem Medium

Design a Ride-Sharing Service

The Uber/Ola problem. Match a rider to the nearest available driver, price the trip (with surge), and walk the trip through a clean state machine while keeping everyone notified.

📋 Problem Statement

Design a ride-sharing service like Uber or Ola. A rider requests a ride between two locations. The system finds the best available driver nearby, creates a trip, and moves it through a lifecycle: REQUESTED → ASSIGNED → ONGOING → COMPLETED. Fares are computed by a pricing engine that can apply surge. Riders and drivers receive notifications at every step. The two design hot-spots are the matching algorithm and the trip state machine.

Step 1 — Gather Requirements

Always start by pinning down scope. State your assumptions out loud; interviewers want to see you separate the must-haves from the nice-to-haves.

✓ Functional

  • Rider requests a ride from a pickup to a drop location.
  • Match the rider to the nearest available driver.
  • Trip lifecycle: REQUESTED → ASSIGNED → ONGOING → COMPLETED (and CANCELLED).
  • Compute fare, including surge pricing at peak demand.
  • Notify rider and driver on every state change.

◆ Non-Functional

  • Extensibility: swap matching and pricing algorithms freely.
  • Valid transitions only: a trip can't jump from REQUESTED to COMPLETED.
  • Concurrency: one driver can't be assigned to two trips.
  • Decoupled notifications: adding a channel shouldn't touch trip logic.

Step 2 — Identify the Core Entities

Find the nouns in the problem. These become your classes:

  • Rider / Driver — users; a driver has a current Location and an availability flag.
  • Location — latitude/longitude; supports a distance calculation.
  • Trip — the ride; holds rider, driver, pickup/drop, fare, and a current state.
  • TripManager — orchestrates request → match → state transitions.
  • DriverMatchingStrategy — pluggable rule to pick a driver (nearest, highest-rated…).
  • PricingStrategy — pluggable fare calculation, including surge.
  • Notification / Observer — pushes updates to rider and driver.
  • TripState — REQUESTED, ASSIGNED, ONGOING, COMPLETED, CANCELLED.

Step 5 — Implementation

A focused skeleton showing request-ride → match a driver → advance the trip state machine, with observers notified along the way. In an interview you'd build this up incrementally; here's a clean, runnable core.

enum TripState { REQUESTED, ASSIGNED, ONGOING, COMPLETED, CANCELLED }

record Location(double lat, double lng) {
    double distanceTo(Location o) {       // simplified Euclidean
        return Math.hypot(lat - o.lat, lng - o.lng);
    }
}
class Driver { String id; Location loc; boolean available = true;
    Driver(String id, Location l) { this.id = id; loc = l; } }
class Rider  { String id; Rider(String id) { this.id = id; } }

// Strategy: how to pick a driver
interface DriverMatchingStrategy { Driver match(List<Driver> drivers, Location pickup); }
class NearestDriverStrategy implements DriverMatchingStrategy {
    public Driver match(List<Driver> drivers, Location pickup) {
        return drivers.stream().filter(d -> d.available)
            .min(Comparator.comparingDouble(d -> d.loc.distanceTo(pickup)))
            .orElse(null);
    }
}
// Strategy: pricing with surge multiplier
interface PricingStrategy { double price(double distance); }
class SurgePricingStrategy implements PricingStrategy {
    final double surge;
    SurgePricingStrategy(double surge) { this.surge = surge; }
    public double price(double distance) { return (50 + distance * 12) * surge; }
}

// Observer: notify rider + driver without Trip knowing the channels
interface TripObserver { void onUpdate(Trip t); }
class RiderNotifier implements TripObserver {
    public void onUpdate(Trip t) { System.out.println("[Rider] trip is " + t.state); }
}

class Trip {
    Rider rider; Driver driver; Location pickup, drop;
    double fare; TripState state = TripState.REQUESTED;
    private final List<TripObserver> observers = new ArrayList<>();
    void subscribe(TripObserver o) { observers.add(o); }

    // State machine: only allow legal transitions, then fan out to observers
    void transitionTo(TripState next) {
        boolean ok = switch (state) {
            case REQUESTED -> next == TripState.ASSIGNED  || next == TripState.CANCELLED;
            case ASSIGNED  -> next == TripState.ONGOING   || next == TripState.CANCELLED;
            case ONGOING   -> next == TripState.COMPLETED;
            default        -> false;
        };
        if (!ok) throw new IllegalStateException(state + " -x-> " + next);
        state = next;
        observers.forEach(o -> o.onUpdate(this));
    }
}

class TripManager {
    private final List<Driver> drivers;
    private final DriverMatchingStrategy matcher = new NearestDriverStrategy();
    private final PricingStrategy pricing = new SurgePricingStrategy(1.5);
    TripManager(List<Driver> drivers) { this.drivers = drivers; }

    // request → match → assign, guarded so no driver is double-booked
    synchronized Trip requestRide(Rider r, Location pickup, Location drop) {
        Trip trip = new Trip();
        trip.rider = r; trip.pickup = pickup; trip.drop = drop;
        trip.subscribe(new RiderNotifier());
        Driver d = matcher.match(drivers, pickup);
        if (d == null) { trip.transitionTo(TripState.CANCELLED); return trip; }
        d.available = false;
        trip.driver = d;
        trip.fare = pricing.price(pickup.distanceTo(drop));
        trip.transitionTo(TripState.ASSIGNED);
        return trip;
    }
}
from enum import Enum
from math import hypot
import threading

class TripState(Enum):
    REQUESTED = 0; ASSIGNED = 1; ONGOING = 2; COMPLETED = 3; CANCELLED = 4

# Legal transitions for the trip state machine
TRANSITIONS = {
    TripState.REQUESTED: {TripState.ASSIGNED, TripState.CANCELLED},
    TripState.ASSIGNED:  {TripState.ONGOING, TripState.CANCELLED},
    TripState.ONGOING:   {TripState.COMPLETED},
}

class Location:
    def __init__(self, lat, lng): self.lat, self.lng = lat, lng
    def distance_to(self, o): return hypot(self.lat - o.lat, self.lng - o.lng)

class Driver:
    def __init__(self, id, loc): self.id, self.loc, self.available = id, loc, True
class Rider:
    def __init__(self, id): self.id = id

# Strategy: driver matching
class NearestDriverStrategy:
    def match(self, drivers, pickup):
        free = [d for d in drivers if d.available]
        return min(free, key=lambda d: d.loc.distance_to(pickup), default=None)

# Strategy: pricing with surge multiplier
class SurgePricingStrategy:
    def __init__(self, surge=1.0): self.surge = surge
    def price(self, distance): return (50 + distance * 12) * self.surge

# Observer: notifies a party on every trip update
class RiderNotifier:
    def on_update(self, trip): print(f"[Rider] trip is {trip.state.name}")
class DriverNotifier:
    def on_update(self, trip): print(f"[Driver] trip is {trip.state.name}")

class Trip:
    def __init__(self, rider, pickup, drop):
        self.rider, self.pickup, self.drop = rider, pickup, drop
        self.driver, self.fare = None, 0.0
        self.state = TripState.REQUESTED
        self.observers = []
    def subscribe(self, obs): self.observers.append(obs)

    # State machine: reject illegal jumps, then notify observers
    def transition_to(self, nxt):
        if nxt not in TRANSITIONS.get(self.state, set()):
            raise ValueError(f"{self.state.name} -x-> {nxt.name}")
        self.state = nxt
        for o in self.observers:
            o.on_update(self)

class TripManager:
    def __init__(self, drivers):
        self.drivers = drivers
        self.matcher = NearestDriverStrategy()
        self.pricing = SurgePricingStrategy(surge=1.5)
        self.lock = threading.Lock()

    # request → match → assign; locked so a driver isn't double-booked
    def request_ride(self, rider, pickup, drop):
        with self.lock:
            trip = Trip(rider, pickup, drop)
            trip.subscribe(RiderNotifier())
            trip.subscribe(DriverNotifier())
            d = self.matcher.match(self.drivers, pickup)
            if d is None:
                trip.transition_to(TripState.CANCELLED)
                return trip
            d.available = False
            trip.driver = d
            trip.fare = self.pricing.price(pickup.distance_to(drop))
            trip.transition_to(TripState.ASSIGNED)
            return trip

# --- demo ---
drivers = [Driver("D1", Location(0, 0)), Driver("D2", Location(5, 5))]
mgr = TripManager(drivers)
trip = mgr.request_ride(Rider("R1"), Location(1, 1), Location(9, 9))
trip.transition_to(TripState.ONGOING)
trip.transition_to(TripState.COMPLETED)
Why this design scores well: three patterns map cleanly to three concerns. DriverMatchingStrategy and PricingStrategy let you experiment with matching rules and surge formulas without touching trip logic (Open/Closed). The transitionTo guard is a lightweight State machine — a trip can never skip from REQUESTED straight to COMPLETED. And Observer means adding SMS or push notifications is a new subscriber, not an edit to Trip. The lock around matching is the concurrency detail interviewers probe: without it, two riders can grab the same driver.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Geo-indexing: replace the linear "nearest driver" scan with a QuadTree / geohash / S2 cell index for O(log n) spatial lookup at city scale.
  • Real surge: compute the multiplier from live supply/demand per region instead of a constant.
  • Driver acceptance: a matched driver can decline → re-match with a timeout; add a DECLINED path.
  • Ride types & pooling: Mini/Sedan/XL via a Factory; shared rides that pick up multiple riders.
  • Payments & ratings: a PaymentProcessor Strategy on completion; bidirectional ratings.
  • Full State objects: promote each TripState to its own class with onEnter/handle behavior when transitions grow complex.
Common mistakes: hardcoding the matching rule with if/else instead of a Strategy; letting a trip transition to any state (no guard); coupling Trip directly to SMS/push code instead of using Observer; forgetting to lock the match step so a driver gets double-booked; and treating fare as a fixed field instead of a pluggable pricing engine. Avoid these and you're in the top tier.

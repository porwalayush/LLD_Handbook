Design an Online Shopping Cart

An e-commerce staple. Model browsing a catalog, managing a cart, applying pluggable discounts, and checking out into an order that walks through a lifecycle of states.

📋 Problem Statement

Design the shopping cart for an online store. A user browses a catalog of products, adds items to a cart (each a cart item with a quantity), and updates or removes them. The cart computes a subtotal and applies discounts (percentage off, buy-one-get-one, coupon codes). At checkout, the system charges a payment method and produces an order that moves through states: PLACED → PAID → SHIPPED → DELIVERED (or CANCELLED).

Step 1 — Gather Requirements

Always start by pinning down scope. State your assumptions out loud; interviewers want to see you separate the must-haves from the nice-to-haves.

✓ Functional

  • Browse a catalog; look up products.
  • Add / update quantity / remove cart items.
  • Compute the cart subtotal.
  • Apply a discount: percentage, BOGO, or coupon.
  • Checkout: pick a payment method, charge, create an order.
  • Track order state through its lifecycle.

◆ Non-Functional

  • Extensibility: new discount & payment types without edits.
  • Stock safety: don't oversell inventory.
  • Valid transitions: can't ship an unpaid order.
  • Money precision on totals & discounts.
  • Cart survives across sessions (persistable).

Step 2 — Identify the Core Entities

Find the nouns in the problem. These become your classes:

  • Product — an item with id, name, price, and stock.
  • Catalog — the collection of products; supports lookup.
  • CartItem — a product plus a quantity.
  • Cart — a user's items; computes subtotal and applies a discount.
  • User — owns a cart and places orders.
  • DiscountStrategy — pluggable pricing (percentage / BOGO / coupon).
  • PaymentStrategy — pluggable payment (card / wallet / COD).
  • Order — a frozen snapshot of the cart with an OrderState.

Step 4 — Patterns in Play

Name the patterns as you design — it signals maturity. This problem naturally uses:

Strategy — discounts & payments State — order lifecycle Factory — build payments

Step 5 — Implementation

A focused skeleton: add/remove items, apply a discount via Strategy, and checkout into an Order whose State enforces a legal lifecycle.

class Product {
    final String id, name; final double price;
    Product(String id, String name, double price) {
        this.id = id; this.name = name; this.price = price;
    }
}

class CartItem {
    final Product product; int qty;
    CartItem(Product p, int qty) { product = p; this.qty = qty; }
    double lineTotal() { return product.price * qty; }
}

// Strategy: discount rules vary independently of the cart
interface DiscountStrategy { double apply(List<CartItem> items, double subtotal); }
class PercentageDiscount implements DiscountStrategy {
    final double pct;
    PercentageDiscount(double pct) { this.pct = pct; }
    public double apply(List<CartItem> items, double sub) { return sub * pct / 100.0; }
}
class BogoDiscount implements DiscountStrategy {   // buy-one-get-one
    public double apply(List<CartItem> items, double sub) {
        double off = 0;
        for (CartItem i : items) off += (i.qty / 2) * i.product.price;
        return off;
    }
}
class CouponDiscount implements DiscountStrategy {
    final double flatOff;
    CouponDiscount(double flatOff) { this.flatOff = flatOff; }
    public double apply(List<CartItem> items, double sub) { return Math.min(flatOff, sub); }
}

// Strategy: payment method is pluggable
interface PaymentStrategy { boolean pay(double amount); }
class CardPayment implements PaymentStrategy {
    public boolean pay(double amount) { /* call gateway */ return true; }
}

// State pattern: only valid transitions allowed
enum OrderState { PLACED, PAID, SHIPPED, DELIVERED, CANCELLED }

class Order {
    final List<CartItem> items; final double total;
    OrderState state = OrderState.PLACED;
    Order(List<CartItem> items, double total) { this.items = items; this.total = total; }

    void markPaid()   { require(OrderState.PLACED);  state = OrderState.PAID; }
    void ship()       { require(OrderState.PAID);    state = OrderState.SHIPPED; }
    void deliver()    { require(OrderState.SHIPPED); state = OrderState.DELIVERED; }
    private void require(OrderState s) {
        if (state != s) throw new IllegalStateException("Bad transition from " + state);
    }
}

class Cart {
    final Map<String, CartItem> items = new LinkedHashMap<>();
    DiscountStrategy discount;                // nullable

    void add(Product p, int qty) {
        items.merge(p.id, new CartItem(p, qty),
                   (a, b) -> { a.qty += b.qty; return a; });
    }
    void remove(String productId) { items.remove(productId); }

    double subtotal() {
        return items.values().stream().mapToDouble(CartItem::lineTotal).sum();
    }
    double total() {
        double sub = subtotal();
        double off = discount == null ? 0
                   : discount.apply(new ArrayList<>(items.values()), sub);
        return sub - off;
    }

    Order checkout(PaymentStrategy payment) {
        if (items.isEmpty()) throw new IllegalStateException("Empty cart");
        double total = total();
        if (!payment.pay(total)) throw new IllegalStateException("Payment failed");
        Order order = new Order(new ArrayList<>(items.values()), total);
        order.markPaid();
        items.clear();
        return order;
    }
}

from abc import ABC, abstractmethod
from enum import Enum

class Product:
    def __init__(self, id, name, price):
        self.id, self.name, self.price = id, name, price

class CartItem:
    def __init__(self, product, qty):
        self.product, self.qty = product, qty
    def line_total(self): return self.product.price * self.qty

# Strategy: discount rules are pluggable
class DiscountStrategy(ABC):
    @abstractmethod
    def apply(self, items, subtotal): ...

class PercentageDiscount(DiscountStrategy):
    def __init__(self, pct): self.pct = pct
    def apply(self, items, sub): return sub * self.pct / 100.0

class BogoDiscount(DiscountStrategy):        # buy-one-get-one
    def apply(self, items, sub):
        return sum((i.qty // 2) * i.product.price for i in items)

class CouponDiscount(DiscountStrategy):
    def __init__(self, flat_off): self.flat_off = flat_off
    def apply(self, items, sub): return min(self.flat_off, sub)

# Strategy: payment method is pluggable
class PaymentStrategy(ABC):
    @abstractmethod
    def pay(self, amount): ...

class CardPayment(PaymentStrategy):
    def pay(self, amount): return True      # call gateway

# State: only valid transitions allowed
class OrderState(Enum):
    PLACED = 1; PAID = 2; SHIPPED = 3; DELIVERED = 4; CANCELLED = 5

class Order:
    def __init__(self, items, total):
        self.items, self.total = items, total
        self.state = OrderState.PLACED
    def _require(self, s):
        if self.state != s:
            raise RuntimeError(f"Bad transition from {self.state}")
    def mark_paid(self): self._require(OrderState.PLACED);  self.state = OrderState.PAID
    def ship(self):      self._require(OrderState.PAID);    self.state = OrderState.SHIPPED
    def deliver(self):   self._require(OrderState.SHIPPED); self.state = OrderState.DELIVERED

class Cart:
    def __init__(self):
        self.items = {}          # product_id -> CartItem
        self.discount = None

    def add(self, p, qty):
        if p.id in self.items: self.items[p.id].qty += qty
        else: self.items[p.id] = CartItem(p, qty)
    def remove(self, product_id):
        self.items.pop(product_id, None)

    def subtotal(self):
        return sum(i.line_total() for i in self.items.values())
    def total(self):
        sub = self.subtotal()
        off = self.discount.apply(list(self.items.values()), sub) if self.discount else 0
        return sub - off

    def checkout(self, payment):
        if not self.items: raise RuntimeError("Empty cart")
        total = self.total()
        if not payment.pay(total): raise RuntimeError("Payment failed")
        order = Order(list(self.items.values()), total)
        order.mark_paid()
        self.items.clear()
        return order
Why this design scores well: discounts and payments are both Strategy objects, so a flash-sale rule or a UPI gateway is a new class — never a change to Cart (Open/Closed). The OrderState transitions are guarded by require(...), the heart of the State pattern: you physically cannot ship an unpaid order. And checkout snapshots the items into the Order and clears the cart, so later cart edits never mutate a placed order.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Stock & inventory: decrement product stock atomically at checkout; reject if insufficient (the same concurrency concern as seat booking).
  • Stacking discounts: chain multiple DiscountStrategy objects (Decorator) or define precedence rules.
  • Full State objects: replace the enum + require with one class per state (PaidState, ShippedState) so each owns its own legal transitions.
  • Order events: notify the user on each transition via Observer.
  • Persistence: store cart and orders in a DB; carts expire after N days.
Common mistakes: hardcoding discount math with if (coupon) ... else if (bogo) ... instead of polymorphic strategies; letting Order jump straight to SHIPPED without a paid check; sharing the same list reference between the cart and the order so clearing the cart wipes the order; storing money as double and showing $19.999; and forgetting to handle an empty-cart checkout.

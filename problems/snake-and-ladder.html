LLD Problem Easy

Design Snake & Ladder

A classic board-game LLD. Players roll a die, advance along a numbered board, ride ladders up and slide down snakes — first to the last cell wins. Great for showing a clean game loop and a turn queue.

📋 Problem Statement

Design a Snake & Ladder game. The board has cells numbered 1 to 100. Each turn the current player rolls a die and advances by that amount. If they land on the head of a snake they slide down to its tail; if they land on the bottom of a ladder they climb to its top. The first player to reach cell 100 wins. Multiple players take turns in a fixed rotation.

Step 1 — Gather Requirements

Pin scope down first. Two details earn points here: how the board is configured (snakes/ladders as data, not hardcoded) and the turn rotation.

✓ Functional

  • Configurable board size (default 100) with a list of snakes and ladders.
  • 2+ players, taking turns in a fixed rotation (a queue).
  • Roll a die, advance, then apply any jump on the landing cell.
  • First player to land exactly on the final cell wins.
  • Snakes move you down; ladders move you up.

◆ Non-Functional

  • Extensibility: swap a single die for multiple dice or a loaded die.
  • Configurable setup: build any board without touching game logic.
  • O(1) jump lookup from a cell to its destination.
  • Validation: reject overlapping or off-board snakes/ladders.

Step 2 — Identify the Core Entities

Pull the nouns from the statement — they become your classes:

  • Game — runs the loop: holds the board, the dice, and the player queue.
  • Board — knows its size and stores the jumps keyed by their start cell.
  • Jump — a start → end pair; a snake if end < start, a ladder if end > start. (Snake/Ladder collapse into one entity.)
  • Dice — produces a roll; a strategy so single vs. multiple dice is swappable.
  • Player — has a name and a current position.

Step 4 — Patterns in Play

Name the patterns as you design — it signals maturity. This problem naturally uses:

Strategy — single vs. multiple dice Builder — assemble the board Factory — create jumps

Step 5 — Implementation

A clean, runnable core: a Dice strategy, a Board with a jump map and an O(1) destination(), and a Game loop that cycles players through a queue until someone wins.

import java.util.*;

// Strategy: how the dice are rolled is swappable
interface Dice { int roll(); }
class SingleDice implements Dice {
    private final Random rng = new Random();
    public int roll() { return 1 + rng.nextInt(6); }
}
class MultiDice implements Dice {
    private final Random rng = new Random();
    private final int count;
    MultiDice(int count) { this.count = count; }
    public int roll() {
        int sum = 0;
        for (int i = 0; i < count; i++) sum += 1 + rng.nextInt(6);
        return sum;
    }
}

class Player {
    final String name;
    int position = 0;
    Player(String name) { this.name = name; }
}

class Board {
    final int size;
    private final Map<Integer, Integer> jumps = new HashMap<>();  // start -> end

    Board(int size) { this.size = size; }

    // a snake (end < start) or ladder (end > start)
    void addJump(int start, int end) {
        if (start <= 0 || start >= size || end <= 0 || end > size || start == end)
            throw new IllegalArgumentException("Invalid jump");
        jumps.put(start, end);
    }

    // O(1): resolve where a landing cell actually sends the player
    int destination(int cell) { return jumps.getOrDefault(cell, cell); }
}

class Game {
    private final Board board;
    private final Dice dice;
    private final Queue<Player> turns = new LinkedList<>();

    Game(Board board, Dice dice, List<Player> players) {
        this.board = board; this.dice = dice;
        turns.addAll(players);
    }

    // the game loop: rotate players until one reaches the final cell
    Player play() {
        while (true) {
            Player p = turns.poll();          // dequeue current player
            int roll = dice.roll();
            int next = p.position + roll;

            if (next <= board.size) {          // must land exactly; overshoot = stay
                p.position = board.destination(next);  // apply snake/ladder
            }
            if (p.position == board.size) return p;  // winner

            turns.offer(p);                  // re-enqueue at the back
        }
    }
}

import random
from collections import deque

# Strategy: how the dice are rolled is swappable
class SingleDice:
    def roll(self): return random.randint(1, 6)

class MultiDice:
    def __init__(self, count): self.count = count
    def roll(self):
        return sum(random.randint(1, 6) for _ in range(self.count))

class Player:
    def __init__(self, name):
        self.name, self.position = name, 0

class Board:
    def __init__(self, size):
        self.size = size
        self.jumps = {}                       # start -> end

    # a snake (end < start) or ladder (end > start)
    def add_jump(self, start, end):
        if not (0 < start < self.size and 0 < end <= self.size) or start == end:
            raise ValueError("Invalid jump")
        self.jumps[start] = end

    # O(1): resolve where a landing cell actually sends the player
    def destination(self, cell):
        return self.jumps.get(cell, cell)

class Game:
    def __init__(self, board, dice, players):
        self.board, self.dice = board, dice
        self.turns = deque(players)

    # the game loop: rotate players until one reaches the final cell
    def play(self):
        while True:
            p = self.turns.popleft()          # current player
            nxt = p.position + self.dice.roll()

            if nxt <= self.board.size:        # must land exactly; overshoot = stay
                p.position = self.board.destination(nxt)  # apply snake/ladder
            if p.position == self.board.size:
                return p                      # winner

            self.turns.append(p)             # re-enqueue at the back
Why this design scores well: snakes and ladders collapse into a single Jump concept (a start → end map), so there's no duplicated logic — the sign of end - start tells you which it is. The board stores them as data, so any board configuration is built without touching Game. The Dice interface makes single/multiple/loaded dice a swap-in (Strategy), and a Queue of players gives clean, fair turn rotation with O(1) enqueue/dequeue.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Roll-a-six-again rule: have play() grant an extra turn before re-enqueuing when the roll is a 6.
  • Exact-finish variants: bounce back on overshoot instead of staying put — a small tweak to the advance step.
  • Observer for events: notify a UI/logger when a player hits a snake, climbs a ladder, or wins (Observer).
  • Builder for setup: a BoardBuilder validating that no two jumps share a start cell and none overlap a final/start cell.
  • Multiplayer scaling: support N players and persist game state so a match can resume.
Common mistakes: modeling Snake and Ladder as two separate classes with copy-pasted logic instead of one Jump; hardcoding the 100 cells and the snake/ladder positions into the game logic; coupling the dice roll directly into Game so you can't vary it; forgetting the "land exactly on the last cell" rule (allowing overshoot to win); and using a list with O(n) removals for turns instead of a queue.

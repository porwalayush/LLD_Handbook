Design Splitwise

A classic expense-sharing problem. Model how a group splits a bill — equally, by exact amounts, or by percentages — and keep a running ledger of who owes whom.

📋 Problem Statement

Design a system like Splitwise. Users belong to groups. Any member can add an expense (e.g. "Dinner, $120, paid by Alice") and choose how it's split among participants — equally, by exact amounts, or by percentage. The system updates a balance sheet that records pairwise debts (who owes whom how much), and can show each user their net balance. A bonus is simplifying debts so the group settles with the fewest transactions.

Step 1 — Gather Requirements

Always start by pinning down scope. State your assumptions out loud; interviewers want to see you separate the must-haves from the nice-to-haves.

✓ Functional

  • Create users and groups; add members to a group.
  • Add an expense: amount, who paid, and how it's split.
  • Support split types: equal, exact, percentage.
  • Update pairwise balances after each expense.
  • Show a user's balances (who owes them / whom they owe).
  • Settle up between two users.

◆ Non-Functional

  • Correctness: split amounts must sum exactly to the total.
  • Extensibility: new split types without touching existing code.
  • Precision: avoid floating-point drift on money.
  • Fast lookups of any pair's balance.
  • Optional: minimize the number of settle transactions.

Step 2 — Identify the Core Entities

Find the nouns in the problem. These become your classes:

  • User — a person with an id and name.
  • Group — a named set of users that share expenses.
  • Expense — a total amount, the payer, and a list of splits.
  • Split (abstract) — one participant's share of an expense.
  • EqualSplit / ExactSplit / PercentSplit — concrete share rules (Strategy / polymorphism).
  • BalanceSheet — the ledger of pairwise balances (who owes whom).
  • ExpenseManager — orchestrates adding an expense and updating balances.

Step 5 — Implementation

A focused skeleton showing the Split hierarchy, adding an expense with different split types, and updating pairwise balances. The validation that splits sum to the total is the detail interviewers love.

class User {
    final String id, name;
    User(String id, String name) { this.id = id; this.name = name; }
}

// Strategy: each split type knows its own share amount
abstract class Split {
    final User user;
    double amount;                       // resolved share
    Split(User user) { this.user = user; }
}
class EqualSplit extends Split { EqualSplit(User u) { super(u); } }
class ExactSplit extends Split {
    ExactSplit(User u, double amt) { super(u); this.amount = amt; }
}
class PercentSplit extends Split {
    final double percent;
    PercentSplit(User u, double pct) { super(u); this.percent = pct; }
}

class Expense {
    final double total; final User paidBy; final List<Split> splits;
    Expense(double total, User paidBy, List<Split> splits) {
        this.total = total; this.paidBy = paidBy; this.splits = splits;
    }
}

// BalanceSheet: balances.get(a).get(b) = how much a owes b
class BalanceSheet {
    final Map<String, Map<String, Double>> bal = new HashMap<>();

    // debtor owes creditor `amt` more
    void adjust(String debtor, String creditor, double amt) {
        bal.computeIfAbsent(debtor, k -> new HashMap<>())
           .merge(creditor, amt, Double::sum);
        bal.computeIfAbsent(creditor, k -> new HashMap<>())
           .merge(debtor, -amt, Double::sum);   // mirror
    }
    double owes(String a, String b) {
        return bal.getOrDefault(a, Map.of()).getOrDefault(b, 0.0);
    }
}

class ExpenseManager {
    final BalanceSheet sheet = new BalanceSheet();

    void addExpense(Expense e) {
        resolveShares(e);                     // fill each split.amount
        double sum = 0;
        for (Split s : e.splits) sum += s.amount;
        if (Math.abs(sum - e.total) > 0.01)
            throw new IllegalArgumentException("Splits must sum to total");

        for (Split s : e.splits) {
            if (s.user == e.paidBy) continue;     // payer doesn't owe self
            sheet.adjust(s.user.id, e.paidBy.id, s.amount);
        }
    }

    // Strategy dispatch: each type computes its share differently
    private void resolveShares(Expense e) {
        for (Split s : e.splits) {
            if (s instanceof EqualSplit)
                s.amount = e.total / e.splits.size();
            else if (s instanceof PercentSplit p)
                s.amount = e.total * p.percent / 100.0;
            // ExactSplit already carries its amount
        }
    }
}

// --- usage ---
// Dinner $120 paid by Alice, split equally among 3:
// addExpense(new Expense(120, alice,
//     List.of(new EqualSplit(alice), new EqualSplit(bob), new EqualSplit(cara))));
// => bob owes alice 40, cara owes alice 40

from abc import ABC
from collections import defaultdict

class User:
    def __init__(self, id, name):
        self.id, self.name = id, name

# Strategy: each split type resolves its own share
class Split(ABC):
    def __init__(self, user, amount=0.0):
        self.user, self.amount = user, amount

class EqualSplit(Split): pass

class ExactSplit(Split):
    def __init__(self, user, amount):
        super().__init__(user, amount)

class PercentSplit(Split):
    def __init__(self, user, percent):
        super().__init__(user)
        self.percent = percent

class Expense:
    def __init__(self, total, paid_by, splits):
        self.total, self.paid_by, self.splits = total, paid_by, splits

# bal[a][b] = how much a owes b
class BalanceSheet:
    def __init__(self):
        self.bal = defaultdict(lambda: defaultdict(float))

    def adjust(self, debtor, creditor, amt):
        self.bal[debtor][creditor] += amt
        self.bal[creditor][debtor] -= amt          # mirror

    def owes(self, a, b):
        return self.bal[a][b]

class ExpenseManager:
    def __init__(self):
        self.sheet = BalanceSheet()

    def add_expense(self, e):
        self._resolve_shares(e)                  # fill split.amount
        if abs(sum(s.amount for s in e.splits) - e.total) > 0.01:
            raise ValueError("Splits must sum to total")
        for s in e.splits:
            if s.user is e.paid_by:               # payer owes nothing
                continue
            self.sheet.adjust(s.user.id, e.paid_by.id, s.amount)

    # Strategy dispatch by type
    def _resolve_shares(self, e):
        for s in e.splits:
            if isinstance(s, EqualSplit):
                s.amount = e.total / len(e.splits)
            elif isinstance(s, PercentSplit):
                s.amount = e.total * s.percent / 100.0
            # ExactSplit already carries its amount

# --- usage ---
# alice, bob, cara = User("a","Alice"), User("b","Bob"), User("c","Cara")
# mgr = ExpenseManager()
# mgr.add_expense(Expense(120, alice,
#     [EqualSplit(alice), EqualSplit(bob), EqualSplit(cara)]))
# mgr.sheet.owes("b", "a")  # => 40.0
Why this design scores well: the Split hierarchy is textbook Strategy — adding a "split by shares" rule is a new subclass, never an edit to ExpenseManager (Open/Closed). The BalanceSheet stores balances as a mirrored map so owes(a,b) == -owes(b,a) always holds, and any pair lookup is O(1). The validation that splits sum to the total catches the most common real-world bug before it corrupts the ledger.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Debt simplification (minimize transactions): compute each person's net balance (paid − owed), then greedily match the biggest creditor with the biggest debtor until everyone is at zero. This reduces a tangle of pairwise debts to the fewest settle-up payments. With n people you can always settle in at most n−1 transactions.
  • Multiple payers: let an expense have several payers, each with a paid amount.
  • Money precision: store amounts in integer cents (or BigDecimal) to avoid float drift.
  • Currencies: attach a currency to each expense and convert on display.
  • Persistence & notifications: back the ledger with a DB; notify members on new expenses.
Greedy debt-simplification sketch: build net[user] by adding what each user paid and subtracting their shares. Push users with positive net into a max-heap (creditors) and negative net into another (debtors). Repeatedly pop the largest of each, transfer min(credit, debt), and re-push any leftover. Each transfer zeroes out at least one person, so it terminates fast.
Common mistakes: using a chain of if/else on a split-type string instead of polymorphic Split subclasses; forgetting to validate that splits sum to the total (off-by-a-cent percentage rounding); storing money as double and accumulating rounding errors; making the payer owe themselves; and forgetting to mirror the balance so owes(a,b) and owes(b,a) disagree.

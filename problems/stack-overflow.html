LLD Problem Easy

Design Stack Overflow

Model a Q&A platform: users post questions, others answer and comment, everyone votes, and reputation flows from those votes. A great test of entity relationships and event-driven design.

📋 Problem Statement

Design a Stack Overflow-style Q&A system. A user can post a question with tags. Other users post answers and comments. Anyone can upvote or downvote a question or answer; votes adjust the content author's reputation. The question's owner can accept one answer. Users earn badges as they hit reputation milestones, and should be notified when their post gets a new answer.

Step 1 — Gather Requirements

Pin scope down first. Reputation rules and notifications are the parts interviewers probe, so call them out as pluggable from the start.

✓ Functional

  • Post questions with one or more tags.
  • Post answers and comments on questions/answers.
  • Upvote / downvote questions and answers.
  • Votes change the content author's reputation.
  • Accept an answer; award badges at milestones.
  • Search questions by tag.

◆ Non-Functional

  • Extensible reputation rules (per-action deltas) without editing core classes.
  • Decoupled notifications on events like "new answer".
  • One vote per user per post — no double counting.
  • Fast tag lookup for browse/search.

Step 2 — Identify the Core Entities

Pull the nouns from the statement — they become your classes:

  • User — has a name, a reputation score, and a set of badges.
  • Question — authored by a user; has many Answers, Comments, and Tags; can mark one accepted answer.
  • Answer — authored by a user; belongs to a question; can be accepted.
  • Comment — short note attached to a question or answer.
  • Vote — a user's up/down vote on a votable post.
  • Tag — a topic label shared across questions.
  • Badge — a reputation milestone reward.
  • ReputationStrategy — the pluggable rule mapping actions → score deltas.

Step 3 — Class Diagram

Step 4 — Patterns in Play

Name the patterns as you design — it signals maturity. This problem naturally uses:

Observer — notify on new answer Strategy — reputation rules Factory — create posts Composite — posts & comments

Step 5 — Implementation

A clean, runnable core: a Post base that holds votes, a ReputationStrategy for scoring, and a Question that notifies observers when a new answer arrives.

import java.util.*;

// Strategy: reputation rules vary without touching Post/User
interface ReputationStrategy {
    int onUpvote(); int onDownvote(); int onAccepted();
}
class DefaultReputation implements ReputationStrategy {
    public int onUpvote()   { return 10; }
    public int onDownvote() { return -2; }
    public int onAccepted() { return 15; }
}

class User {
    final String name;
    int reputation = 0;
    final Set<String> badges = new HashSet<>();
    User(String name) { this.name = name; }

    void addReputation(int delta) {
        reputation += delta;
        if (reputation >= 1000) badges.add("Trusted");   // milestone badge
    }
    // Observer hook: react to events on the user's posts
    void notify(String event) { /* push / email */ }
}

// shared base for things that can be voted on
abstract class Post {
    final User author;
    final String body;
    final List<Comment> comments = new ArrayList<>();
    private final Map<User, Integer> votes = new HashMap<>();  // one per user
    private final ReputationStrategy rep;

    Post(User author, String body, ReputationStrategy rep) {
        this.author = author; this.body = body; this.rep = rep;
    }

    // dir = +1 upvote, -1 downvote; replaces any prior vote by this user
    void vote(User voter, int dir) {
        if (voter == author) throw new IllegalStateException("No self-vote");
        Integer prev = votes.put(voter, dir);
        if (prev != null && prev == dir) return;        // no change
        author.addReputation(dir > 0 ? rep.onUpvote() : rep.onDownvote());
    }
    int score() { return votes.values().stream().mapToInt(Integer::intValue).sum(); }
    void addComment(Comment c) { comments.add(c); }
    ReputationStrategy rep() { return rep; }
}

class Answer extends Post {
    boolean accepted = false;
    Answer(User a, String b, ReputationStrategy r) { super(a, b, r); }
}
class Comment {
    final User author; final String text;
    Comment(User a, String t) { author = a; text = t; }
}

class Question extends Post {
    final List<Answer> answers = new ArrayList<>();
    final Set<String> tags = new HashSet<>();
    Answer acceptedAnswer;
    Question(User a, String b, ReputationStrategy r) { super(a, b, r); }

    // Observer: posting an answer notifies the question's author
    void addAnswer(Answer ans) {
        answers.add(ans);
        author.notify("New answer on your question");
    }
    // only the asker accepts, and only once
    void accept(Answer ans, User by) {
        if (by != author) throw new IllegalStateException("Only asker accepts");
        if (acceptedAnswer != null) throw new IllegalStateException("Already accepted");
        ans.accepted = true;
        acceptedAnswer = ans;
        ans.author.addReputation(rep().onAccepted());
    }
}

from abc import ABC

# Strategy: reputation rules vary without touching Post/User
class DefaultReputation:
    UPVOTE, DOWNVOTE, ACCEPTED = 10, -2, 15

class User:
    def __init__(self, name):
        self.name, self.reputation, self.badges = name, 0, set()

    def add_reputation(self, delta):
        self.reputation += delta
        if self.reputation >= 1000:
            self.badges.add("Trusted")        # milestone badge

    # Observer hook: react to events on the user's posts
    def notify(self, event): pass      # push / email

# shared base for things that can be voted on
class Post(ABC):
    def __init__(self, author, body, rep):
        self.author, self.body, self.rep = author, body, rep
        self.comments = []
        self.votes = {}                       # user -> +1 / -1 (one per user)

    # direction = +1 upvote, -1 downvote; replaces any prior vote
    def vote(self, voter, direction):
        if voter is self.author:
            raise ValueError("No self-vote")
        prev = self.votes.get(voter)
        self.votes[voter] = direction
        if prev == direction:
            return                            # no change
        delta = self.rep.UPVOTE if direction > 0 else self.rep.DOWNVOTE
        self.author.add_reputation(delta)

    def score(self):
        return sum(self.votes.values())

    def add_comment(self, comment):
        self.comments.append(comment)

class Answer(Post):
    def __init__(self, author, body, rep):
        super().__init__(author, body, rep)
        self.accepted = False

class Comment:
    def __init__(self, author, text):
        self.author, self.text = author, text

class Question(Post):
    def __init__(self, author, body, rep):
        super().__init__(author, body, rep)
        self.answers, self.tags, self.accepted_answer = [], set(), None

    # Observer: posting an answer notifies the question's author
    def add_answer(self, answer):
        self.answers.append(answer)
        self.author.notify("New answer on your question")

    # only the asker accepts, and only once
    def accept(self, answer, by):
        if by is not self.author:
            raise ValueError("Only asker accepts")
        if self.accepted_answer is not None:
            raise ValueError("Already accepted")
        answer.accepted = True
        self.accepted_answer = answer
        answer.author.add_reputation(self.rep.ACCEPTED)
Why this design scores well: votes live in a Map<User, dir>, so a user can change their vote but never double-count it — the detail interviewers check. The ReputationStrategy means new scoring rules (bounty, tag-weighted rep) are a new class, not an edit to Post (Open/Closed). Pulling Question and Answer up to a shared Post base removes duplicated voting/commenting logic, and the notify hook keeps notifications decoupled from posting (Observer).

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Real Observer infrastructure: a NotificationService with subscribers (email, push, in-app) registered per user.
  • Tag index: a Map<Tag, List<Question>> for O(1) browse-by-tag, plus full-text search.
  • Reputation gates: require a minimum rep to downvote or comment — enforce in the strategy.
  • Edit history / versioning: Memento or an append-only revision log on each post.
  • Concurrency & persistence: atomic vote counters in the DB; cache hot questions.
  • Bounties & moderation: flagging, closing duplicates, and a reputation-weighted review queue.
Common mistakes: storing a single vote count instead of per-user votes (lets one user spam upvotes); duplicating voting/commenting logic across Question and Answer rather than a shared Post base; hardcoding reputation deltas with if/else instead of a Strategy; baking notification logic directly into addAnswer so it can't be swapped or scaled; and letting anyone (not just the asker) accept an answer.

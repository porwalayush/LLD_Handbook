LLD Problem Easy

Design a Vending Machine

The textbook State-pattern problem. A machine takes money, lets you pick a product, dispenses it, and returns change — but the same button means different things depending on what the machine is currently doing.

📋 Problem Statement

Design a vending machine that holds several products in numbered slots. A customer inserts coins/notes, selects a product, and the machine dispenses it along with any change. It must refund on demand, reject selections when out of stock, and behave correctly no matter what order the buttons are pressed.

Step 1 — Gather Requirements

Start by pinning down scope. Say your assumptions out loud — interviewers want to see you separate the must-haves from the nice-to-haves.

✓ Functional

  • Insert coins/notes; machine tracks the running balance.
  • Select a product by slot code.
  • Dispense the product and return change.
  • Refund all inserted money on request.
  • Reject selection if out of stock or balance is too low.

◆ Non-Functional

  • Correctness: actions are only legal in certain states.
  • Extensibility: adding a new state (e.g. Maintenance) shouldn't rewrite existing logic.
  • No invalid transitions — can't dispense before paying.
  • Clear feedback to the user on every action.

Step 2 — Identify the Core Entities

Find the nouns in the problem. These become your classes:

  • VendingMachine — the context; holds the current state, balance, and inventory.
  • State (interface) — declares the four actions: insertMoney, selectProduct, dispense, refund.
  • IdleState, HasMoneyState, DispensingState, OutOfStockState — concrete states; each implements the actions differently.
  • Inventory — maps slot codes to (Product, count).
  • Product — name + price.
  • Coin / Note — accepted denominations.

Step 3 — Class Diagram

The key idea: the machine delegates every action to its current State object. Changing behavior = swapping the state.

  Idle      --insertMoney-->  HasMoney
  HasMoney  --selectProduct(in stock)--> Dispensing

Step 4 — Patterns in Play

Name the patterns as you design — it signals maturity. This problem is the canonical home of the State pattern:

State — behavior changes with the machine's mode Singleton — one machine instance Factory — create state objects

Step 5 — Implementation

The same four methods live on every state, but each state responds differently. Notice how HasMoneyState.selectProduct moves the machine into DispensingState, while IdleState.selectProduct simply refuses.

import java.util.*;

class Product {
    final String name; final int price;
    Product(String name, int price) { this.name = name; this.price = price; }
}

// The State interface: every state must answer all four actions
interface State {
    void insertMoney(VendingMachine m, int amount);
    void selectProduct(VendingMachine m, String code);
    void dispense(VendingMachine m);
    void refund(VendingMachine m);
}

class VendingMachine {                          // the context
    final Map<String, Product> products = new HashMap<>();
    final Map<String, Integer> stock = new HashMap<>();
    State state = new IdleState();
    int balance = 0;
    String selected = null;

    void addSlot(String code, Product p, int qty) { products.put(code, p); stock.put(code, qty); }

    // Context delegates every action to the current state
    void insertMoney(int a) { state.insertMoney(this, a); }
    void selectProduct(String c) { state.selectProduct(this, c); }
    void dispense() { state.dispense(this); }
    void refund() { state.refund(this); }
    void setState(State s) { this.state = s; }
}

class IdleState implements State {
    public void insertMoney(VendingMachine m, int a) {
        m.balance += a;
        System.out.println("Balance: " + m.balance);
        m.setState(new HasMoneyState());      // Idle -> HasMoney
    }
    public void selectProduct(VendingMachine m, String c) { System.out.println("Insert money first"); }
    public void dispense(VendingMachine m) { System.out.println("Nothing selected"); }
    public void refund(VendingMachine m) { System.out.println("No money to refund"); }
}

class HasMoneyState implements State {
    public void insertMoney(VendingMachine m, int a) { m.balance += a; System.out.println("Balance: " + m.balance); }
    public void selectProduct(VendingMachine m, String c) {
        Product p = m.products.get(c);
        if (p == null || m.stock.getOrDefault(c, 0) == 0) {
            System.out.println("Out of stock");
            m.setState(new OutOfStockState());     // HasMoney -> OutOfStock
        } else if (m.balance < p.price) {
            System.out.println("Insufficient balance");
        } else {
            m.selected = c;
            m.setState(new DispensingState());     // HasMoney -> Dispensing
            m.dispense();
        }
    }
    public void dispense(VendingMachine m) { System.out.println("Select a product first"); }
    public void refund(VendingMachine m) {
        System.out.println("Refunding " + m.balance);
        m.balance = 0; m.setState(new IdleState());
    }
}

class DispensingState implements State {
    public void insertMoney(VendingMachine m, int a) { System.out.println("Please wait, dispensing"); }
    public void selectProduct(VendingMachine m, String c) { System.out.println("Please wait, dispensing"); }
    public void dispense(VendingMachine m) {
        Product p = m.products.get(m.selected);
        m.stock.put(m.selected, m.stock.get(m.selected) - 1);
        int change = m.balance - p.price;
        System.out.println("Dispensed " + p.name + ", change " + change);
        m.balance = 0; m.selected = null;
        m.setState(new IdleState());            // Dispensing -> Idle
    }
    public void refund(VendingMachine m) { System.out.println("Cannot refund mid-dispense"); }
}

class OutOfStockState implements State {
    public void insertMoney(VendingMachine m, int a) { System.out.println("Item unavailable, refunding soon"); }
    public void selectProduct(VendingMachine m, String c) { System.out.println("Item unavailable"); }
    public void dispense(VendingMachine m) { System.out.println("Nothing to dispense"); }
    public void refund(VendingMachine m) {
        System.out.println("Refunding " + m.balance);
        m.balance = 0; m.setState(new IdleState());      // OutOfStock -> Idle
    }
}

from abc import ABC, abstractmethod

class Product:
    def __init__(self, name, price):
        self.name, self.price = name, price

# The State interface: every state must answer all four actions
class State(ABC):
    @abstractmethod
    def insert_money(self, m, amount): ...
    @abstractmethod
    def select_product(self, m, code): ...
    @abstractmethod
    def dispense(self, m): ...
    @abstractmethod
    def refund(self, m): ...

class VendingMachine:                      # the context
    def __init__(self):
        self.products, self.stock = {}, {}
        self.state = IdleState()
        self.balance, self.selected = 0, None

    def add_slot(self, code, product, qty):
        self.products[code] = product; self.stock[code] = qty

    # Context delegates every action to the current state
    def insert_money(self, a): self.state.insert_money(self, a)
    def select_product(self, c): self.state.select_product(self, c)
    def dispense(self): self.state.dispense(self)
    def refund(self): self.state.refund(self)

class IdleState(State):
    def insert_money(self, m, a):
        m.balance += a
        print(f"Balance: {m.balance}")
        m.state = HasMoneyState()              # Idle -> HasMoney
    def select_product(self, m, c): print("Insert money first")
    def dispense(self, m): print("Nothing selected")
    def refund(self, m): print("No money to refund")

class HasMoneyState(State):
    def insert_money(self, m, a):
        m.balance += a; print(f"Balance: {m.balance}")
    def select_product(self, m, c):
        p = m.products.get(c)
        if p is None or m.stock.get(c, 0) == 0:
            print("Out of stock")
            m.state = OutOfStockState()         # HasMoney -> OutOfStock
        elif m.balance < p.price:
            print("Insufficient balance")
        else:
            m.selected = c
            m.state = DispensingState()         # HasMoney -> Dispensing
            m.dispense()
    def dispense(self, m): print("Select a product first")
    def refund(self, m):
        print(f"Refunding {m.balance}")
        m.balance = 0; m.state = IdleState()

class DispensingState(State):
    def insert_money(self, m, a): print("Please wait, dispensing")
    def select_product(self, m, c): print("Please wait, dispensing")
    def dispense(self, m):
        p = m.products[m.selected]
        m.stock[m.selected] -= 1
        change = m.balance - p.price
        print(f"Dispensed {p.name}, change {change}")
        m.balance, m.selected = 0, None
        m.state = IdleState()                  # Dispensing -> Idle
    def refund(self, m): print("Cannot refund mid-dispense")

class OutOfStockState(State):
    def insert_money(self, m, a): print("Item unavailable, refunding soon")
    def select_product(self, m, c): print("Item unavailable")
    def dispense(self, m): print("Nothing to dispense")
    def refund(self, m):
        print(f"Refunding {m.balance}")
        m.balance = 0; m.state = IdleState()   # OutOfStock -> Idle

# --- demo ---
m = VendingMachine()
m.add_slot("A1", Product("Cola", 25), 2)
m.insert_money(30)        # Idle -> HasMoney, balance 30
m.select_product("A1")   # dispenses Cola, change 5, back to Idle
Why this design scores well: the machine never says "if state == X do this else if state == Y...". Each state is a small class that knows only its own rules, and transitions happen by swapping m.state. Adding a MaintenanceState is a new class — zero edits to existing states (Open/Closed). The "same action, different behavior" requirement falls out naturally: selectProduct dispenses in HasMoneyState but is rejected in IdleState.

Step 6 — Follow-up Extensions

Strong candidates anticipate where the interviewer pushes next:

  • Exact-change tracking: model the cash drawer with denomination counts and refuse if change can't be made.
  • Concurrency: a real machine has one user, but if the controller is multi-threaded, guard state transitions with a lock.
  • Singleton machine: there's one physical machine — make VendingMachine a Singleton.
  • Multiple payment types: add card/UPI via a payment Strategy.
  • Restock & admin: a MaintenanceState that unlocks refill/empty operations.
Common mistakes: implementing the whole machine as one giant if/else on a state enum (the anti-pattern the State pattern fixes); letting illegal actions through (dispensing before paying); forgetting to reset balance/selected after a transaction; and not returning change. Model states as objects and the transitions become self-documenting.

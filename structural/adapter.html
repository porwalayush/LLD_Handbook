Structural Pattern

Adapter

Convert the interface of a class into another interface clients expect. Adapter lets classes work together that otherwise couldn't because of incompatible interfaces.

🔌 Real-World Analogy

Think of a travel power-plug adapter. Your laptop charger has a flat US plug, but the wall socket in Europe expects round EU pins. You can't change the charger, and you certainly can't rewire the building — so you slip a small adapter between them. The adapter exposes EU pins to the wall and accepts the US plug on the other side, quietly translating one shape into the other.

An Adapter in code does exactly that: it sits between two incompatible parties, presenting the interface one side expects while delegating the real work to the other — like a memory-card reader that lets a USB port "speak SD card".

The Problem It Solves

You have a class that does exactly what you need, but its interface doesn't match what your code expects. Crucially, you can't just edit that class:

  • It's a third-party library you only have as a compiled JAR or pip package.
  • It's legacy code that other systems depend on — changing it would break them.
  • It exposes an awkward API (an old XML payment gateway) while the rest of your app speaks a clean modern interface.

You don't want to litter your codebase with conversion logic at every call site, and you don't want to rewrite the existing class. Adapter introduces a small wrapper that implements the interface your code wants and translates each call into the form the existing class understands. Your client code stays clean and ignorant of the mismatch.

How It Works

There are four roles:

  1. Target — the interface your client code expects and calls (e.g. PaymentProcessor).
  2. Adaptee — the existing class with the incompatible interface that you can't change (the legacy/3rd-party gateway).
  3. Adapter — implements Target and holds (or extends) the Adaptee, translating each Target call into one or more Adaptee calls.
  4. Client — works only against Target, blissfully unaware an Adaptee exists behind it.

Two flavors exist:

  • Object adapter (composition) — the adapter holds a reference to the adaptee. Flexible, works in any language, can adapt subclasses too. This is the preferred, idiomatic form.
  • Class adapter (inheritance) — the adapter extends both the target and the adaptee. Needs multiple inheritance, so it's awkward in Java (single-inheritance) but natural in Python.

Implementation

Below we adapt a legacy XML-based payment gateway (the Adaptee, which we cannot modify) to a clean modern PaymentProcessor interface (the Target) that the rest of the application uses.

// --- Target: the clean interface our app expects ---
public interface PaymentProcessor {
    String pay(String account, double amount);
}

// --- Adaptee: legacy 3rd-party gateway we CANNOT change ---
public class LegacyXmlGateway {
    // awkward API: takes an XML string, returns an XML string
    public String submitTransaction(String xmlRequest) {
        // ...talks to the old payment network...
        return "<response status='OK' ref='TX-9981'/>";
    }
}

// --- Adapter (object adapter / composition) ---
public class XmlGatewayAdapter implements PaymentProcessor {
    private final LegacyXmlGateway gateway;   // holds the adaptee

    public XmlGatewayAdapter(LegacyXmlGateway gateway) {
        this.gateway = gateway;
    }

    @Override
    public String pay(String account, double amount) {
        // translate the modern call into the legacy XML form
        String xml = "<txn account='" + account
                   + "' amount='" + amount + "'/>";
        String rawResponse = gateway.submitTransaction(xml);
        // ...parse XML, extract the reference id...
        return parseRef(rawResponse);
    }

    private String parseRef(String xml) {
        return xml.replaceAll(".*ref='([^']+)'.*", "$1");
    }
}

// --- Client: depends ONLY on the Target interface ---
PaymentProcessor processor =
        new XmlGatewayAdapter(new LegacyXmlGateway());
String ref = processor.pay("acct-42", 19.99);
System.out.println("Paid, ref=" + ref);   // Paid, ref=TX-9981

from abc import ABC, abstractmethod
import re

# --- Target: the clean interface our app expects ---
class PaymentProcessor(ABC):
    @abstractmethod
    def pay(self, account: str, amount: float) -> str: ...


# --- Adaptee: legacy 3rd-party gateway we CANNOT change ---
class LegacyXmlGateway:
    def submit_transaction(self, xml_request: str) -> str:
        # ...talks to the old payment network...
        return "<response status='OK' ref='TX-9981'/>"


# --- Adapter (object adapter / composition) ---
class XmlGatewayAdapter(PaymentProcessor):
    def __init__(self, gateway: LegacyXmlGateway):
        self._gateway = gateway              # holds the adaptee

    def pay(self, account: str, amount: float) -> str:
        # translate the modern call into the legacy XML form
        xml = f"<txn account='{account}' amount='{amount}'/>"
        raw = self._gateway.submit_transaction(xml)
        # ...parse XML, extract the reference id...
        return re.search(r"ref='([^']+)'", raw).group(1)


# --- Client: depends ONLY on the Target interface ---
processor: PaymentProcessor = XmlGatewayAdapter(LegacyXmlGateway())
ref = processor.pay("acct-42", 19.99)
print(f"Paid, ref={ref}")               # Paid, ref=TX-9981
Object vs class adapter: the example above is an object adapter (it holds the adaptee). A class adapter would instead inherit from the adaptee — natural in Python (class XmlGatewayAdapter(PaymentProcessor, LegacyXmlGateway)) but impossible in Java because a class can extend only one parent. Prefer composition: it's more flexible and avoids tight coupling to the adaptee's implementation.

When to Use It

  • You want to use an existing class but its interface doesn't match what you need, and you can't (or shouldn't) change it.
  • You're integrating a third-party or legacy library and want to isolate the rest of your code from its quirks behind a clean interface.
  • You need several existing classes to conform to one common interface so client code can treat them uniformly.

Trade-offs

✓ Pros

  • Reuses existing/legacy code without modifying it.
  • Decouples client from the adaptee's concrete interface.
  • Honors the Single Responsibility Principle — conversion logic lives in one place.
  • Open/Closed: add new adapters without touching the client.

✗ Cons

  • Adds an extra layer of indirection and a new class.
  • Can hide a deeper design mismatch better solved by refactoring.
  • Too many adapters across a codebase can hurt readability.
Interview tip: Adapter is the "make it fit" pattern — emphasize that it retrofits two existing, finished interfaces that weren't designed to work together. Contrast it with Bridge, which is planned up front to let an abstraction and its implementation vary independently. Also distinguish it from Decorator (same interface, adds behavior) and Facade (simplifies a whole subsystem). Mentioning object-adapter-via-composition vs class-adapter-via-inheritance shows depth.

Related Patterns

Facade defines a brand-new, simpler interface to a whole subsystem, whereas Adapter makes an existing interface fit an expected one. Decorator keeps the same interface but adds responsibilities, while Adapter deliberately changes the interface. Proxy also wraps an object but preserves its interface, controlling access rather than translating it.

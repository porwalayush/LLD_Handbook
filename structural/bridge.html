Structural Pattern

Bridge

Decouple an abstraction from its implementation so that the two can vary independently.

📺 Real-World Analogy

Think of a TV and its remote control. The remote is the abstraction — it offers high-level operations like power, channel up, volume down. The TV is the implementation — it knows how to actually turn on the screen and change the channel.

You can swap in a fancier remote (a basic one, a voice remote) without touching the TV, and you can point any remote at a different TV brand (Sony, Samsung) without redesigning the remote. The two hierarchies — remotes and devices — evolve on their own, connected only by a thin "bridge": the remote holds a reference to a device.

The Problem It Solves

Imagine you model shapes that can be drawn with different rendering APIs. With plain inheritance you'd create a class for every combination:

  • VectorCircle, RasterCircle, VectorSquare, RasterSquare…
  • Add one new shape (Triangle) and you must add it for every renderer.
  • Add one new renderer (SVG) and you must add it for every shape.

This is a combinatorial explosion: shapes × renderers classes that grows multiplicatively. The root cause is mixing two independent dimensions — what you're drawing and how it's rendered — into a single inheritance tree.

Bridge splits those two dimensions into separate hierarchies and joins them by composition instead of inheritance. Now you have shapes + renderers classes (additive), and either side can grow freely.

How It Works

There are four roles:

  1. Abstraction — the high-level control layer (e.g. Shape). It holds a reference to an Implementor and delegates the real work to it.
  2. Refined Abstraction — concrete variants of the abstraction (Circle, Square) that extend its behavior.
  3. Implementor — the interface for the implementation hierarchy (e.g. Renderer), with primitive operations.
  4. Concrete Implementors — actual implementations (VectorRenderer, RasterRenderer).

The "bridge" is the reference the Abstraction keeps to an Implementor. The two trees never inherit from each other — they're wired together at runtime.

Implementation

Below, a Shape abstraction is bridged to a Renderer implementor. Shapes describe what to draw; renderers decide how (vector vs raster). Each side can add new types without forcing changes on the other.

// --- Implementor: the "how to draw" hierarchy ---
public interface Renderer {
    String renderCircle(double radius);
}

public class VectorRenderer implements Renderer {
    @Override
    public String renderCircle(double radius) {
        return "Drawing circle r=" + radius + " as vector paths";
    }
}

public class RasterRenderer implements Renderer {
    @Override
    public String renderCircle(double radius) {
        return "Drawing circle r=" + radius + " with pixels";
    }
}

// --- Abstraction: holds a reference to an Implementor (the bridge) ---
public abstract class Shape {
    protected final Renderer renderer;   // the bridge
    protected Shape(Renderer renderer) { this.renderer = renderer; }
    public abstract String draw();
}

// --- Refined Abstraction ---
public class Circle extends Shape {
    private final double radius;
    public Circle(Renderer renderer, double radius) {
        super(renderer);
        this.radius = radius;
    }
    @Override
    public String draw() {
        return renderer.renderCircle(radius);   // delegate "how" to impl
    }
}

// --- Client: mix and match the two dimensions freely ---
Shape vectorCircle = new Circle(new VectorRenderer(), 5);
Shape rasterCircle = new Circle(new RasterRenderer(), 5);
System.out.println(vectorCircle.draw());  // ...as vector paths
System.out.println(rasterCircle.draw());  // ...with pixels

from abc import ABC, abstractmethod

# --- Implementor: the "how to draw" hierarchy ---
class Renderer(ABC):
    @abstractmethod
    def render_circle(self, radius: float) -> str: ...


class VectorRenderer(Renderer):
    def render_circle(self, radius: float) -> str:
        return f"Drawing circle r={radius} as vector paths"


class RasterRenderer(Renderer):
    def render_circle(self, radius: float) -> str:
        return f"Drawing circle r={radius} with pixels"


# --- Abstraction: holds a reference to an Implementor (the bridge) ---
class Shape(ABC):
    def __init__(self, renderer: Renderer):
        self.renderer = renderer            # the bridge

    @abstractmethod
    def draw(self) -> str: ...


# --- Refined Abstraction ---
class Circle(Shape):
    def __init__(self, renderer: Renderer, radius: float):
        super().__init__(renderer)
        self.radius = radius

    def draw(self) -> str:
        return self.renderer.render_circle(self.radius)  # delegate "how"


# --- Client: mix and match the two dimensions freely ---
vector_circle = Circle(VectorRenderer(), 5)
raster_circle = Circle(RasterRenderer(), 5)
print(vector_circle.draw())   # ...as vector paths
print(raster_circle.draw())   # ...with pixels

When to Use It

  • You have two (or more) independent dimensions that would otherwise multiply into a combinatorial class explosion (shape × renderer, message × channel, device × remote).
  • You want to switch implementations at runtime rather than bind them at compile time.
  • Both the abstraction and the implementation should be extensible via subclassing, independently of each other.
  • You want to hide implementation details from clients completely.

Trade-offs

✓ Pros

  • Eliminates the abstraction × implementation class explosion (additive, not multiplicative).
  • Abstraction and implementation evolve independently.
  • Implementations can be swapped at runtime.
  • Open/Closed & Single Responsibility: each hierarchy has one concern.

✗ Cons

  • Adds upfront complexity — overkill for a single, stable implementation.
  • More indirection and more classes/interfaces to navigate.
  • Requires foreseeing the two dimensions early in the design.
Interview tip — Bridge vs Adapter: they look similar (both use composition to delegate to another object) but differ in intent and timing. Bridge is designed up front to let two dimensions vary independently — you plan it before writing either hierarchy. Adapter is a retrofit — you bolt it on after the fact to make an existing, unchangeable interface fit one your code expects. If you're preventing a future explosion of combinations, it's Bridge; if you're patching an interface mismatch with code that already exists, it's Adapter.

Related Patterns

Adapter looks structurally similar but is applied after designs exist to reconcile incompatible interfaces, whereas Bridge is planned ahead. Abstract Factory pairs well with Bridge to create and wire the right Implementor into an Abstraction. Proxy also wraps an object but to control access, not to separate an abstraction from its implementation.

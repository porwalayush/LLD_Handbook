Structural Pattern

Composite

Compose objects into tree structures to represent part-whole hierarchies. Composite lets clients treat individual objects and compositions of objects uniformly.

🗂️ Real-World Analogy

Think of a computer file system. A file is a single, indivisible thing — it has a size. A folder is a container that holds files and other folders, which in turn hold more files and folders, nesting as deep as you like.

Now ask: "How big is this?" For a file the answer is just its own size. For a folder the answer is the sum of everything inside it — recursively. The beautiful part is that you ask the question the same way regardless. You don't care whether you're pointing at a single file or a folder containing thousands of items; you just call getSize(). That uniform treatment of a leaf and a whole subtree is exactly what Composite gives you.

The Problem It Solves

You have a structure that is naturally a tree: things that can either be a single item (a leaf) or a group of items (a branch), where a group can contain both leaves and other groups. Examples are everywhere — file systems, organization charts, nested UI components (a panel containing buttons and other panels), arithmetic expressions, and graphic shapes grouped together.

Without Composite, client code drowns in type checks:

  • if (node is File) sum += node.size; else if (node is Folder) { loop through children, recurse... }
  • Every operation (size, render, print, delete) repeats that same branching logic.
  • Adding a new kind of node forces you to revisit every if/else block.

What you actually want: treat a single object and a group of objects the same way. Compute the total size of a single file or of an entire folder tree with the same method call — and let the objects themselves handle the recursion.

How It Works

Composite has three roles:

  1. Component — the common interface (or abstract class) shared by everything in the tree. It declares the operations that both leaves and branches must support (e.g. getSize(), print()). This is the type the client codes against.
  2. Leaf — a single object with no children (a file, a button). It implements the Component operations directly.
  3. Composite — a node that holds a collection of child Components and implements child-management (add, remove). For each operation it delegates to its children recursively and combines their results.

Because both Leaf and Composite implement Component, the client holds a reference to a Component and calls operations on it without ever knowing which concrete type it is. The recursion lives inside the Composite, not in the client.

Implementation

Below is the file-system example. FileSystemNode is the Component; File is the Leaf; Folder is the Composite. Notice how Folder.getSize() simply sums getSize() over its children — that single line drives the whole recursion through arbitrarily deep trees.

import java.util.*;

// --- Component: the common interface for files AND folders ---
interface FileSystemNode {
    int getSize();                 // size in bytes
    void print(String indent);     // show the tree
}

// --- Leaf: a single file, no children ---
class File implements FileSystemNode {
    private final String name;
    private final int size;

    File(String name, int size) {
        this.name = name;
        this.size = size;
    }

    public int getSize() { return size; }   // base case

    public void print(String indent) {
        System.out.println(indent + "📄 " + name + " (" + size + "b)");
    }
}

// --- Composite: a folder that holds other nodes ---
class Folder implements FileSystemNode {
    private final String name;
    private final List<FileSystemNode> children = new ArrayList<>();

    Folder(String name) { this.name = name; }

    // child management lives on the composite
    public Folder add(FileSystemNode node) {
        children.add(node);
        return this;
    }

    // recurse: a folder's size is the sum of its children's sizes
    public int getSize() {
        int total = 0;
        for (FileSystemNode child : children) {
            total += child.getSize();        // uniform call - leaf or folder
        }
        return total;
    }

    public void print(String indent) {
        System.out.println(indent + "📁 " + name + "/ (" + getSize() + "b)");
        for (FileSystemNode child : children) {
            child.print(indent + "  ");
        }
    }
}

// --- Usage: client treats leaves and composites identically ---
Folder root = new Folder("root");
Folder src  = new Folder("src");
src.add(new File("Main.java", 1200))
   .add(new File("Utils.java", 800));
root.add(src)
    .add(new File("README.md", 300));

FileSystemNode node = root;     // client only knows the Component type
System.out.println(node.getSize());   // 2300  (recurses the whole tree)
node.print("");

from abc import ABC, abstractmethod

# --- Component: common interface for files AND folders ---
class FileSystemNode(ABC):
    def __init__(self, name):
        self.name = name

    @abstractmethod
    def get_size(self) -> int: ...

    @abstractmethod
    def print_tree(self, indent=""): ...


# --- Leaf: a single file, no children ---
class File(FileSystemNode):
    def __init__(self, name, size):
        super().__init__(name)
        self.size = size

    def get_size(self):              # base case
        return self.size

    def print_tree(self, indent=""):
        print(f"{indent}📄 {self.name} ({self.size}b)")


# --- Composite: a folder that holds other nodes ---
class Folder(FileSystemNode):
    def __init__(self, name):
        super().__init__(name)
        self.children = []

    def add(self, node):           # child management on composite
        self.children.append(node)
        return self

    def get_size(self):
        # recurse: sum over children, leaf or folder alike
        return sum(child.get_size() for child in self.children)

    def print_tree(self, indent=""):
        print(f"{indent}📁 {self.name}/ ({self.get_size()}b)")
        for child in self.children:
            child.print_tree(indent + "  ")


# --- Usage: client treats leaves and composites identically ---
root = Folder("root")
src = Folder("src")
src.add(File("main.py", 1200)).add(File("utils.py", 800))
root.add(src).add(File("README.md", 300))

node: FileSystemNode = root      # client only knows the Component type
print(node.get_size())             # 2300  (recurses the whole tree)
node.print_tree()
Where to put add/remove? The GoF book discusses two designs. Transparency puts child-management methods on the Component interface so leaves and composites look identical — but a Leaf must then reject add() (often by throwing). Safety puts them only on Composite — cleaner types, but the client sometimes has to check the type before adding. The example above favors safety: add() lives only on Folder.

When to Use It

  • You need to represent a part-whole hierarchy as a tree (files/folders, UI widget trees, org charts, menus and submenus).
  • You want client code to ignore the difference between a single object and a composition of objects.
  • An operation should apply to a whole subtree by recursing through it (sum sizes, render, serialize, search).

Trade-offs

✓ Pros

  • Client code is simple — one uniform interface, no type checks.
  • Easy to add new leaf or composite types (Open/Closed Principle).
  • Naturally models recursive tree structures.

✗ Cons

  • Can over-generalize — a too-broad Component interface forces leaves to implement meaningless methods.
  • Type safety vs. transparency is a genuine tension (the add on a leaf problem).
  • Hard to restrict what a composite may contain (e.g. "only images in this folder").
Interview tip: The keyword that should trigger Composite in your head is "tree" or "part-whole hierarchy." Say explicitly that the value is letting the client treat a leaf and a subtree uniformly, and that the recursion lives in the Composite, not the client. If asked, discuss the transparency vs. safety trade-off for add/remove — it's the classic follow-up.

Related Patterns

Decorator is often used with Composite — both rely on recursive composition, and a decorator can wrap a whole tree. Flyweight lets you share leaf nodes to save memory when a tree contains many identical leaves. The Prototype pattern is handy for cloning complex composite structures. Tree traversal in a Composite is frequently exposed through an Iterator or visited with a Visitor.

Structural Pattern

Decorator

Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality.

☕ Real-World Analogy

Walk into a coffee shop. You start with a base drink — a plain espresso, $2.00. Then you start adding things: a shot of milk (+$0.50), some sugar (+$0.20), a swirl of whipped cream (+$0.70). Each add-on wraps the drink you had so far, contributing to both its price and its description ("espresso, milk, sugar, whip").

Crucially, the cup is still "a beverage" the whole time — you can still ask its cost and its description no matter how many add-ons you've stacked. That's exactly a Decorator: each wrapper adds behavior while keeping the same interface, and you can combine them in any order at runtime.

The Problem It Solves

Imagine modelling those drinks with inheritance. You'd need Espresso, EspressoWithMilk, EspressoWithMilkAndSugar, EspressoWithMilkAndSugarAndWhip, EspressoWithWhip... and then repeat the entire combinatorial explosion for every base drink. This is the classic subclass explosion: N optional features mean up to 2N subclasses, and the set is fixed at compile time.

Decorator solves this by letting you stack behaviors at runtime. Each optional feature becomes one small wrapper class. Need milk + sugar? Wrap the drink in a MilkDecorator, then a SugarDecorator. Need a different combo? Wrap differently. Three features give you three classes, not eight, and the combinations are chosen dynamically.

Real-world library example: Java's I/O streams are textbook Decorators. new BufferedReader(new InputStreamReader(new FileReader("f.txt"))) wraps a raw FileReader with character decoding, then with buffering. Each wrapper is a Decorator adding one responsibility while still being a Reader. The same idea appears in BufferedOutputStream, GZIPInputStream, etc.

How It Works

Decorator has four roles:

  1. Component — the interface shared by the real object and all wrappers (e.g. Beverage with cost() and description()).
  2. ConcreteComponent — the base object being decorated (e.g. Espresso).
  3. Decorator — an abstract base that implements Component and holds a reference to a wrapped Component, delegating to it by default.
  4. ConcreteDecorators — subclasses (MilkDecorator, SugarDecorator) that add behavior before or after delegating to the wrapped object.

The key trick: a Decorator is-a Component and has-a Component. Because it implements the same interface, it can stand in for the object it wraps — and because it wraps another Component, decorators can be nested arbitrarily deep.

Implementation

Below, Beverage is the Component, Espresso the ConcreteComponent, CondimentDecorator the base Decorator, and Milk/Sugar the concrete decorators. Each cost() adds to the wrapped cost; each description() appends to the wrapped description.

// --- Component: the shared interface ---
interface Beverage {
    double cost();
    String description();
}

// --- ConcreteComponent: the base object ---
class Espresso implements Beverage {
    public double cost() { return 2.00; }
    public String description() { return "Espresso"; }
}

// --- Base Decorator: is-a Beverage, has-a Beverage ---
abstract class CondimentDecorator implements Beverage {
    protected final Beverage wrapped;        // the object we decorate
    CondimentDecorator(Beverage wrapped) {
        this.wrapped = wrapped;
    }
}

// --- ConcreteDecorators: add to cost + description ---
class Milk extends CondimentDecorator {
    Milk(Beverage b) { super(b); }
    public double cost() {
        return wrapped.cost() + 0.50;        // add after delegating
    }
    public String description() {
        return wrapped.description() + ", Milk";
    }
}

class Sugar extends CondimentDecorator {
    Sugar(Beverage b) { super(b); }
    public double cost() { return wrapped.cost() + 0.20; }
    public String description() {
        return wrapped.description() + ", Sugar";
    }
}

// --- Usage: stack decorators at runtime ---
Beverage drink = new Espresso();
drink = new Milk(drink);              // wrap with milk
drink = new Sugar(drink);             // then sugar

System.out.println(drink.description()); // Espresso, Milk, Sugar
System.out.println(drink.cost());        // 2.70

from abc import ABC, abstractmethod

# --- Component: the shared interface ---
class Beverage(ABC):
    @abstractmethod
    def cost(self) -> float: ...
    @abstractmethod
    def description(self) -> str: ...


# --- ConcreteComponent: the base object ---
class Espresso(Beverage):
    def cost(self): return 2.00
    def description(self): return "Espresso"


# --- Base Decorator: is-a Beverage, has-a Beverage ---
class CondimentDecorator(Beverage):
    def __init__(self, wrapped: Beverage):
        self.wrapped = wrapped          # the object we decorate


# --- ConcreteDecorators: add to cost + description ---
class Milk(CondimentDecorator):
    def cost(self):
        return self.wrapped.cost() + 0.50   # add after delegating
    def description(self):
        return self.wrapped.description() + ", Milk"


class Sugar(CondimentDecorator):
    def cost(self):
        return self.wrapped.cost() + 0.20
    def description(self):
        return self.wrapped.description() + ", Sugar"


# --- Usage: stack decorators at runtime ---
drink: Beverage = Espresso()
drink = Milk(drink)                 # wrap with milk
drink = Sugar(drink)                # then sugar

print(drink.description())          # Espresso, Milk, Sugar
print(drink.cost())                 # 2.70
Don't confuse this with Python's @decorator syntax. The class-based pattern above is the GoF Decorator — wrapping an object to add behavior. Python's @my_decorator function decorators are a related but different language feature: they wrap a function or class at definition time. Both share the "wrap to extend" idea, but the GoF pattern composes objects at runtime, while @-decorators rewrite a callable once when it's defined.

When to Use It

  • You need to add responsibilities to individual objects dynamically, without affecting other objects of the same class.
  • Extension by subclassing is impractical because of a combinatorial explosion of optional features.
  • You want to add and remove behaviors at runtime, or stack them in different orders.

Trade-offs

✓ Pros

  • Add behavior at runtime without touching existing code (Open/Closed).
  • Avoids subclass explosion — one small class per feature.
  • Mix and match decorators in any combination and order.

✗ Cons

  • Produces many small classes that can be hard to learn.
  • Deeply nested wrapping is hard to debug — stack traces and identity checks get confusing.
  • Order of decorators can matter, introducing subtle bugs.
  • A decorated object is not identical (==) to the object it wraps.
Interview tip: Contrast Decorator with inheritance — inheritance is static and chosen at compile time, while a Decorator composes behavior dynamically at runtime. Also contrast it with Proxy: both wrap an object behind the same interface, but a Decorator adds new responsibilities, whereas a Proxy controls access (lazy loading, caching, permissions) without changing the core behavior. Citing Java I/O streams as a real-world example earns easy credit.

Related Patterns

Adapter changes an object's interface, while Decorator keeps the interface the same and adds behavior. Composite and Decorator both rely on recursive composition — a Decorator can be seen as a Composite with a single child whose job is to add behavior rather than aggregate. Proxy shares Decorator's wrapping structure but exists to control access rather than enrich behavior.

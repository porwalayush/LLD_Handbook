Structural Pattern

Facade

Provide a unified, simplified interface to a set of interfaces in a complex subsystem. Facade defines a higher-level interface that makes the subsystem easier to use.

🎬 Real-World Analogy

Think of the single "Watch a Movie" button on a home-theater remote. Behind that one tap, a dozen things must happen in the right order: the lights dim, the projector screen lowers, the projector powers on and switches to HDMI, the amplifier turns on and selects surround sound at the right volume, and the DVD/streaming player starts playing.

You don't want to memorize and operate each device by hand every time. The remote acts as a facade: one simple call hides all that coordination. The individual devices still exist and can be controlled directly if you ever need to — the facade just gives you an easy front door.

The Problem It Solves

As a subsystem grows, it accumulates many classes with intricate, interdependent relationships. A client that wants to "just do the thing" ends up drowning in details:

  • It must know which classes to instantiate, in what order to call them, and how to wire them together.
  • It becomes tightly coupled to the subsystem's internals — every refactor inside the subsystem ripples out to every client.
  • The same multi-step ritual gets copy-pasted across the codebase, each copy a chance to get the sequence wrong.

The Facade pattern introduces one class that knows the whole dance. Clients talk only to the facade and express intent ("watch a movie", "place an order") instead of orchestrating steps. The complex subsystem is still there — and still usable directly for advanced cases — but the common path is now a single, clean call.

How It Works

  1. Identify the subsystem. A group of cooperating classes (Amplifier, Projector, Lights, DvdPlayer) that together do real work but are awkward to drive from outside.
  2. Create a Facade class that holds references to the subsystem objects and exposes a small number of convenient, high-level methods.
  3. Each facade method coordinates the subsystem: it calls the right objects in the right order, encapsulating the workflow.
  4. Clients depend only on the facade. They no longer import or reference the subsystem classes for the common case — yet those classes remain public, so power users can still bypass the facade when needed.

Implementation

Below, a HomeTheaterFacade.watchMovie() hides the choreography of four subsystem devices. Notice the client code shrinks to a single line, while the subsystem classes stay completely unaware that a facade exists.

// --- Subsystem classes (complex, independent, unaware of the facade) ---
class Amplifier {
    void on() { System.out.println("Amplifier on"); }
    void setSurroundSound() { System.out.println("Surround sound enabled"); }
    void setVolume(int level) { System.out.println("Volume = " + level); }
    void off() { System.out.println("Amplifier off"); }
}
class Projector {
    void on() { System.out.println("Projector on"); }
    void wideScreenMode() { System.out.println("Projector in widescreen"); }
    void off() { System.out.println("Projector off"); }
}
class Lights {
    void dim(int pct) { System.out.println("Lights dimmed to " + pct + "%"); }
    void on() { System.out.println("Lights on"); }
}
class DvdPlayer {
    void on() { System.out.println("DVD player on"); }
    void play(String movie) { System.out.println("Playing \"" + movie + "\""); }
    void stop() { System.out.println("DVD stopped"); }
    void off() { System.out.println("DVD player off"); }
}

// --- The Facade: one simple front door over the subsystem ---
public class HomeTheaterFacade {
    private final Amplifier amp;
    private final Projector projector;
    private final Lights lights;
    private final DvdPlayer dvd;

    public HomeTheaterFacade(Amplifier amp, Projector projector,
                             Lights lights, DvdPlayer dvd) {
        this.amp = amp; this.projector = projector;
        this.lights = lights; this.dvd = dvd;
    }

    public void watchMovie(String movie) {
        System.out.println("Get ready to watch a movie...");
        lights.dim(10);
        projector.on();
        projector.wideScreenMode();
        amp.on();
        amp.setSurroundSound();
        amp.setVolume(8);
        dvd.on();
        dvd.play(movie);
    }

    public void endMovie() {
        System.out.println("Shutting the theater down...");
        dvd.stop(); dvd.off();
        amp.off(); projector.off();
        lights.on();
    }
}

// --- Client: expresses intent in one line ---
HomeTheaterFacade theater = new HomeTheaterFacade(
        new Amplifier(), new Projector(),
        new Lights(), new DvdPlayer());
theater.watchMovie("The Matrix");
theater.endMovie();

# --- Subsystem classes (complex, independent, unaware of the facade) ---
class Amplifier:
    def on(self):                 print("Amplifier on")
    def set_surround_sound(self):  print("Surround sound enabled")
    def set_volume(self, level):    print(f"Volume = {level}")
    def off(self):                print("Amplifier off")

class Projector:
    def on(self):                 print("Projector on")
    def wide_screen_mode(self):    print("Projector in widescreen")
    def off(self):                print("Projector off")

class Lights:
    def dim(self, pct):           print(f"Lights dimmed to {pct}%")
    def on(self):                 print("Lights on")

class DvdPlayer:
    def on(self):                 print("DVD player on")
    def play(self, movie):        print(f'Playing "{movie}"')
    def stop(self):               print("DVD stopped")
    def off(self):                print("DVD player off")


# --- The Facade: one simple front door over the subsystem ---
class HomeTheaterFacade:
    def __init__(self, amp, projector, lights, dvd):
        self.amp = amp
        self.projector = projector
        self.lights = lights
        self.dvd = dvd

    def watch_movie(self, movie):
        print("Get ready to watch a movie...")
        self.lights.dim(10)
        self.projector.on()
        self.projector.wide_screen_mode()
        self.amp.on()
        self.amp.set_surround_sound()
        self.amp.set_volume(8)
        self.dvd.on()
        self.dvd.play(movie)

    def end_movie(self):
        print("Shutting the theater down...")
        self.dvd.stop(); self.dvd.off()
        self.amp.off(); self.projector.off()
        self.lights.on()


# --- Client: expresses intent in one line ---
theater = HomeTheaterFacade(Amplifier(), Projector(),
                            Lights(), DvdPlayer())
theater.watch_movie("The Matrix")
theater.end_movie()
A facade is not a "no new classes" rule. The same idea scales up to services: an OrderService.placeOrder() facade can coordinate Inventory (reserve stock), Payment (charge the card), and Shipping (schedule delivery) so callers never touch those three subsystems directly. The facade owns the workflow; the subsystems stay focused and reusable.

When to Use It

  • You want a simple default view of a complex subsystem that most clients can use without learning its internals.
  • You need to decouple clients from a subsystem, reducing dependencies so the subsystem can evolve freely behind the facade.
  • You want to layer your system — each layer's facade becomes the single entry point for the layer above it.
  • You're wrapping a messy or legacy library and want to expose only the clean operations your app actually needs.

Trade-offs

✓ Pros

  • Shields clients from subsystem complexity.
  • Loosens coupling between clients and the subsystem.
  • Centralizes a multi-step workflow in one place.
  • Subsystem stays fully usable directly for advanced cases.

✗ Cons

  • The facade can become a "god object" coupled to everything if it grows unchecked.
  • Adds an extra layer that can hide useful flexibility.
  • Doesn't prevent clients from reaching the subsystem directly — it only offers an easier path.
  • One more class to maintain as the subsystem changes.
Interview tip: The classic trap is Facade vs Adapter. Both wrap something, but the intent differs: a Facade simplifies — it gives a cleaner, smaller interface over a complex subsystem you already control. An Adapter converts — it makes an existing class fit an interface the client expects, with no goal of simplification. Rule of thumb: Facade wraps many classes to hide complexity; Adapter wraps one class to change its interface.

Related Patterns

Adapter also wraps an object but changes its interface rather than simplifying a subsystem. A Facade is often implemented as a Singleton since one facade instance usually suffices. Proxy shares the wrapping structure too, but its purpose is to control access to a single object, not to simplify many.

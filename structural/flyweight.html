Structural Pattern

Flyweight

Use sharing to support large numbers of fine-grained objects efficiently, keeping memory low by separating what can be shared from what cannot.

🌲 Real-World Analogy

Imagine a game that renders a forest of 1,000,000 trees. Every tree has a 3D mesh, a texture bitmap, and a color — data that's heavy in memory but identical for every tree of the same kind. What actually differs from tree to tree is just its position (x, y) and maybe a slight color tint.

Storing the full mesh and texture a million times would blow up your RAM. Instead, you store each unique tree type exactly once and let all million trees point to the shared type, carrying only their own coordinates. A million objects, but only a handful of heavy ones. That's the Flyweight pattern — the same trick used for glyphs in a text editor and particles in a physics engine.

The Problem It Solves

Some programs need a huge number of similar objects: every character in a document, every bullet in a game, every cell in a spreadsheet. If each object stores a full copy of its data, memory usage explodes:

  • 1,000,000 trees × (mesh + texture + color) per tree = gigabytes wasted on duplicate data.
  • Most of that data is invariant — it's the same for every object of a given kind.
  • The few bytes that genuinely vary (position, state) are tiny by comparison.

Flyweight says: stop duplicating the invariant part. Store it once in a shared object, and have every instance reference it. You trade a little indirection for an enormous reduction in memory.

How It Works: Intrinsic vs Extrinsic State

The heart of the pattern is splitting an object's data into two buckets. Getting this split right is the whole game:

  • Intrinsic state — the part that is shared and context-independent. It's the same for many objects, so it lives inside the flyweight and is stored only once (e.g., a tree's name, color palette, mesh, and texture). Because it's shared across many users, it must be immutable.
  • Extrinsic state — the part that is unique per use and varies by context. It is not stored in the flyweight; instead the client holds it and passes it in when calling the flyweight's methods (e.g., a tree's x/y position on the map).

A FlyweightFactory manages the shared pool. When a client asks for a flyweight with given intrinsic state, the factory checks a cache: if a matching flyweight already exists, it returns that same instance; otherwise it creates one, caches it, and returns it. This guarantees objects with identical intrinsic state are never duplicated.

Implementation

Here, TreeType is the flyweight — it holds the intrinsic state (name, color, texture) and is created once per unique combination via the factory. Each Tree holds only the extrinsic state (x, y) plus a reference to its shared TreeType. Watch how the factory hands back the same TreeType for repeated requests.

import java.util.*;

// --- Flyweight: holds INTRINSIC (shared, immutable) state ---
public final class TreeType {
    private final String name;     // intrinsic
    private final String color;    // intrinsic
    private final String texture;  // intrinsic (heavy)

    public TreeType(String name, String color, String texture) {
        this.name = name; this.color = color; this.texture = texture;
    }

    // EXTRINSIC state (x, y) is passed in by the caller, not stored here
    public void draw(int x, int y) {
        System.out.println("Draw " + name + " (" + color +
                           ") at [" + x + "," + y + "]");
    }
}

// --- Flyweight Factory: caches and reuses shared flyweights ---
public class TreeFactory {
    private static final Map<String, TreeType> pool = new HashMap<>();

    public static TreeType getTreeType(String name, String color, String texture) {
        String key = name + "-" + color + "-" + texture;
        // return the SAME instance if we've seen this combo before
        return pool.computeIfAbsent(key,
                k -> new TreeType(name, color, texture));
    }

    public static int distinctTypes() { return pool.size(); }
}

// --- Context object: holds only EXTRINSIC state + a shared reference ---
public class Tree {
    private final int x, y;            // extrinsic (unique per tree)
    private final TreeType type;       // shared flyweight reference

    public Tree(int x, int y, TreeType type) {
        this.x = x; this.y = y; this.type = type;
    }
    public void draw() { type.draw(x, y); }
}

// --- Client: plant a million trees, but only a few TreeType objects ---
List<Tree> forest = new ArrayList<>();
for (int i = 0; i < 1_000_000; i++) {
    // every "Oak" request returns the SAME shared TreeType
    TreeType oak = TreeFactory.getTreeType("Oak", "green", "oak.png");
    forest.add(new Tree(rand(), rand(), oak));
}
System.out.println(TreeFactory.distinctTypes()); // 1, not 1,000,000

from dataclasses import dataclass

# --- Flyweight: holds INTRINSIC (shared, immutable) state ---
# frozen=True makes it immutable, which is required for safe sharing
@dataclass(frozen=True)
class TreeType:
    name: str       # intrinsic
    color: str      # intrinsic
    texture: str    # intrinsic (heavy)

    # EXTRINSIC state (x, y) is passed in, not stored on the flyweight
    def draw(self, x, y):
        print(f"Draw {self.name} ({self.color}) at [{x},{y}]")


# --- Flyweight Factory: caches and reuses shared flyweights ---
class TreeFactory:
    _pool: dict = {}

    @classmethod
    def get_tree_type(cls, name, color, texture):
        key = (name, color, texture)
        if key not in cls._pool:
            cls._pool[key] = TreeType(name, color, texture)
        return cls._pool[key]   # same instance for repeat requests

    @classmethod
    def distinct_types(cls):
        return len(cls._pool)


# --- Context object: only EXTRINSIC state + a shared reference ---
class Tree:
    __slots__ = ("x", "y", "type")   # keep per-instance cost tiny
    def __init__(self, x, y, tree_type):
        self.x, self.y = x, y       # extrinsic (unique per tree)
        self.type = tree_type       # shared flyweight reference
    def draw(self):
        self.type.draw(self.x, self.y)


# --- Client: a million trees, but only a few TreeType objects ---
import random
forest = []
for _ in range(1_000_000):
    oak = TreeFactory.get_tree_type("Oak", "green", "oak.png")
    forest.append(Tree(random.randint(0, 999),
                       random.randint(0, 999), oak))

print(TreeFactory.distinct_types())   # 1, not 1,000,000
Flyweights must be immutable. Because a single flyweight instance is shared by potentially millions of contexts, mutating its intrinsic state would silently change every object pointing at it. Keep the shared state read-only (Java final fields, Python frozen=True), and never let extrinsic state leak into the flyweight.

When to Use It

  • Your app creates a huge number of objects and memory is the bottleneck.
  • Most of each object's state can be made extrinsic (moved out and passed in).
  • After removing extrinsic state, many objects collapse into a few shared groups.
  • The app doesn't depend on object identity — it's fine for many references to point at the same shared instance.

Trade-offs

✓ Pros

  • Dramatic memory savings when objects share state.
  • Fewer object instantiations, less GC pressure.
  • Centralizes shared state in one cached place.

✗ Cons

  • Adds real complexity — you must split intrinsic vs extrinsic state cleanly.
  • Trades RAM for CPU: extrinsic state must be recomputed or passed on every call.
  • Flyweights must be immutable, which constrains the design.
  • Pointless unless object counts are genuinely large.
Interview tip: Cite real flyweights from the standard library. Java's Integer.valueOf() caches boxed integers from -128 to 127 (the Integer cache), so Integer.valueOf(100) == Integer.valueOf(100) is true while 200 == 200 via boxing is false — a flyweight in action. The Java String pool (interned string literals) is the same idea. In Python, small ints (-5 to 256) and short interned strings are cached for exactly this reason. Naming these shows you recognize the pattern in the wild.

Related Patterns

The Flyweight factory is often a Singleton, and creating flyweights can be wrapped behind a Factory Method. Flyweight pairs naturally with Composite — shared leaf nodes in a large tree are frequently implemented as flyweights to keep the structure memory-efficient.

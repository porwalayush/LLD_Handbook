Structural Pattern

Proxy

Provide a surrogate or placeholder for another object to control access to it.

💳 Real-World Analogy

A credit card is a proxy for your bank account. It exposes the same capability — "pay for things" — but stands in for the real money behind the scenes. When you swipe it, the card doesn't hold the cash itself; it controls access to your account, checks your limit, and forwards the charge. You, the merchant, never touch the bank directly.

The same idea shows up everywhere access needs a middleman: a security checkpoint at a building lobby (verify your badge before letting you in), a cheque (a stand-in for cash), or a lawyer/agent acting on your behalf. In each case the proxy has the same "interface" as the real thing but adds a layer of control.

The Problem It Solves

Sometimes you can't — or shouldn't — let a client talk to an object directly. You need to inject some control around access without changing the real object or the client that uses it:

  • Lazy loading — the real object is expensive to create (a huge image, a big DB result). Defer creating it until it's actually needed.
  • Access control — only certain callers should be allowed to invoke certain operations.
  • Remote access — the real object lives on another machine; the proxy hides the network plumbing.
  • Caching / logging — you want to remember results or record every call without polluting the real object.

The Proxy pattern solves all of these the same way: a stand-in object implements the same interface as the real object, so clients can't tell the difference. The proxy adds its control logic, then delegates the actual work to the real object.

How It Works

  1. Subject interface — a common interface that both the real object and the proxy implement, so the proxy is a drop-in substitute.
  2. RealSubject — the actual object that does the real work.
  3. Proxy — implements Subject, holds a reference to the RealSubject (often created lazily), and on each call performs its control logic (check, log, cache, connect) before delegating to the real object.

The four classic proxy types

TypeWhat it controls
VirtualLazy initialization — delays creating an expensive RealSubject until first use.
ProtectionAccess control — checks the caller's permissions before delegating.
RemoteLocal stand-in for an object in another address space / over the network.
CachingStores results of expensive operations and returns them on repeat calls.

Implementation

Below is a Virtual Proxy: ImageProxy shows the lightweight metadata immediately but only loads the heavy RealImage from disk the first time display() is actually called. The client codes against the Image interface and never knows it's holding a proxy.

// --- Subject: the shared interface ---
public interface Image {
    void display();
}

// --- RealSubject: expensive to create (loads bytes from disk) ---
public class RealImage implements Image {
    private final String filename;

    public RealImage(String filename) {
        this.filename = filename;
        loadFromDisk();          // heavy work happens here
    }
    private void loadFromDisk() {
        System.out.println("Loading " + filename + " from disk...");
    }
    public void display() {
        System.out.println("Displaying " + filename);
    }
}

// --- Virtual Proxy: defers the expensive load until first display() ---
public class ImageProxy implements Image {
    private final String filename;
    private RealImage real;     // not created yet!

    public ImageProxy(String filename) {
        this.filename = filename;   // cheap: just remember the name
    }

    public void display() {
        if (real == null) {           // lazy init on first real use
            real = new RealImage(filename);
        }
        real.display();              // delegate to the real object
    }
}

// --- Client: codes against Image, unaware it holds a proxy ---
Image img = new ImageProxy("photo.png");
// nothing loaded yet -- the proxy is cheap
img.display();   // NOW it loads, then displays
img.display();   // already loaded, just displays

from abc import ABC, abstractmethod

# --- Subject: the shared interface ---
class Image(ABC):
    @abstractmethod
    def display(self): ...

# --- RealSubject: expensive to create (loads bytes from disk) ---
class RealImage(Image):
    def __init__(self, filename):
        self.filename = filename
        self._load_from_disk()     # heavy work happens here
    def _load_from_disk(self):
        print(f"Loading {self.filename} from disk...")
    def display(self):
        print(f"Displaying {self.filename}")

# --- Virtual Proxy: defers the expensive load until first display() ---
class ImageProxy(Image):
    def __init__(self, filename):
        self.filename = filename     # cheap: just remember the name
        self._real = None            # not created yet!
    def display(self):
        if self._real is None:        # lazy init on first real use
            self._real = RealImage(self.filename)
        self._real.display()        # delegate to the real object


# --- Client: codes against Image, unaware it holds a proxy ---
img = ImageProxy("photo.png")
img.display()   # NOW it loads, then displays
img.display()   # already loaded, just displays


# --- Pythonic alternative: a generic proxy via __getattr__ ---
# __getattr__ forwards ANY attribute access to the wrapped object,
# so one proxy can transparently stand in for many types.
class LoggingProxy:
    def __init__(self, target):
        self._target = target
    def __getattr__(self, name):
        attr = getattr(self._target, name)   # look up on real object
        if callable(attr):
            def wrapped(*a, **kw):
                print(f"-> calling {name}{a}")   # control logic
                return attr(*a, **kw)            # then delegate
            return wrapped
        return attr
Protection Proxy variant. The same skeleton becomes a protection proxy by checking permissions before delegating: display() would first verify currentUser.hasRole("VIEWER") and throw (or no-op) if not, otherwise call real.display(). The interface, the delegation, and the client all stay identical — only the control logic inside the proxy changes.

When to Use It

  • An object is expensive to create and you want to defer it until needed (virtual proxy).
  • You need access control / permission checks in front of an object (protection proxy).
  • The real object is remote and you want a local stand-in that hides the network (remote proxy).
  • You want to cache results, log calls, or count references without modifying the real object.

Trade-offs

✓ Pros

  • Controls access transparently — clients are unaware.
  • Enables lazy loading, saving startup cost and memory.
  • Adds cross-cutting concerns (security, caching, logging) without touching the real object.
  • Open/Closed: new proxies don't change the RealSubject or client.

✗ Cons

  • Adds another layer of indirection and a new class.
  • Can introduce latency (the proxy does work before delegating).
  • Lazy creation can make response time less predictable.
  • More moving parts to keep in sync with the Subject interface.
Interview tip: The key contrast is Proxy vs Decorator vs Adapter — all three wrap an object, but the intent differs. A Decorator ADDS behavior (and you can stack many), a Proxy CONTROLS access to the same behavior (lazy-load, guard, cache) and usually keeps the same interface, while an Adapter CHANGES the interface so an incompatible class fits. Same structure, three different purposes — say that and you've nailed it.

Related Patterns

Decorator has the same wrapping structure but its purpose is to add behavior, not control access. Adapter also wraps an object but deliberately changes its interface. A Facade wraps a whole subsystem to simplify it, whereas a Proxy wraps a single object to govern access to it.
